## 概要

Ubieのプロダクトデザイナー・はたけ（Hatake）氏が、認知科学の「記憶の4段階モデル（符号化→保持→忘却→想起）」をAI秘書（AIエージェント）の記憶設計に応用し、AI秘書の応答品質を劇的に改善した実践記録。情報を溜めるだけでなく「忘れる仕組み」と「思い出す仕組み」を設計することの重要性を説き、読者がすぐに導入できるスターターキットとカスタマイズ用プロンプトも公開している。

## 主要なトピック

### 1. 問題：AI秘書の記憶が"腐っていた"

著者のAI秘書は議事録の自動生成など**フロー情報の入力**は整備されていたが、使い込むほどに以下の問題が発生：

- 先週の重要な意思決定の文脈を拾ってくれない
- 無関係な過去の議事録を参照して的外れな提案をする
- コンテキストが肥大化し、レスポンスが遅くなる

> 入ってくる情報は溜まり続けるが、整理されていない。コードにリファクタリングが必要なように、**記憶にもリファクタリングが必要**だった。

### 2. 転換点：認知科学の「記憶の4段階モデル」

Anthropicのeラーニングや認知科学の基礎を学ぶ中で、Atkinson & Shiffrin（1968）の多重貯蔵モデルに基づく記憶の4段階フレームワークに出会う：

| 段階 | 説明 | 日常の例 |
|------|------|----------|
| **符号化（Encoding）** | 何を記憶に入れるかフィルタリング | 会議中に重要な決定だけメモを取る |
| **保持（Storage）** | 短期→長期記憶に整理して保管 | メモをカテゴリ別ノートに転記 |
| **忘却（Forgetting）** | 不要な記憶を手放し検索精度を維持 | 去年の買い物リストを捨てる |
| **想起（Retrieval）** | 文脈に合った情報を引き出す | 「あの件どうなった？」に答える |

**核心的な気づき**: AI秘書に足りなかったのは「忘却」と「想起」の設計がまるごと抜けていたこと。

Stanford の Generative Agents 論文（Park et al., 2023）でも、Recency × Importance × Relevance のスコアリング導入で行動の質が劇的に向上したと報告されている。

### 3. 記憶の3分類とAIへのマッピング

認知科学の記憶3分類（Tulving, 1972）がAIエージェントの記憶設計にそのまま適用できる：

| 記憶タイプ | 人間の例 | AIでの対応 |
|-----------|---------|-----------|
| **エピソード記憶** | 体験・出来事のログ | 議事録・日報 |
| **意味記憶** | 知識・パターン・ルール | スキル・ルール |
| **手続き記憶** | 繰り返す行動パターン | フック・手順書 |

> 異なる種類の記憶を混ぜて管理していたから、検索精度が落ちていた。

#### hatake-agent の実装

記憶タイプごとに保存先を分離：

- **エピソード記憶** → `context-log.md`（時系列、文脈付き、90日で減衰）
- **意味記憶** → `frameworks.md` / `decisions.md`（抽象化済み、長寿命）
- **手続き記憶** → `preferences.md` / `hooks/`（自動適用、変更まで永続）

### 4. 「忘れる」は機能である

**本記事の最大の発見**。AIは MDファイルに書かれていれば1年前の情報も引き出してしまう。忘れないことがノイズになり、本当に重要な情報が埋もれる。

> 忘却は失敗ではなく機能。不要な情報を削除することで、重要な記憶の検索精度を維持できる。
> — Ebbinghaus（1885）の忘却曲線の研究に通じる考え方

#### hatake-agent の忘却設計（3つのメカニズム）

**① TTL（Time To Live）— 記憶タイプ別の賞味期限**

| 記憶タイプ | TTL | 備考 |
|-----------|-----|------|
| エピソード記憶（context-log） | 90日 | 参照時にTTLリセット |
| 意味記憶（frameworks） | 無期限 | 月初に陳腐化チェック |
| 手続き記憶（preferences） | 変更まで永続 | フィードバック時に更新 |
| タスク（tasks.md） | 完了後7日 | — |

**② 昇格ルール（固定化プロセス）**

3回以上参照されたエピソード記憶を `frameworks.md` に意味記憶として昇格。人間の脳における「固定化（consolidation）」—— 海馬の短期記憶が大脳皮質の長期記憶に移行するプロセスの模倣。

**③ 月次メンテナンス**

月初に `context-log` を圧縮し、期限切れの記憶を検出・アーカイブ。CLAUDE.md に記載しておけばAIが自律的に実行。

### 5. 「思い出す」にも設計が必要 — R×I×R スコアリング

人間の無意識のスコアリングを3軸に分解し、AIの想起に適用：

```
Score(memory) = Recency × Importance × Relevance
```

**Recency（鮮度）— 段階減衰モデル**

| 期間 | スコア |
|------|--------|
| 1週間以内 | 1.0 |
| 1〜2週間 | 0.8 |
| 2週間〜1ヶ月 | 0.5 |
| 1ヶ月超 | 0.3 |
| 3ヶ月超 | 0.1（削除候補） |

**Importance（重要度）— ファイル格納先で判定**

| 格納先 | スコア |
|--------|--------|
| MEMORY.md 冒頭の高頻度参照ランキング | 1.0 |
| frameworks.md / decisions.md | 0.8 |
| context-log のみ | 0.5 |
| 引き継ぎ・一時メモ | 0.3 |

**Relevance（関連度）**

ベクトルDBは不使用。MEMORY.md のインデックス（タイトル＋タグ＋1行サマリ）をLLMに読ませて判定。記憶ファイルが数十〜百件規模なら十分機能する。

**想起の3フェーズ**:

```
セッション開始
  ↓
MEMORY.md を読む（インデックス＋タグ＋last_referenced）
  ↓
今日の文脈を取得（日付・予定・Slack）
  ↓
R×I×R で「今日関連しそうなファイル」を判定
  ↓
上位3〜5件をロード → 作業開始
```

### 6. 予想外の発見：AIが映し出した楽観バイアス

1週間のSlack・Notion・Google Meetのログをアクティビティレポート化したところ、異なる文脈で「まあ、なんとかなるやろ」と繰り返し発言していたパターンが可視化された。

> 自分では自覚していなかった楽観バイアスが、AIによって可視化された。

この発見により、AI秘書は「単なる秘書」から**自分を理解するためのパートナー**（コーチ・カウンセラー的存在）へと役割が昇華した。

### 7. 付録：5分で始めるAI記憶設計スターターキット

**Step 1: ディレクトリ構成（1分）**

```bash
mkdir -p memory
touch memory/MEMORY.md        # インデックス（AIが最初に読む目次）
touch memory/context-log.md   # エピソード記憶（日付付きの出来事・議論）
touch memory/frameworks.md    # 意味記憶（ルール・判断基準・フレームワーク）
touch memory/preferences.md   # 手続き記憶（作業スタイル・好み）
```

**Step 2: CLAUDE.md に記憶ルールを貼る（2分）**

符号化（何を覚えるか）・忘却（いつ忘れるか）・想起（どう思い出すか）の3つのルールをAI設定ファイルに記載。符号化の判定には「再利用性」「非自明性」「判断影響度」の3軸を使い、2つ以上該当すれば記録する。

**Step 3: MEMORY.md を書く（2分）**

よく参照するファイルの一覧と最近の出来事を1行サマリで管理。最初は空でも、使いながら育てる。

### 8. 上級者向け：6層コンテキスト設計

OpenAI のデータエージェント設計（2026年1月公開）を参考にした拡張構造：

| レイヤー | 内容 | 対応ファイル |
|---------|------|------------|
| L1 | 高頻度参照 | MEMORY.md 冒頭のランキング |
| L2 | 人間の判断・注釈 | preferences.md + decisions.md |
| L3 | 実績データ | tasks.md（Jira/GitHub 連携） |
| L4 | 組織知 | frameworks.md + team-directory.md |
| L5 | セッション横断の学び | context-log.md |
| L6 | ランタイム | 今日の予定・Slack を動的取得 |

> **核心**: 精度 = モデル × コンテキスト。モデルは各社が上げてくれるが、コンテキストの設計は自分たちでやるしかない。

## 結論・示唆

### 著者の結論

- **「覚える」だけが記憶設計ではない**。符号化・保持・忘却・想起の4次元すべてを設計してはじめて"使えるAI"になる
- AIの記憶を設計する過程で見えてくるのは、AIのことだけでなく**自分自身のこと**
- AIを理解することは、人間を理解すること

### 実践的な示唆

- **記憶は種類ごとに分離する**: エピソード・意味・手続きの3タイプを別ファイルで管理し、検索精度を向上
- **忘却を意図的に設計する**: TTL・昇格ルール・月次メンテナンスの3メカニズムで情報のノイズを排除
- **想起にスコアリングを導入する**: R×I×R（Recency × Importance × Relevance）で文脈に合った記憶を優先的にロード
- **メタ認知ツールとして活用する**: 行動ログの分析でセルフコーチングの素材を得る

### 記憶を見直す3つの問い

1. **ノイズの確認**: 3ヶ月前の一度きりの会話が不要に残っていないか？
2. **昇格の確認**: 繰り返し参照する重要情報がすぐ引き出せる状態か？
3. **メタ認知の確認**: AIに「自分の傾向を教えて」と聞いて意味のある答えが返るか？

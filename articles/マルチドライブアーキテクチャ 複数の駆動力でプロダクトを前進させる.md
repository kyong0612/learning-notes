## 要約

### 1. 速度差が摩擦を生む：マルチドライブアーキテクチャの背景
プロダクト開発において、「もっと速く市場の変化に追いつきたい（短期）」、「ちゃんと設計して整えたい（中期）」、「未来へ備えたい（長期）」という異なる時間軸の要求は、しばしば板挟み状態（ジレンマ）を引き起こします。
これら異なる速度を持つ関心事を不用意に混ぜると、速度差による「テンション（歪み）」が生じ、双方の良さを殺してしまいます。これを解決するために、異なる時間軸を独立した駆動力として扱う「マルチドライブアーキテクチャ」が提唱されています。

![異なる時間軸を扱うマルチドライブアーキテクチャ](https://files.speakerdeck.com/presentations/590c73844aac4ef997b5a8afd5cb8dd8/slide_11.jpg)

### 2. マルチドライブアーキテクチャの3つの駆動力
経営理論（Anthonyの経営管理の3階層）を参考に、以下の3つの駆動力を定義しています。

![経営理論の3階層と、3つの駆動](https://files.speakerdeck.com/presentations/590c73844aac4ef997b5a8afd5cb8dd8/slide_13.jpg)

*   **ビジョン駆動 (V; Vision)**
    *   **時間軸**: 長期（2〜5年）
    *   **役割**: 長期的な戦略やビジョンを具現化し、プロダクト全体の方向性を作る「羅針盤」。将来的に取れる市場面積そのものを広げる。
*   **イネーブルメント駆動 (E; Enablement)**
    *   **時間軸**: 中期（3ヶ月〜2年）
    *   **役割**: 開発効率を担保し、安定性を確保する。スピードを持続させ、開発効率を最大化するための基盤作り（認証基盤、権限制御など）。
*   **オポチュニティ駆動 (O; Opportunity)**
    *   **時間軸**: 短期
    *   **役割**: 顧客や市場からの機会（Opportunity）に直接対応し、新しい価値を素早く創造・検証・提供する（MVP、Growth施策）。

### 3. 異なる駆動をつなぐ：テンションとクラッチ
異なる速度で動く駆動力が接続されると、そこに「テンション（歪み）」が発生します。これを吸収するために「クラッチ（境界）」が必要です。

![異なる時間軸を調整するクラッチ](https://files.speakerdeck.com/presentations/590c73844aac4ef997b5a8afd5cb8dd8/slide_24.jpg)

*   **テンション**: 異なる時間軸を持つ関心事が混在することで生じる歪みや摩擦。これは「速度差があり、境界の設計が必要だ」というアラートでもあります。
*   **クラッチ**: テンションを吸収する境界。
    *   **技術的クラッチ**: API (REST, gRPC)、非同期化（キューイング）、イベント駆動、マイクロサービス、共有ライブラリ、フィーチャーフラグなど。
    *   **組織的クラッチ**: チームトポロジー（プラットフォームチーム、ストリームアラインドチームなど）による責任分界点の明確化。
*   **トレードオフ**: クラッチ（分離）にはコストがかかります。テンションが高い（速度差が大きい）場合はマイクロサービスなどの重厚な境界が有利ですが、テンションが低い場合に重厚な境界を設けると逆効果になります。

![テンションの変換モデル](https://files.speakerdeck.com/presentations/590c73844aac4ef997b5a8afd5cb8dd8/slide_28.jpg)

### 4. 実践事例

#### 事例1: 基幹製品のリアーキテクチャ (Loglass)
*   **課題**: 創業から6年が経過し、モノリシックな構造の限界が露呈。短期・長期の速度差を意識できず、製品進化が減速していた。
*   **アプローチ**:
    1.  **ビジョン駆動で開始**: 既存開発と距離を置き、専任チームで「正しく作る」ことに集中。
    2.  **適合設計**: ビッグバンリリースのリスク（現実との乖離）を避けるため、理想が現実に耐えうるかを検証する「適合設計」フェーズを経由。
    3.  **並走運用**: 最終的に部分的なリプレースと並走を選択。新アーキテクチャ（ビジョン駆動）と既存プロダクト（オポチュニティ駆動）を接続して運用。

#### 事例2: 全社横断の共通基盤
*   **課題**: 既存製品、新規製品、組織拡大など、複数の大きな変化が同時に進行し、整合を取るのが困難な状況。
*   **アプローチ**:
    1.  **イネーブルメント駆動で初速をつける**: まずは「進度の速い部分」をイネーブルさせる（共通ユーザー管理、データモデルの提供など）ことに集中し、摩擦を減らして自由度を獲得。
    2.  **ビジョン駆動へのギアチェンジ**: 基盤が整ってきた段階で、「どんな世界を実現したいか」という意味・目的を付与し、統合ロードマップ策定へと移行。

### 5. 結論：マルチドライブアーキテクチャの3原則
1.  **分離の原則 (Separation)**: 異なる時間軸を混ぜない。まずは時間軸で切り分ける。
2.  **駆動の原則 (Drive)**: 3つのエンジン（V, E, O）を持ち、それぞれを最適な速度で自律的に回す。
3.  **接続の原則 (Connection)**: テンションを恐れず、適切なクラッチ（技術的・組織的境界）を設計してつなぐ。

## 概要

LayerX社の杉野和馬氏が2026年2月26日のイベント「開発生産性のその先へ、AI生産性について語りたい」で発表した資料。杉田玄白の「腑分け（解剖）」に倣い、AI時代の生産性を構造的に解剖することを試みる。国内外40超の事例調査をもとに、**個人は速くなったが組織は速くなっていない**という逆説の原因を6層の生産性レイヤーモデルで可視化し、組織が取るべき診断・介入のフレームワークを提示する。

## 主要なトピック

### 1. AI開発の現在地

国内外40超の事例を調査し、4つの特徴的カテゴリを抽出：

- **エージェント型開発の台頭**: Copilot/Tabnine → Cursor → Claude Code/Devin → Bolt/Lovable。「人間が書きAIが補う」から自律実行へ進化。Kiroは上流（仕様）への進出（萌芽的）
- **品質・計測領域の動き**: 品質（Qodo / Greptile / Snyk）、計測（LinearB / Faros AI / DORA 2024, 2025 / GitClear）
- **AIがAIを書く**: Anthropic 70-90% / Microsoft ~30% / Salesforce ~30% / Google全域。Boris Cherny：1日22-27 PR、2ヶ月以上手動コーディングなし
- **大規模エンタープライズ**: Goldman Sachs：46,000人展開、S-1 filing 95%をAI生成。Booking.com：+16% / Intercom：41%時間節約

### 2. AI投入解剖図 ── 2軸のモデル

事例の相対位置を比較するための共通座標軸：

| 軸 | 内容 |
|---|---|
| **縦軸（ライフサイクル工程）** | 戦略 → 要件定義 → 設計 → 実装 → テスト・QA → デプロイ → 運用 |
| **横軸（スコープ）** | 個人 → チーム → 組織 |

ヒートマップにプロットすると、**実装 × 個人が突出**し、熱狂の中心は「設計〜テスト・QA」×「個人〜チーム」に集中している。

#### ツール進化の構造

コード補完（近傍コンテキスト予測）→ IDE統合（リポジトリ全体を見渡す編集）→ エージェント（計画〜テストまで自律実行）→ 並列実行（複数タスク同時処理・PR提出）

#### 開発方式の分化

- **SDD（Spec-Driven Development）**: 仕様書起点でAIが実装を生成（Kiro）
- **TDD with AI**: テスト先行、AIがテストを通す実装を生成
- **対話型反復**: 方針を伝え出力を見てフィードバック（多くの開発者が暗黙に採用）
- **自律実行**: Sub-agents / Background Agents でタスクを投げて結果を受け取る

### 3. 熱狂と齟齬 ── 逆説的な現象

複数の独立した調査が同じ逆説を報告：

| 調査 | 発見 |
|---|---|
| **DORA 2025**（約5,000名） | AI導入に対してデリバリー安定性は**負の相関** |
| **METR RCT**（16名の経験豊富なOSS開発者、246タスク） | AIを使うと実際には**-19%遅くなった**（主観は+20%） |
| **Faros AI**（10,000人超） | 個人PR数↑だが**組織デリバリー横ばい**、コードチャーン率+98%（3.1%→倍増） |
| **GitClear**（2億1,100万行） | コピペによるコード複製は**1.5倍に拡大** |
| **NBER 2026年2月**（米英独豪 約6,000名CEO・CFO） | **80%超**が「AIは生産性に影響を与えていない」 |

> **個人は速くなった（と感じている）。しかし、組織は速くなっていない。**

### 4. AI生産性の"解体新書" ── 6層レイヤーモデル

AI投入解剖図は「どこに投入したか」しか教えない。投入の結果として**組織がどの状態にいるか**を見るために、生産性を6つのレイヤーに分解：

| レイヤー | 名称 | 層間の抵抗（≠の関係） |
|---|---|---|
| **L6** | 事業価値層 | 速く届ける ≠ 正しいものを届ける |
| **L5** | 組織デリバリー層 | チームの速度 ≠ 組織の速度 |
| **L4** | チーム・プロセス層 | 接続改善 ≠ チームの成果物 |
| **L3** | 接続・調整層 | 個人の速さ ≠ 接続の質 |
| **L2** | 個人生産性層 | ツール利用量 ≠ 生産性 |
| **L1** | ツール導入層 | ─ |

#### 層間抵抗の本質

| 抵抗 | 説明 |
|---|---|
| 入力の量は出力の質を保証しない | L1→L2 |
| 部分の高速化は接続点の負荷を増大させる | L2→L3 |
| インターフェースの改善は構造の改善ではない | L3→L4 |
| サブシステムの速度はシステム全体の速度を決めない | L4→L5 |
| 効率と有効性は独立した変数である | L5→L6 |

**多くの組織はL2（個人生産性）まで到達したが、L3（接続・調整層）を天井に感じている。**

よくある症状：
- 自分は確かに速くなった。だがチームの出荷ペースは変わらない
- PRはすぐ出せるのに、マージまでが遠い
- デプロイの準備はできているのに、承認が降りない
- リリース頻度は上がったのに、ユーザーの反応は変わらない

### 5. 組織の"基礎体力" ── 複数の投入面

AI投入面だけを濃くしても上位レイヤーには到達できない。共通の軸をもつ**組織の基礎体力**が多面的に掛け合わさる：

#### 開発プロセスの基礎体力

| 投入面 | 具体例 |
|---|---|
| **自動化** | CI/CD、テスト自動化、IaC |
| **プロセス設計** | アジャイル、WIP制限、バッチサイズ縮小 |
| **組織設計** | Team Topologies、職能横断チーム、知識共有 |
| **計測** | DORA Metrics、バリューストリームマッピング |

#### 組織全体のシステム構造（開発領域外）

| 要素 | 内容 |
|---|---|
| **外部環境** | 市場、技術、競合、規制 |
| **経営層インターフェース** | 技術投資の意思決定、AI活用方針の明文化と全社共有 |
| **リーダー自身のシステム** | リーダーシップスタイル、エネルギー管理 |
| **戦略・計画・価値観** | ユーザー中心のフォーカス、「何を作るか」の意思決定 |
| **コミュニケーション** | 知識共有、社内コミュニケーション設計、組織間ネットワーク |
| **組織運営システム** | ミーティング体系、意思決定基準、検証を伴う信頼の構築 |
| **人材ライフサイクル** | 採用→OB→成長→評価→退職 |
| **測定・フィードバック** | 全レイヤーのミスマッチを可視化 |

#### Booking.comの成功事例

スループット+16%を達成した背景には、AI活用だけでなく：
- 計測（DORA Metrics + VSM）
- プロセス設計（リーンな改善サイクル）
- 自動化（成熟したCI/CD）
- AI活用（3,500人超へ展開）

→ 複数工程にわたって計測・プロセス設計・自動化が厚く敷かれていたことでL4（チーム・プロセス層）に到達。

### 6. AIはミラーとマルチプライヤーである（DORA 2025）

> AIは組織の基礎体力を映す鏡だ。プロセス設計、組織設計、計測といった基盤が厚い組織にAIを重ねると、掛け合わせが正の方向に作用し、レイヤーの到達が加速する。だが基盤の薄い組織にAIだけを重ねると、既存の問題を増幅する。

- レビュープロセスが機能していなければ → レビューキューが膨れ上がるだけ
- テスト自動化が不十分なら → コードチャーンが増えるだけ

**強い組織はAIによってさらに強くなり、弱い組織はAIによってさらに弱くなる。**

### 7. AI生産性の"基礎診断" ── 4ステップの介入フレームワーク

| ステップ | 内容 |
|---|---|
| **1. 現状の診断** | 3つのスケール（AI投入、開発プロセスの基礎体力、組織のシステム構造）の入力を重ね、どの投入面が薄く、どのレイヤーで詰まっているかを可視化 |
| **2. 問題の構造化** | ボトルネックはマトリクスの中か外か。バリューストリームで「作業」か「待ち」かを判別 |
| **3. 介入点の選択** | 介入には深さがある。ツール追加（浅い）→ チーム構造を変える（中間）→ 評価制度・前提を変える（深い） |
| **4. レバレッジの設計** | 単一面ではなく**複数面の掛け合わせ**で上位レイヤーへの到達を設計。制約は移動するため、継続的に追い続けるリズムを設計 |

#### 介入の具体例

| 投入面 | 表層的アプローチ（×） | 本質的アプローチ（○） |
|---|---|---|
| **自動化** | 手作業をスクリプトに置き換える | 「なぜその手作業が存在するのか」を問い直し、待ちの発生源を設計から除去 |
| **組織設計** | チーム間の調整の質を上げる | 調整が不要になるようにチーム境界を引き直す |
| **組織運営** | 承認プロセスを速くする | 「なぜその承認が必要なのか」の前提を問い直し、検証を伴う信頼に置き換える |

#### レバレッジの設計例

- **自動化**: CI/CD × テスト自動化 × 計測を同一工程に重ね、コード生成の高速化がデリバリーに変換される経路を設計
- **組織設計**: チーム境界の再設計 × 知識共有 × AI活用を重ね、L3の天井を引き上げ
- **組織運営**: 意思決定の分権化 × 計測の定期化 × 診断サイクルの組織リズム化

## 結論・示唆

### 著者の結論

1. AI投入は「実装 × 個人」に集中しており、多くの組織はL2（個人生産性）止まり
2. 個人の速さは組織の速さに自動変換されない。層間に「抵抗」が存在する
3. AIは組織の**ミラー（鏡）**かつ**マルチプライヤー（増幅器）**。基礎体力が厚い組織では正に、薄い組織では負に作用する
4. 上位レイヤーへ到達するには、AI投入面だけでなく、自動化・プロセス設計・組織設計・計測の**複数面を掛け合わせる**必要がある
5. 開発プロセスの外側にある**組織全体のシステム構造**（経営層インターフェース、戦略、コミュニケーション等）も投入面の厚みを規定する
6. 診断→構造化→介入点選択→レバレッジ設計の4ステップで、継続的に制約を追い続けるリズムを設計せよ

### 実践的な示唆

- AI導入の効果を最大化するには、まず組織の基礎体力（CI/CD、テスト自動化、計測、プロセス設計）を整えることが前提
- L3（接続・調整層）の天井を突破するには、チーム境界の再設計や知識共有の仕組みなど「構造」への介入が必要
- 表層的な改善（ツール追加、プロセスの高速化）ではなく、「なぜその仕組みが必要なのか」という前提を問い直す深い介入が効果的
- 制約は一箇所解消すると次に移動する。一度きりの改善ではなく、継続的な診断サイクルを組織のリズムとして組み込むべき

## 要旨

意思決定を伴わないコード品質チェックについては、人間のレビューは不要になったという主張。AIエージェント（Codex gpt-5.4 と Composer 2）を組み合わせ、**ハーネスエンジニアリング**の考え方で安定したレビューワークフローを構築した実践記録。

---

## 人間のコードレビューをやめた理由

- 人間のレビュワーはコードベースの一部しか把握しておらず、疲れ・遠慮・見落としがある
- AIはコードベース全体を読み、一貫した基準で指摘を出せる
- AIが数分で書いたコードを、人間が数時間〜数日かけてレビューするのはボトルネック
- hentekoの「[コードレビューをなくすことを考える](https://zenn.dev/henteko/articles/871526c9fffcec)」に共感

---

## ハーネスエンジニアリングとは

HashiCorp創業者の **Mitchell Hashimoto** が[自身のブログ](https://mitchellh.com/writing/my-ai-adoption-journey)で提唱し、OpenAIのCodexチームの[記事](https://openai.com/index/harness-engineering/)で広まった概念。AIエージェントが安定して成果を出せるように、**モデルの外側にある環境・制約・フィードバックループを設計する技術体系**。「ハーネス（harness）」は馬具の意味で、強力だが予測しづらい力を正しい方向に導く装備一式を指す。

### ハーネスの4つの役割

| 役割 | 説明 |
|---|---|
| **制約（Constrain）** | エージェントに何ができるかを制限する |
| **情報提供（Inform）** | エージェントに何をすべきかを知らせる |
| **検証（Verify）** | エージェントが正しく実行したかを検証する |
| **修正（Correct）** | エージェントが間違ったときに修正する |

### 実績

LangChainのコーディングエージェントが、**モデルを一切変えずにハーネスだけを改善**することでベンチマークスコアを **52.8% → 66.5%** に向上させた[事例](https://blog.langchain.com/the-anatomy-of-an-agent-harness/)がある。モデルの性能ではなく、モデルの外側の設計が結果を決める。

## 実際のレビューフロー（5ステップ × 最大6ループ）

以下を1ループとし、指摘がゼロになるまで最大6回回す。通常 **2〜3回で収束** する。

### Step 1: レビュー（Codex / gpt-5.4）

- 現在のブランチとmainの差分をCodexの `review` モードでレビュー
- 構造化された findings を返すため、指摘件数をプログラマブルに判定可能（ループの終了条件）

### Step 2: トリアージ（Composer 2）

- Codexの指摘をそのまま全部直すと問題が起きるため、Composer 2でフィルタリング
- **severity分類**: CRITICAL / IMPORTANT / LOW に分類し、LOWは捨てる
- **オシレーション検出**: commit diff からA→B→Aの振り子パターンを検出し、優れた方を directive として固定
- **スコープ判定**: 変更していない既存コードへの指摘は対象外として除外

### Step 3: 修正（Codex / gpt-5.4）

- トリアージを通過した指摘を受けてCodexが修正
- directive がある場合はそれに従う義務がある（オシレーション防止）
- プロジェクト内のコーディングガイドラインを参照し、判断基準をプロンプトにハードコードしない

### Step 4: バリデーション（Composer 2）

- 別のComposer 2セッションで修正の適切さを検証
- 根本原因に対処しているか、バンドエイド修正になっていないかをチェック
- 不十分ならComposer 2自身が再修正

### Step 5: コミット

- 修正があればコミットし、Step 1に戻る

## CodexとComposer 2の使い分け

| エージェント | 役割 | 理由 |
|---|---|---|
| **Codex (gpt-5.4)** | レビュー・修正 | レビュー精度が高く、バグやロジックの問題を的確に検出。修正能力も高い |
| **Composer 2** | トリアージ・バリデーション | Codexは精度は高いが遅い。Composer 2はコードベースの探索が速く、複数ファイルを横断的に読むのが得意。「広く読んで素早く判断する」タスクに向いている |

---

## Codexの過剰指摘への対策

Codexのレビューは真面目すぎるため、以下の問題が起きる：

- 実際には起きない過剰なエッジケースへの指摘
- 過剰な後方互換性維持の要求
- 過去にトレードオフを受け入れた修正を元に戻そうとする指摘

### 3つの対策

1. **トリアージ**: severity分類でLOWを捨てる
2. **オシレーション検出**: commit diffからA→B→Aパターンを検出し、directive で片方に固定
3. **バリデーション**: 修正が根本原因に対処しているかを別セッションで検証

各ステップのプロンプトからはプロジェクト内のコーディングガイドラインを参照させ、判断基準をプロンプトにハードコードしない設計にしている。

## 人間のレビューが完全に不要になったわけではない

以下のレベルの判断は人間がやるべき：

- 今いちばん大事な課題は何か
- それをどういう機能として落とし込むのか
- 何をやって、何をやらないのか

→ hentekoの記事でいう「**事前のコンセプト確認**」にあたる部分。

自動化したのはその先の話で、**方向性が決まった後のコード品質チェック**をAIに任せている。

## ワークフローの定義ツール「Rigg」

以前は手動でターミナルを行き来していたが、自動化のために **[Rigg](https://github.com/tryrigg/rigg)** というCLIを作成。

### Riggの特徴

- Codex CLI、Claude Code、Cursor CLIといったローカルAIエージェントCLIを組み合わせたワークフローをYAMLで定義
- エージェント間の出力の受け渡し、ループ・分岐などの制御フローを記述可能
- `.rigg/` ディレクトリにYAMLを置いてGit管理 → チーム全員が同じワークフローを実行可能
- 実際のレビューワークフロー: [review.yaml](https://github.com/tryrigg/rigg/blob/main/.rigg/review.yaml)

### インストールと実行

```bash
curl -fsSL https://tryrigg.com/install | bash
rigg init
rigg run review
```

### ワークフロー改善のフィードバックループ

- 最初から完璧なワークフローは作れない
- 問題が起きるたびに追加で改善していく
  - 「Codexの指摘が過剰でループが止まらない」→ トリアージを追加
  - 「修正が振り子になる」→ オシレーション検出を追加
- YAMLで定義しておくことで改善がGitの差分として残り、変更履歴を追える

---

## 重要な結論

1. **モデル単体の能力に頼らず、モデルの外側の設計（ハーネス）で品質を安定させる**ことが重要
2. **複数のAIエージェントを役割分担**させ、レビュー・トリアージ・修正・バリデーションの各フェーズで最適なモデルを使う
3. **ワークフローをYAMLで宣言的に定義**し、Git管理することでチーム共有と継続的改善を実現
4. 人間は「何を作るか」の意思決定に集中し、コード品質チェックはAIに委譲する
5. ハーネスエンジニアリングは**フィードバックループを回して改善し続ける**ことが本質

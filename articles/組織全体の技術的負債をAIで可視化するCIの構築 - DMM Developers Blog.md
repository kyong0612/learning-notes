## 概要

DMM.comのプラットフォーム開発本部コード品質チームが、AIを活用した技術的負債の分析ツール「Modifius（モディフィウス）」と、それをGitHub Actionsで組織全体に展開する「Modifius CI」を構築した事例を紹介する記事。静的解析ツールでは捉えきれない「変更容易性」の構造的課題を、Claude CodeとMCPサーバーの連携により分析し、120名以上のエンジニアが関わるプラットフォーム全体の技術的負債を定期的・定量的に可視化する仕組みを実現した。

## 主要なトピック

### 背景：DMMプラットフォームと技術的負債の課題

- **DMMプラットフォーム**は決済（月300億円規模）、ID基盤（4,000万人）、ポイント基盤（月100億円規模）、不正対策など14のプロダクトで構成される全サービス共通基盤
- プラットフォーム全体で**120名以上のエンジニア**が開発に関与
- サービス規模の拡大に伴い、**変更容易性が低いコード**（技術的負債）による開発生産性低下が問題化
- 「コード品質チーム」はミノ駆動**1名のみ**で運営 → スケールの限界

### 静的解析ツールの限界

従来の静的解析ツール（コード行数、サイクロマチック複雑度、引数の数、依存の度合い等）は指標として一定有用だが、変更容易性の**構造的観点**での評価には不十分：

- 同じ関心事でまとまっているか
- 異なる関心事が混在していないか
- 抽象レベルが揃っているか

→ 真の意味で負債かどうかの判断、解消コストの見積もりが困難だった

### Modifius（モディフィウス）の開発

- **変更容易性**に関して包括的な支援機能を提供する**MCPサーバー**
- 名前の由来：modifiability（変更容易性）+ ius（ラテン語）＝「変化を司る者」
- 主な機能：
  - 高精度な技術的負債分析
  - 設計改善提案
  - 高品質なテストコード実装
- 実態は変更容易性に関するプロンプトを用途に合わせて加工し返すサーバー
- 知見のベースは著書『改訂新版 良いコード／悪いコードで学ぶ設計入門』

**AIの優位性**：コードの意図を推論できるため、ドメインロジック・DBロジック・フロントエンドの区別、関心事の違い（予約注文 vs 抽選注文など）を判別可能。

**分析結果の例**：ゲームの攻撃処理コードに対し、以下のような欠陥を検出：

| 欠陥モジュール | 分離すべき関心 | 欠陥スコア |
|---|---|---|
| AttackUseCaseクラス | 武器の耐久性管理 | 1 |
| AttackUseCaseクラス | スペシャルゲージ管理 | 1 |
| AttackUseCaseクラス | ダメージ計算のビジネスルール | 1 |

### Modifius CIの構築

#### CI化の狙い（3点）

1. 各チームが自リポジトリの負債状況を**定量的に把握**できるようにする
2. 共通指標で**横断的に比較**し、改善の優先度付けを支援する
3. 定期実行による**変化の追跡**を可能にする

#### アウトプット

- **Slack通知**：CIの定期実行後、分析結果のサマリーをチームに通知
- **HTMLレポート**：GitHub Pagesにデプロイ、ブラウザ上で欠陥スコアやファイルごとの詳細を確認
- **ダッシュボード**：スプレッドシート＋GASで全リポジトリの欠陥スコアを俯瞰

#### パイプライン構成（8ジョブ）

GitHub Actionsの**Reusable Workflow**として設計：

1. `parse-config` — 設定ファイル読み込み
2. `prep` — バリデーション、対象ファイル列挙、MCPサーバーセットアップ、バッチ分割
3. `analyze` — Claude Code + Modifius MCPで負債分析（中核）
4. `aggregate` — 欠陥スコア集計
5. `build` — HTMLレポート生成
6. `deploy` — GitHub Pagesデプロイ
7. `notify` — Slack通知
8. `update-spreadsheet` — スプレッドシート自動転記

#### 利用側で必要なファイル

1. `.github/modifius-config.yml`（分析対象・通知先定義）
2. ワークフロー呼び出しファイル（スケジュール実行設定）

### 技術スタックと仕組みの裏側

- **Claude Code Base Action + Modifius MCP**：prepでMCPサーバー起動 → analyzeでファイルごとにClaude CodeがMCPツールを呼び出し分析 → Markdownテーブル形式で出力
- **LiteLLM基盤**で推論、temperatureなどのパラメータ調整可能
- **バッチ分割と並列実行**：matrix strategyの上限（256個/ジョブ）に対応し、256ファイルごとにバッチ分割、各バッチ内で最大5並列実行
- **レポート**：GitHub Pagesにデプロイ、過去の実行履歴も保持し時系列追跡可能
- **Slack通知 + スプレッドシート連携 + GAS**：複数リポジトリの横断的確認

### 導入過程で工夫した点

#### 分析結果の繰り返し精度（再現性）

定点観測のため、LLMの推論パラメータで決定性を最大化：

| パラメータ | 値 | 効果 |
|---|---|---|
| temperature | 0 | 確率分布のシャープさを最大 |
| top_k | 1 | 最も確率の高いトークンのみ選択 |
| top_p | 1（または未指定） | top_kに委ねる |

#### 運用コストの設計

- 非キャッシュ時：1ファイルあたり約**$0.27**
- PR時ではなく**隔週〜月1回のスケジュール実行**を推奨
- 分析対象は最大**50ファイル**に制限
- 変更頻度が高く重要度の高いファイルに絞って分析

#### Prompt Caching の効果（約12日間の実績）

- キャッシュ利用率：約**88.6%**
  - Cache Read Tokens：52,566,550
  - Cache Creation Tokens：6,726,924
- コスト削減率：約**88.7%**（約90%削減）
  - キャッシュなし想定：約$530 → 実際：**$58.51**
- 20個前後のプロダクトに導入した状態で計測

#### レート制限とリトライ

- `max-parallel: 5`で同時実行数を制限（10個以上の並列で高確率でレート超過エラー発生）
- 分析ステップでClaude Code Actionを**2回実行**する構成（1回目失敗時に自動で2回目を実行）

#### 導入のセルフサービス化

- 設定テンプレートとオンボーディングガイドの提供
- Slackワークフローによる利用申請の自動化
- 専用問い合わせチャンネルの開設

### 導入成果

- プラットフォーム内の複数リポジトリに対して技術的負債の**定期的な可視化**を実現
- HTMLレポートにより分析結果の**チーム内共有**が可能に
- ダッシュボードで複数リポジトリの欠陥スコアを**横断的に把握**
- 共通指標で改善の**優先度付けの議論**が容易に

### 利用者の声：DMMポイントクラブの事例

- DIコンテナ構築のボイラープレートコードが1クラスに集中（Dagger2の制約でやむを得ないと認識）
- Modifius CIの分析で「30以上のDaggerコンポーネント生成と依存関係構築ロジックが混在」として**欠陥スコア約200**が付与
- DIツールの置き換え（Dagger2 → **Dagger Hilt**）を決定・完遂
- 数年前からIssueとして積まれていたが、数値として問題が明確になりメンバー全員の合意を得られた
- 置き換え後、該当クラスの欠陥スコアが**大きく減少**

### 今後の展望

- 欠陥スコアを元にした**リファクタリング計画の策定と実施**
- PR差分を対象とした**負債検知機能**の開発
- カスタムプロンプトによる**分析の拡張**

## 重要な事実・データ

- **DMMプラットフォーム規模**：14プロダクト、120名以上のエンジニア、決済月300億円、ID4,000万人
- **コード品質チーム**：ミノ駆動1名のみ
- **分析コスト**：非キャッシュ時$0.27/ファイル、Prompt Cachingで約90%削減（$530→$58.51）
- **Prompt Caching利用率**：約88.6%（12日間、20プロダクト導入時）
- **並列実行制限**：max-parallel: 5（10以上で高確率のレート超過エラー）
- **分析ファイル上限**：50ファイル/回（コスト効率考慮）
- **導入事例**：DMMポイントクラブでDagger2→Dagger Hiltへの移行を完遂、欠陥スコア大幅減少

## 結論・示唆

### 著者の結論

AIの推論能力を活かすことで、静的解析ツールでは捉えきれなかった変更容易性の構造的な課題を分析できるようになった。CIとして自動化することで、組織全体の技術的負債を定期的・定量的に可視化する仕組みを実現。「見える化」により問題提起と改善の優先度付けがしやすくなった。

### 実践的な示唆

- **MCPサーバー + Claude Code**の組み合わせにより、ドメイン知識を組み込んだAI分析基盤を構築可能
- **Prompt Caching**の効果は絶大（約90%のコスト削減）で、大規模組織での運用を現実的にする
- LLMの**非決定性**への対策（temperature=0, top_k=1）は定点観測型CIで必須
- 定量的な指標（欠陥スコア）の提示が、数年間放置されていた改善課題を**組織的な合意形成**に導ける
- Reusable Workflowによるセルフサービス型の展開で導入障壁を下げる設計が重要

## 制限事項・注意点

- 導入のセルフサービス化の整備が追いついておらず、運用負荷の軽減が課題
- コスト制約から、PR作成・マージ時ではなく隔週〜月1回のスケジュール実行を推奨
- 分析対象は最大50ファイルに制限（仕組み上は1,000ファイル規模にも対応可能）
- AIによる分析は非決定的な挙動があり、パラメータ調整で可能な限り再現性を高めている
- 現段階は「負債の可視化」までであり、自動的なリファクタリング実行には至っていない

## 概要

AI によるコード生成が急速に普及する中、従来の人間によるコードレビューはもはや持続不可能になりつつある。本記事では、コードレビューを「殺す」ために、人間のチェックポイントをコードの下流から仕様の上流へ移動させるべきだと主張している。

> *"We're not going to outread the machines. We need to outthink them—upstream, where the decisions actually matter."*

---

## 人間はもうすべてのコードを読めない

AI導入率の高いチームのデータ（10,000人以上の開発者、1,255チーム対象）:

| 指標 | 変化 |
|---|---|
| 完了タスク数 | **+21%** |
| マージされたPR数 | **+98%** |
| PRレビュー時間 | **+91%** 増加 |

指数関数的に増加しているのは、変更の**数**と変更の**サイズ**の両方である。開発者たちは「AI生成コードのレビューは同僚が書いたコードのレビューより労力がかかる」と報告している。手動コードレビューでこの戦いに勝つ方法はない。

## AIコードレビューもまだ「レビュー」でしかない

AIコードレビューツールは時間を稼いでいるに過ぎない。AIがコードを書き、AIがレビューするなら、なぜそのためのUIが必要なのか。AIレビューの価値はイテレーションループにあり、承認ゲートとしてではない。

> エージェントは常に信頼できるわけではなく、「AIが愚かなことをしたのを見つけた→だから常にチェックしなければ」という本能は理解できる。しかし現在のスケールでは手動検証はもう実行不可能である。

## コードのレビューから意図のレビューへ

人間のチェックポイントを上流に移す。チェックポイントは以前にも移動したことがある（ウォーターフォール → CI）。再び移動させることができる。

**Spec駆動開発**が主流になりつつある:

- **仕様（Spec）**がソースオブトゥルースになる
- **コード**は仕様の成果物になる
- コードをレビューする必要はない。**ステップ**をレビューし、**検証ルール**をレビューし、コードが満たすべき**契約**をレビューする

人間の最も価値ある判断は、最初の1行が生成される**前に**行使される。

## 信頼をレイヤーで構築する（Swiss-cheese モデル）

単一のゲートですべてを捕捉することはできない。不完全なフィルターを積み重ねて、穴が揃わないようにする。

### Layer 1: 複数のオプションを比較

1つのエージェントに正解を求めるのではなく、**3つのエージェントに異なるアプローチで試させて最良の結果を選ぶ**。選択基準:

- 最も多くの検証ステップをパスしたもの
- 最小のdiffを生成したもの
- 新しい依存関係を導入しなかったもの

> オプション性のコストは、ソフトウェアエンジニアリング史上最低である。

### Layer 2: 決定論的ガードレール

LLMに「うまくいった？」と聞くのではなく、**pass/failの成果物を生成する検証ステップ**を定義する。エージェントは失敗するテストと交渉できない。

ガードレールの階層:

1. **コーディングガイドライン** — カスタムリンター
2. **組織全体の不変条件** — 例: ハードコードされた認証情報・APIキー・トークンの禁止
3. **ドメイン契約** — 例: Payments ドメインでは全金額に Money 型を使用
4. **受け入れ基準** — タスク固有

**重要**: 検証ステップはコードが書かれる**前に**定義すべきであり、後から確認のために発明するものではない。エージェントがコードとテストの両方を書くと、問題を移動させただけになる。

### Layer 3: 人間が受け入れ基準を定義する

**BDD（振る舞い駆動開発）**が新たに重要になる。BDDは常に良いアイデアだったが、コードも書くのにスペックを書くのは余計な作業に感じられて普及しなかった。

エージェントの時代では方程式が反転する:

- スペックは余計な作業ではなく、**主要な成果物**になる
- エージェントが実装し、BDDフレームワークが検証する
- 何か失敗しない限り、実装を読む必要はない

> 人間が書いた受け入れ基準を、機械が検証する。それが本当に重要なゲートである。

### Layer 4: 権限システムをアーキテクチャとして

エージェントのスコープは可能な限り狭くすべき:

- タスクが「`utils/dates.py` の日付パースバグ修正」なら、そのファイルとテストファイルだけにアクセスを制限
- エスカレーショントリガー: 認証ロジック、DBスキーマ変更、新しい依存関係追加など → エージェントの確信度に関係なく自動的に人間レビューにフラグ

### Layer 5: 敵対的検証

責任の分離: あるエージェントが作業し、別のエージェントが検証する。互いを信頼しない設計。

- コーディングエージェントは検証エージェントが何をチェックするか知らない
- 検証エージェントはコードを変更する能力を持たない
- 3番目のエージェントがエッジケースと失敗モードを狙って破壊を試みる（Red team / Blue team の自動化）

## 結論: 「良いコード」の定義が変わる

エージェントシステムのインセンティブは単純: タスクを完了できるか？依頼者を満足させられるか？長期的な正確性やビジネス要件はエージェント固有の動機ではない。**制約にエンコードするのは我々の仕事**である。

エージェントが生成しエージェントが読むコードにおいて、「良いコード」はより標準化される。

未来のパラダイム:

> **速く出荷し、すべてを観察し、さらに速く元に戻す。**
>
> ~~ゆっくりレビューし、どのみちバグを見逃し、本番でデバッグする。~~

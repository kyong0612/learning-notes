## タイムスタンプ付き要約

### [00:00] イントロダクション

- Gergely OroszがMartin FowlerとKent Beckを紹介
- 1週間前にMartin主催の「Future of Software Engineering」カンファレンス（ユタ州ディアバレー）に参加したばかり
- 25年前のアジャイルマニフェスト作成を振り返りつつ、現在のAI時代を議論する流れを設定

### [02:11] アジャイル・TDD・リファクタリングから定着したアイデア

- **Martin Fowler**: リファクタリングやアジャイル全般について感謝されることが多いが、特定の一つの要素が突出しているわけではない
- **Kent Beck**: TDDについては二極化した反応を受ける
  - 「TDDに感謝しています」と言われる一方、「TDDが人生を台無しにした」とも言われる
  - **AI時代のTDD**: AI推進者からも「20年間TDDを推進してくれて感謝。AIエージェントの時代に本当に重要になった」という声
  - 引用: *「大きな力を持つジーニーを手に入れたら、それが正しいことをしているか検証する方法を学ばなければならない。それを我々は25年間実践してきた」*

### [09:04] 歴史的アナロジー：マイクロプロセッサ、インターネット、OO、アジャイル

- **Martin Fowler**:
  - 「AIほどの規模で衝撃を与えたものはない。まったく異なるスケール」
  - 過去の変化との比較:
    - **オブジェクト指向言語**: 多くの人を怖がらせた（推進者側だったので自分たちはそうでもなかった）
    - **インターネット**: 巨大な影響（当時は重要性を認めない人もいた）
    - **アジャイル**: 企業が抵抗した度合いで影響の大きさがわかる
  - AIの違い: 重要性を否定する人がいない。「ブリンカーをかけてAIの重要性を否定することはできない」

- **Kent Beck**:
  - **マイクロプロセッサ（Intel 4004、1971年）との類比**:
    - それ以前、コンピュータは巨大な箱。もう一台欲しければ家を抵当に入れる必要があった
    - Intel 4004を見て「待って、これがコンピュータ？」と衝撃を受けた
    - ソフトウェアの書き方とハードウェア設計を理解すれば、想像もしなかったことが可能になった
  - AIも同じ「想像力の拡張」
  - 引用: *「私は途方もなく野心的なプロジェクトを書いている。永続的Smalltalkに取り組み、Rustのライブラリ品質のコードを書いている」*

#### 懐疑主義と好奇心のバランス

- **Martin Fowler**:
  - 常に「ハイプに飛びつく人」と「たいしたことないと言う人」の混在がある
  - 「懐疑主義は絶対的で完全でなければならない。つまり、自分の懐疑主義に対しても懐疑的でなければならない」
  - GitHub Copilotの初体験は低評価だった。たまに素晴らしいものを出すが、大半はゴミだった
  - しかし初期の印象だけで判断してはならない。Simon Willison のブログなどを通じて「うまく使うには学ぶ必要がある」と気づいた
  - 引用: *「良いことと悪いことのバランスを示し、『わからない』と言える人の話こそ聞く価値がある」*

- **Kent Beck**:
  - 引用: *「最小限の実験を実行して、この主張が正しいかどうか自分自身が満足できるレベルで検証するスキルが、この1年で1000倍も価値を持つようになった」*
  - AI能力は週ごとに変わる。先週ダメだったものが今週は動く。誰も「答え」を持っていない
  - 引用: *「悪いニュースは誰も答えを持っていないこと。良いニュースも誰も答えを持っていないこと。全員が同じくらい無知なのだから、全員同じくらい賢い」*

### [16:44] なぜ良い慣習でも失敗するのか — アジャイルとAIの類似点

- **Kent Beck**:
  - 引用: *「人々は実は、より速く・より安く・より良いものを望んでいない」*
  - 企業内のインセンティブが実際の改善と乖離している
  - 「40%良くなった、12%安くなった」と言っても、それが組織内のインセンティブと合わなければ罰せられる
  - AIが同じことを約束するなら、同じ反応が繰り返される

- **Martin Fowler**:
  - AIはアジャイルとの類似点がある：
    1. **うまく使う人と使えない人の差が大きい**
    2. **スネークオイル産業の出現**（「アジャイル産業複合体」の再来）
  - 違いは規模とスピード

#### AIは増幅器

- **Kent Beck**:
  - 引用: *「AIは増幅器だ。若くて学習が速い人にとって、AIはそれを増幅する。今はジュニアプログラマーの黄金時代だ」*
  - 大工と電動ノコギリの比喩: 電動ノコギリが登場したからといって誰でも家を建てられるわけではない。より強力なツールを手にしたということ

#### 「中間層」の問題

- **Kent Beck**:
  - お金のためにプログラミングに入った「中間層」が最も影響を受ける
  - ドットコムクラッシュ時、この層は不動産業界などに移った。今回はさらに大きい
- **Martin Fowler**:
  - ゼロ金利時代の終わりによるリストラで、この中間層はすでにある程度「洗い流された」
  - AIブームと経済的逆風が同時に起きているのはドットコム時代にはなかった

#### 「プログラマー排除」の再来

- **Kent Beck**:
  - COBOLから始まった「プログラマー不要論」が定期的に繰り返される
  - アジャイルはプログラマーをより効果的にすることが目的だった
  - 今、AIで再び「プログラマー不要」が叫ばれている
  - 引用: *「なぜ人々は我々を排除し続けたいのか、考える価値がある。それは一部は我々プログラマー自身の問題でもある」*

- **Martin Fowler**:
  - 「コードがなくなる」と言う人がいるが、何らかの形でコードは存在し続ける
  - コードの性質は根本的に変わるかもしれないが、生成し操作する必要は残る

### [21:21] AIシフトはどう違うか — 現場で何が起きているか

- **Martin Fowler**（ThoughtWorksでの経験から）:
  - 引用: *「大規模な混乱とパニックが、全般的に起きている」*
  - 大企業は巨大で複雑なコードベースを抱えている。「100万行のコードを扱えるか」と聞かれるが、それは大企業にとって「小さい」方
  - 航空会社が1-2日オフラインになるようなリスクは許容できない
  - **セキュリティリスクへの警鐘**: LLMにメール全体のアクセス権を与える企業がある → 「NO」
  - 引用: *「今年、本当にひどいセキュリティインシデントが起きるだろう。人々は注意を払っていない」*

### [25:50] エンジニアは複数エージェントの管理者か — プログラミングの「再ソロ化」

- **Kent Beck**:
  - XPの核心は「基本的に反社会的な人々のための安全な社会環境を作ること」だった
  - 今の傾向: 「6つのエージェントを使っているから、チームを管理しているのと同じ」 → 違う
  - 引用: *「6つのツールを同時に使っているだけで、異なる信念や異なるエネルギーレベルを持つ人との会話とは全く別物だ」*
  - 以前は個室でドアを閉めてピザをドアの下から差し入れる時代があり、それが管理しやすかった
  - XPで生まれた「混沌とした社会的プロセス」が良い結果を生んだのに、今また孤立化に向かっている

### [27:11] ワンピザチーム vs ツーピザチーム

- **Martin Fowler**:
  - ツーピザチーム（5-10人）がワンピザチームになるか、それともより効果的なツーピザチームになるか？
  - 引用: *「私の賭けは、より効果的なツーピザチームの方だ」*
  - ペアプログラミングの進化:
    - 「人間 + AI」のペアか、「2人の人間 + n個のAI」のペアか？
    - 2人の人間がいれば、AIをよりうまくコントロールできる
    - モブプログラミング + AIの組み合わせも興味深い
  - 「1人 + 多数のAI」が最適解とは限らない

- **Kent Beck**:
  - 「2人の人間 + AI」でのペアプログラミングは非常にポジティブな経験
  - AIが遅いことの利点: AIがプロンプトを処理している3分間に、命名規約や条件式の表現、次のステップについて話し合える
  - 引用: *「モデルが出てくるたびに速くなる。すると『ああ、話す時間が減る』と思う」*

### [29:07] 開発者体験とエージェント体験

- **Martin Fowler**:
  - 引用: *「開発者体験（Developer Experience）とエージェント体験（Agent Experience）のベン図は円だ」*
  - うまくモジュール化されたコードはエージェントにも扱いやすい
  - 良いテストはエージェントにも人間にも役立つ
  - 引用: *「我々の技術を正しく行えば、それはエージェントにとっても機能する」*
  - ThoughtWorksの同僚Unashiの例: AIと効率的にコミュニケーションするために「ドメインについての正確な言語」を開発 → ドメイン駆動設計そのもの

- **Kent Beck**:
  - クラフトへのOCD的な楽しみを手放す必要がある
  - 引用: *「一つの関数を完璧にする満足感は、もう意味をなさない」*
  - 以前は「ゾーンに入って混沌としたファイルを小さな安全なステップで整理し、パッと焦点が合う瞬間」を楽しんでいた → もうそれはできない
  - しかし全体的な理解の開発はまだ可能
  - 引用: *「ドメインとプログラムの関連を理解することを楽しむ方向にシフトする必要がある。以前はプログラム自体がドメインだった」*

## キーメッセージ

1. **AIは増幅器**: 良い慣習も悪い慣習も等しく増幅する。うまく使う人と使えない人の差が拡大する
2. **最小限の実験スキルが最重要**: 誰も答えを持っていない今、「最小限の実験で検証する」スキルが1000倍の価値を持つ
3. **TDDの再評価**: AIエージェントの出力を検証するために、TDDの25年間の実践が今こそ活きる
4. **開発者体験 ≈ エージェント体験**: モジュール化、テスト、DDD（ドメイン駆動設計）といったクラフトの実践は、AI時代にも直接的に有効
5. **チームの社会的側面を失ってはならない**: 「1人 + 複数エージェント」は「チームを管理している」のとは別物。人間同士の対話に不可欠な価値がある

## 結論・示唆

### 話者の結論

- **Martin Fowler**: AIは過去のどの技術革新よりも大きな規模であり、懐疑主義と好奇心のバランスが不可欠。良いソフトウェア慣習（モジュール化、テスト、DDD）はAI時代にも有効であり、開発者体験とエージェント体験は重なっている
- **Kent Beck**: 現在は誰も答えを持っていない探索期であり、シニアエンジニアの役割は「答えを教える」から「答えの見つけ方を示す」へ変わった。クラフトへのこだわりからドメイン理解への楽しみにシフトする必要がある

### 実践的な示唆

- [ ] 懐疑主義と好奇心のバランスを保ち、初期印象だけで判断しない
- [ ] 「最小限の実験」で仮説を検証するスキルを磨く
- [ ] TDDを導入・強化し、AIの出力を検証する仕組みを整える
- [ ] コードのモジュール化、テスト、ドメイン言語の整備を継続する（DX = AX）
- [ ] ペアプログラミング・モブプログラミングをAIと組み合わせて実践する（2人+AI）
- [ ] 個々のコード品質への執着から、全体的なドメイン理解の深化に重点を移す

- **不健全なパフォーマンスメトリクス**: PR頻度などの計測は誤り。成果（outcomes）と結果（results）を定量化すべき
- **意図的な品質低下のリスク**: AIで速度を優先し品質を犠牲にする企業がある。長期的な技術負債の蓄積に注意
- **「未来への投資」の見えにくさ**: AIを使うと機能構築が目に見えやすくなるが、将来のための基盤投資が後回しにされがち

## 用語解説

- **XP（エクストリームプログラミング）**: Kent Beckが提唱したアジャイル開発手法。ペアプログラミング、TDD、継続的インテグレーションなどを特徴とする
- **ツーピザチーム**: Amazon起源の概念。ピザ2枚で全員が食べられる規模（5-10人）のチーム
- **スネークオイル**: 根拠の薄い商法・誇大宣伝のこと。効果の疑わしい製品やサービスを指す
- **DDD（ドメイン駆動設計）**: Eric Evansが提唱した設計手法。ビジネスドメインの知識を中心にソフトウェアを設計する。Martinが言及した「正確なドメイン言語」はDDDの**ユビキタス言語（Ubiquitous Language）**の概念に相当する
- **Intel 4004**: 1971年にIntelが発表した世界初の商用マイクロプロセッサ（4ビット）。コンピュータの小型化・普及の転換点となった

## 概要

Martin Fowlerのサイトに掲載された本記事は、Thoughtworksの Distinguished Engineer である Birgitta Böckeler による、**コーディングエージェントのユーザーが「ハーネス（harness）」をどう設計すべきか**を論じたものである。「ハーネス」とは、AIエージェントにおけるモデル以外のすべての要素（Agent = Model + Harness）を指すが、本記事ではコーディングエージェントの文脈に限定し、ユーザーが構築する**外部ハーネス（outer harness）**に焦点を当てている。

良く設計された外部ハーネスは、(1) エージェントが最初から正しい結果を出す確率を高め、(2) 問題が人間の目に触れる前にセルフコレクションするフィードバックループを提供する。これにより、レビューの負担軽減、システム品質の向上、トークン消費の削減が期待できる。

## 主要なトピック

### フィードフォワード（ガイド）とフィードバック（センサー）

ハーネスの核となる制御メカニズムは2種類に分かれる：

- **ガイド（Feedforward controls）**: エージェントが行動する**前に**、望ましくない出力を予防するもの。最初の試行で良い結果を出す確率を高める
- **センサー（Feedback controls）**: エージェントが行動した**後に**観察し、セルフコレクションを促すもの。特にLLM消費に最適化されたシグナル（例：修正指示を含むカスタムリンターメッセージ）が効果的

どちらか片方だけでは不十分：
- フィードバックのみ → 同じ間違いを繰り返すエージェント
- フィードフォワードのみ → ルールを守るが効果を検証できないエージェント

### 計算的（Computational）vs 推論的（Inferential）

ガイドとセンサーにはそれぞれ2つの実行タイプがある：

| タイプ | 特徴 | 例 |
|--------|------|-----|
| **計算的（Computational）** | 決定論的・高速・CPU実行 | テスト、リンター、型チェッカー、構造分析 |
| **推論的（Inferential）** | 意味解析・非決定論的・GPU/NPU実行 | AIコードレビュー、LLM-as-judge |

| 方向 | 計算的/推論的 | 実装例 |
|------|-------------|--------|
| コーディング規約 | フィードフォワード / 推論的 | AGENTS.md, Skills |
| 新規プロジェクトのブートストラップ手順 | フィードフォワード / 両方 | 指示書+ブートストラップスクリプト |
| コード変換 | フィードフォワード / 計算的 | OpenRewriteレシピを利用するツール |
| 構造テスト | フィードバック / 計算的 | ArchUnitテストを実行するpre-commitフック |
| レビュー手順 | フィードバック / 推論的 | Skills |

### ステアリングループ（The Steering Loop）

人間の役割は、ハーネスを反復的に改善してエージェントを操舵すること。問題が複数回発生した場合、フィードフォワードとフィードバックの制御を改善し、将来の発生を低減または防止する。AIを活用してハーネス自体を改善することも可能で、構造テストの記述、パターンからのルール生成、カスタムリンターの足場作り、コードベースの考古学からのハウツーガイド作成などが挙げられる。

### タイミング：品質を左にシフト（Keep Quality Left）

継続的インテグレーション/デリバリーの原則に従い、フィードバックセンサーはライフサイクル全体に適切に配置する：

- **統合前・コミット前**: リンター、高速テストスイート、基本的なコードレビューエージェント
- **統合後（パイプライン内）**: ミューテーションテスト、より広範なコードレビュー
- **継続的ドリフト・健全性センサー**: デッドコード検出、テストカバレッジ品質分析、依存関係スキャナー
- **ランタイムフィードバック**: SLO劣化の検出、AIジャッジによるレスポンス品質のサンプリング、ログ異常のフラグ付け

### 規制カテゴリ

ハーネスが規制する対象は3つのカテゴリに分類される：

#### 1. 保守性ハーネス（Maintainability Harness）

内部コード品質と保守性を規制する。既存ツールが豊富で最も構築しやすい。

- **計算的センサーが確実に検出できるもの**: 重複コード、循環的複雑度、テストカバレッジ不足、アーキテクチャドリフト、スタイル違反
- **LLMが部分的に対処できるもの（高コスト・確率的）**: 意味的に重複するコード、冗長テスト、力技の修正、過剰設計
- **どちらも確実に検出できないもの**: 問題の誤診断、過剰エンジニアリングと不要な機能、指示の誤解

#### 2. アーキテクチャ適性ハーネス（Architecture Fitness Harness）

アプリケーションのアーキテクチャ特性を定義・検証する（Fitness Functions の概念）。

- パフォーマンス要件をフィードフォワードするSkillsと、改善/劣化をフィードバックするパフォーマンステスト
- オブザーバビリティ向上のためのコーディング規約（ログ標準など）を記述するSkillsと、ログ品質の振り返りを求めるデバッグ指示

#### 3. 振る舞いハーネス（Behaviour Harness）

アプリケーションが機能的に正しく動作するかをガイド・検知する、**最も困難な課題**。

現在の一般的なアプローチ：
- **フィードフォワード**: 機能仕様（簡単なプロンプトから複数ファイルの詳細な記述まで）
- **フィードバック**: AI生成テストのグリーン確認、カバレッジ確認、ミューテーションテスト＋手動テスト

AI生成テストへの過度な依存は問題であり、[Approved Fixtures パターン](https://lexler.github.io/augmented-coding-patterns/patterns/approved-fixtures/)が一部で成果を上げているが、万能ではない。

### ハーネサビリティ（Harnessability）

すべてのコードベースが同等にハーネス構築に適しているわけではない：

- **強い型付け言語** → 型チェックをセンサーとして活用可能
- **明確なモジュール境界** → アーキテクチャ制約ルールを適用可能
- **Springのようなフレームワーク** → 暗黙的にエージェントの成功確率を向上

**グリーンフィールド** vs **レガシー**の違い：
- グリーンフィールド: 初日からハーネサビリティを組み込める
- レガシー: 技術的負債が蓄積したコードベースほどハーネスが必要だが、構築が最も困難

### ハーネステンプレート

多くの企業では共通のサービストポロジー（API公開のビジネスサービス、イベント処理サービス、データダッシュボードなど）がニーズの80%をカバーしている。既存のサービステンプレートは将来的に**ハーネステンプレート**へ進化する可能性がある。ガイドとセンサーをバンドルし、コーディングエージェントを特定のトポロジーの構造・規約・技術スタックに束縛するものとなる。

ただし、サービステンプレートと同様の課題（インスタンス化後の上流改善との乖離、バージョニング問題）が発生し、非決定論的なガイドとセンサーはテストが困難なため、さらに悪化する可能性がある。

### 人間の役割

人間の開発者は、コードベースに対して暗黙のハーネスとして経験やスキルを持ち込む：

- 規約やベストプラクティスの内面化
- 複雑さに対する認知的な痛みの感覚
- コミットに自分の名前が残るという社会的責任感
- 組織的なアラインメント（チームの目標、許容される技術的負債の認識）

コーディングエージェントにはこれらがない：社会的責任感、300行の関数への嫌悪感、「ここではそうしない」という直感、組織的記憶のいずれも持たない。

> ハーネスは人間の開発者経験がもたらすものを外部化・明示化する試みだが、限界がある。良いハーネスは人間のインプットを完全に排除することを目指すのではなく、**人間のインプットが最も重要な場所に向けること**を目指すべきである。

## 重要な事実・データ

- **ハーネスの定義**: Agent = Model + Harness（LangChainブログより）。本記事ではコーディングエージェントの外部ハーネスに焦点
- **制御の2軸**: フィードフォワード/フィードバック × 計算的/推論的の4象限で分類
- **3つの規制カテゴリ**: 保守性、アーキテクチャ適性、振る舞い
- **OpenAIチームの事例**: カスタムリンターと構造テストによる層構造アーキテクチャの強制、定期的な「ガベージコレクション」によるドリフト検出
- **Stripeの事例（Minions）**: ヒューリスティックに基づく pre-push フックで関連リンターを実行、「shift feedback left」の重視、Blueprintsによるフィードバックセンサーのワークフロー統合
- **ミューテーションテストと構造テスト**: 従来は活用不足だったが、エージェント時代に復権しつつある

## 結論・示唆

### 著者の結論

ハーネスエンジニアリングは、一度きりの設定ではなく、**継続的なエンジニアリングプラクティス**として確立されつつある。機能レベル（SkillsやMCPサーバー）を超え、エージェントの生成物に真の信頼を与える制御システムをどう戦略的に設計するかが重要である。

### 未解決の課題

- ガイドとセンサーの一貫性維持（矛盾しないこと）
- 指示とフィードバック信号が異なる方向を指す場合のエージェントのトレードオフ判断
- センサーが発火しない場合の解釈（高品質なのか検出メカニズムの不備か）
- テストに対するコードカバレッジやミューテーションテストのような、ハーネスのカバレッジと品質の評価方法
- 配信ステップに分散するフィードフォワード/フィードバック制御を、システムとして構成・同期・推論するツール

### 実践的な示唆

- フィードフォワードとフィードバックの両方を組み合わせたハーネスを設計する
- 計算的な制御（リンター、テスト、型チェック）を基盤として活用し、推論的な制御で補完する
- 品質チェックをライフサイクルの可能な限り左にシフトする
- 問題の反復発生をトリガーにハーネスを反復的に改善する
- グリーンフィールドプロジェクトではハーネサビリティを初期の技術選定に組み込む
- 人間のレビューを排除するのではなく、最も価値ある場所に集中させる

## 制限事項・注意点

- 振る舞いハーネス（機能的な正確性の検証）は未解決の大きな課題であり、AI生成テストへの依存だけでは不十分
- 推論的制御は非決定論的であり、コミットごとに実行するにはコストが高い
- ハーネステンプレートはバージョニングや同期の課題を抱え、非決定論的要素がテストをさらに困難にする
- 本記事は著者が「現時点で有用」と考えるメンタルモデルであり、確立されたフレームワークではない
- GenAI（Claude/Claude Code）がリサーチ、関連アイデアの統合、言語の洗練に使用されている

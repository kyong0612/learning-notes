## 概要

Ubieが開発・運用するセキュリティアラート分析AIエージェント [Warren](https://github.com/secmon-lab/warren) の実装事例を基に、**ハーネスエンジニアリング** の具体像を解説した記事。ハーネスエンジニアリングは Mitchell Hashimoto 氏が2026年2月に提唱した概念で、「エージェントが一度したミスを次から繰り返さないように仕組み化すること」を指す。本記事ではこれを **モデル以外の部分でAIエージェントをいい感じにするためのエンジニアリング全般** （プロンプト、コンテキスト、評価、観測性、アーキテクチャ制約、改善ループ等を含む包括的概念）と位置づけている。

Warren は各種監視装置から受け取ったセキュリティアラートの分析を担うエージェントであり、1年以上の運用から得られた知見が詰め込まれている。

## Warren 設計の3つの根底思想

1. **LLMの不確実性をアーキテクチャで制御する**
   - 確証バイアス、事実と推測の混同、冗長な出力などLLMの系統的な弱点は「プロンプトで気をつけて」では制御しきれない。
   - フェーズ分割やツール制限など **構造的に弱点が発現しにくい設計** を採用。プロンプトは「望ましい振る舞いの指示」ではなく「アーキテクチャの隙間を埋める最後の手段」として扱う。
2. **入力は寛容に、出力は厳格に**
   - 分析のためWeb上データなど信頼できない入力も取り込む必要がある。
   - 一方で破壊的処理やデータ漏洩に繋がる出力・実行は厳格に遮断。
3. **実行コストを最適化する**
   - セキュリティ調査は無限に深掘りできてしまうため、コスト・時間の上限を明示的に設計に埋め込む。コーディングエージェントとの大きな違い。

## 1. 計測観点

### 可観測性の向上

- 実行全体を **スパン（ルートスパン＝実行全体、子スパン＝計画フェーズ・各タスク・リプラン）** として記録し GCS に永続化。
- 各スパンにはツール入出力、LLM応答、エラー情報を含め、後から「なぜこの判断をしたか」を追跡可能。
- トレース記録をそのまま開発ツール（LLM）に食わせることで、問題分析や改善提案まで自動化。

### 評価容易性の向上

- 計画フェーズの出力を **JSON スキーマで強制** し、タスク一覧・メッセージ・質問を構造化データとして取得。
- 各タスクには **Acceptance Criteria（完了条件）を必須化** し、リプラン時に達成度を評価可能にする。

実例（計画フェーズのタスク一覧）：

```json
{
  "message": "GKEポッド上の不審なプロセスに関する調査を開始します。...",
  "tasks": [
    {
      "id": "indicator-reputation-check",
      "title": "インジケーターの評判分析",
      "description": "通信先IP、ドメイン、ハッシュ値を VirusTotal / OTX / AbuseIPDB / WHOIS で評価...",
      "acceptance_criteria": "各インジケーターの悪性度および関連する脅威名が特定されていること",
      "tools": ["vt", "otx", "ipdb", "whois"]
    },
    {
      "id": "url-domain-analysis",
      "title": "通信先ドメイン・URLの分析",
      "description": "urlscan.io / webfetch で C2 やフィッシング特徴がないか分析...",
      "acceptance_criteria": "通信先 URL の目的（C2、ダウンロード等）が推定されていること",
      "tools": ["urlscan", "webfetch"]
    }
  ]
}
```

## 2. 実行設計観点

### コンテキスト効率化（3層制御）

LLMエージェントの性能はコンテキストの質に大きく依存する。Warren では 3 層でコンテキストを制御。

| 層 | 戦略 |
|----|------|
| **会話履歴** | トークン上限超過時に **古い70%を自動でLLMにより要約圧縮**。直前の会話は残すことで「圧縮直後の記憶喪失」による迷走を防止。 |
| **データ取得** | BigQuery / CrowdStrike Falcon / Slack からのデータ取得は **子エージェントとして独立**させ、「結果を解釈せず生データのみ返す」と強制。加えて「最小限クエリで目的達成」「1000行上限」「LIMIT必須」などを指示。 |
| **タスク出力** | タスクエージェントに「生データのみ返せ、解釈・推測・推奨は一切禁止」と制限し、最終合成フェーズへのインプット質を高める。 |

> **運用知見**: データ探索の行上限を当初 100 行にしたところ、LLM が自発的に `OFFSET` ページネーションを学習し、122行の結果を得るために 11 回クエリを実行する事例が発生。上限を 1000 行に引き上げ、代わりに集約を促すことで解決。

### 制御性・決定性の向上

Warren は3種類の制御を組み合わせる。

#### (a) エージェント実行フローによる制御：5フェーズ構造

処理全体を **Intent Resolution → Plan → Execute → Replan → Final** に分割。

- **Intent Resolution**: ユーザー発言とアラートから [XY Problem](https://en.wikipedia.org/wiki/XY_problem) を検出して調査方針を決定。YAML frontmatter 付き Markdown で定義された **ユーザー定義の調査戦略（例: Exfiltration Detection、Credential Abuse）** を自動選択。
- **Plan**: 解決された Intent をもとにタスク分割・ツール割り当て・完了条件を構造化JSONで出力。**タスクごとにツールセットを分離** し、実行時には必要なツールのみ渡す（BigQuery タスクには BigQuery ツールのみ、等）。
- **Execute**: 各タスクを独立した子エージェントとして並列実行。
- **Replan**: 結果を見て追加調査の要否を判断。情報不足ならユーザーに質問することも可能。
- **Final**: 全タスク結果を統合してセキュリティ評価を生成。

#### (b) プロンプトによる制御：思考フレームの矯正

運用観察から段階的に追加されたセクション群。特に **確証バイアス対策** は大規模で、「深刻度を過大評価して恐怖を煽る／過小評価して安心させすぎる」両極端を抑制するため、「恐怖の増幅でも偽りの安心でもなく、最も蓋然性の高いシナリオで評価せよ」という基本原則を導入し、約210行の大幅改訂を実施。

主要ルール（プロンプト抜粋）:

- **Facts と Hypotheses を明示的に区別**: 仮説を事実として断定しない。不確実性を反映する言葉（"this suggests" 等）を使う。
- **複数解釈の検討**: 任意のセキュリティイベントに対し、少なくとも2つの妥当な解釈を生成。
- **反証探索**: 主仮説に矛盾する事実を積極的に探す（disconfirming evidence）。
- **アンカリング回避**: 最初の印象は複数仮説のうちの一つにすぎない。
- **矛盾時の即時更新**: 新情報が過去の仮説を覆した場合は即座に破棄／更新。

Anti-Patterns セクションでは「確証バイアスの抑止」「調査報告書スタイルの禁止」などを具体例付きで列挙している。

#### (c) 決定的ルールによる制御：Rego ポリシー

アラート受信時の初動パイプライン（取り込み・エンリッチ・トリアージ）は **OPA / Rego ポリシー** で決定論的に制御。LLMに委ねるべきでない部分をコードとして切り出し、判断のブレをなくすとともに自明なアラートを LLM に処理させずコストも抑制。

```rego
package ingest.scc

alerts contains {} if {
  not ignore
}

# Log4j の脆弱性は対処済みなので問題なし
ignore if {
  input.finding.category == "Initial Access: Log4j Compromise Attempt"
}
```

### 権限・実行安全性の確保

#### クレデンシャルの隔離

- 全外部サービスへのアクセスを **Factory パターン** で構成し、Factory がクレデンシャルを受け取ってクライアントを生成。
- LLM に公開される `ToolSet` インターフェースには **パラメータ仕様と実行エントリポイントのみ**。APIキー、サービスアカウント、OAuthトークンは Go 構造体のフィールドに閉じ込められ、LLMのコンテキストに一切現れない。
- → プロンプトインジェクションでLLM挙動が操作されても、**クレデンシャル窃取・漏洩はアーキテクチャ上不可能**。

#### 自動的なガードレール（BigQueryの例）

- **ドライラン必須**: スキャンサイズ上限超過クエリをブロック（スキャン破産防止）。
- **最大1000行で切り捨て**: 超過時は「`OFFSET`でページネーションするな、`GROUP BY`で集約せよ」と指示。
- **デフォルト5分のクエリタイムアウト**でジョブキャンセル。
- **プロンプト制約**: `SELECT *` 禁止、`OFFSET` 禁止等のアンチパターンを明記。
- **最小権限**: 接続SAを `dataViewer` に絞り、`DELETE`/`DROP` 系のクエリを権限レベルで拒否。

#### 攻撃耐性：データ漏洩とプロンプトインジェクション

- 外部通信可能なツールは **WebFetch のみ** に限定し、HITL（Human-in-the-Loop）による人間の承認を必須化。他のツール（Slack検索、BigQueryなど）は読み取り専用。
- → WebFetch という **単一のチョークポイント** でデータ漏洩を構造的に抑制。
- プロンプトインジェクション自体は完全には防げないが、外部干渉手段がないため主リスクは「分析結果の歪曲」に限定され、全トレースの記録・表示により人間が検証可能。

#### 人間による監視と運用制御

- HITL: 特定ツールに承認必須フラグ。Slack の対話ボタン（Approve / Deny）で承認フロー。Presenter 未設定時はツール実行自体をブロック。
- リプランフェーズで不足情報があれば **選択肢と理由付きでオペレーターに質問**。
- Slack コマンドでセッション中断が可能（`running` → `aborted` を Firestore に記録）。ツール呼び出し境界でチェックし、次のチェックポイントで安全に停止。
- Rego ポリシーによる **Slackユーザーベースの認可**（機微データ保護）。
- 緊急時は関連 Slack メッセージ一括削除コマンドも搭載。

### 永続性・耐障害性の確保

- **セッション状態・クエリ・Intent・ユーザー情報**: Firestore。
- **LLM 会話履歴**: 本文は Cloud Storage、メタデータは Firestore の二層保存。LLM呼び出しごとに保存するため中断からも復旧可能。
- 同一チケットへの追加問い合わせ時に過去会話をリロードし、文脈を引き継いで調査継続が可能。

## 3. 実行最適化観点

### エージェント性能の最適化

#### 記憶の管理：Knowledge Reflection

- ナレッジ検索は **BM25 + コサイン類似度 + RRF（Reciprocal Rank Fusion）** によるハイブリッド検索。
- 計画フェーズ・タスク実行フェーズの両方で検索し、ナレッジだけで回答可能ならタスクを生成せず直接回答。
- タスク実行時には **ナレッジ検索ツールが全タスクに自動注入** される。
- **Knowledge Reflection** により、タスク完了後・セッション終了後にバックグラウンドでナレッジを自動抽出：

| カテゴリ | 粒度 | 内容例 |
|---------|------|--------|
| **Technical Knowledge** | タスク単位 | 効果的だったクエリパターン、失敗したアプローチとその理由、データソースの所在 |
| **Fact Knowledge** | セッション単位 | 特定ホストの役割、既知の誤検知パターン、インフラ構成情報 |

> **重要な方針**: 「**失敗の知識は成功の知識より価値がある**」。言語モデルの内部知識ではなく、実際にツール実行で観測された結果のみを記録。既存エントリの検索・更新・削除も行い、矛盾する古い知識を自動整理する。

**進化の歴史**: 現在に至るまでに2つの前身システム（漠然と記憶させるだけの機能、専用エージェント単位の記憶）を全廃止。いずれも **記憶の整理がなかった** ことが問題。現在は記憶の検索・編集ツールを渡したエージェントに「追記・編纂」作業を行わせ、膨張を防いでいる。

#### ドメイン固有の知識：クエリ最適化と Runbook

- BigQuery では **パーティションカラム優先、`SELECT *` 禁止、`OFFSET` 禁止、ゼロ件時の段階的検証手順**（データ存在確認→時間範囲拡大→実値確認→生データサンプル）をプロンプト指示。
- **Runbook**: 組織固有の調査クエリを SQL ファイルとして定義。メタデータ（ID・タイトル・説明）のみシステムプロンプトに動的注入し、SQL 本文は `get_runbook` ツールで必要時に取得してコンテキスト節約。エージェントはそのまま使わず **調査対象に合わせて適応・修正** する。

```sql
-- title: 特定のユーザのGoogle Workspace Drive 利用ログ
-- description: ダウンロードについて調べる場合は event_name = "download" を条件につける

SELECT
  time_usec_timestamp, event_name, event_type, email,
  drive.originating_app_id, drive.doc_id, drive.doc_title,
  drive.doc_type, drive.visibility, drive.owner,
  has_sensitive_content, actor_application_info
FROM actual-my-project.if_google_workspace_audit.activity
WHERE partition_time BETWEEN TIMESTAMP("2025-06-24") AND TIMESTAMP("2025-06-26")
  AND email = "xxx@xxx.com"
LIMIT 1000
```

### レイテンシ・スループットの改善

3つの方針：

1. **クリティカルパス上の処理を並列化**: 同一フェーズ内のタスクを完全並列実行。タスクメッセージを事前一括作成し、完了時に即座に結果投稿。
2. **クリティカルパスから外す**: Knowledge Reflection は別スレッドで非同期実行し、ユーザー応答を遅らせない。
3. **エージェント到達前に不要処理を省く**: アラート受信時に **Ingest → Enrich → Triage の3段階パイプライン** で自動処理。Rego ポリシーで処理要否を制御し、不要アラートはエージェント呼び出し前にフィルタリング。

### コスト効率化：バジェット戦略

- 各タスクに **バジェット（デフォルト100ポイント）** を設定。
- **ツール種別ごとのコスト + 時間コスト（経過秒数ベース）** で消費。
- 二段構えの打ち切り：
  - **ソフトリミット**（バジェット枯渇）: 警告。
  - **ハードリミット**（ソフトリミット後3回超過）: 強制終了し、**ハンドオーバー情報（ツール履歴・経過時間・消費割合）** を生成してリプランフェーズで小さいタスクへ分割。
- BigQuery・Falcon・Slack の子エージェント呼び出しは **親タスクのバジェットから除外**し、内部で個別追跡してコスト分離。

> **背景**: もとは「ループ上限」のみで制御していたが、「深堀り途中で止まって結果なし」「時間のかかる処理を連発して返答が遅い」という問題から、**時間の概念を反映した** 現在の設計に進化した。

## まとめ

### 全体アーキテクチャの鳥瞰

| 観点 | 主要な仕組み |
|------|------------|
| 計測 | スパン階層のGCS永続化、JSONスキーマ強制、Acceptance Criteria |
| 実行設計 - コンテキスト | 3層（会話/データ/タスク）での圧縮・分離・生データ強制 |
| 実行設計 - 制御 | 5フェーズ分割 / ツールセット分離 / 思考フレームプロンプト / Rego ポリシー |
| 実行設計 - 安全性 | Factoryパターンによるクレデンシャル隔離 / ガードレール / WebFetchチョークポイント / HITL |
| 実行設計 - 永続性 | Firestore + Cloud Storage 二層保存 |
| 実行最適化 - 性能 | Knowledge Reflection（Technical/Fact）/ Runbook |
| 実行最適化 - レイテンシ | 並列実行 / 非同期Reflection / 前段パイプラインフィルタ |
| 実行最適化 - コスト | バジェット戦略（ソフト/ハードリミット + ハンドオーバー） |

### 著者の主張

- 全ての工夫に共通するのは **「LLMの能力に期待しすぎず、構造やルールで弱点を補う」** という方針。
- プロンプトで指示するだけでは不十分な部分を **アーキテクチャで埋めていく**。
- これはセキュリティ分析に限らず **AIエージェントを実務投入するあらゆる場面で重要**。
- 紹介された仕組みは **1年以上の運用を通じた試行錯誤の結果** であり、最初から完成していたわけではない。**「動かす → 失敗を観察 → 再発しないよう仕組み化」というサイクルを回し続けることがハーネスエンジニアリングの本質**。

### 限界事項・留保

- 記事中でも明記されているとおり、「ハーネスエンジニアリング」という言葉自体の定義は業界内で不統一であり、バズワード的な側面もある。数ヶ月後には別の単語に置き換わる可能性もある。
- Warren のアプローチはセキュリティ分析という **コストが際限なく膨らみうるドメイン** に最適化されており、コーディングエージェント等とは設計上のトレードオフが異なる。

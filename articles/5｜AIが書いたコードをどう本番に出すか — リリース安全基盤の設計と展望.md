## はじめに

株式会社ACESのテックリード福澤氏（[@fuku_tech](https://x.com/fuku_tech)）によるAI駆動開発シリーズの第5回。AI駆動開発における人間とAIの役割分担を自動運転の4つのPhaseで整理するシリーズの中で、本記事は [#3](https://tech.acesinc.co.jp/entry/2026/03/04/083000)・[#4](https://tech.acesinc.co.jp/entry/2026/03/11/083000) で整備した**品質ゲート**の上に、**リリース安全基盤**を積み上げる話。

前回（#4）までの成果と課題:
- 品質ゲート（Lint・Tests・セルフレビュー・別LLMレビュー）による予防が機能し、**AI実装起因のインシデントは0件**
- しかしAIがPRを量産しても**人間側のレビューが追いつかない**問題が残った
- タスク分解やPR分割は人間の介入を増やす方向であり、スケールする解ではない

## 発想の転換: 「壊れないようにする」→「壊れても速く直せる」

品質ゲートはコードレベルの予防策として機能しているが、以下のようなリスクはコードレビューだけでは見えない:

- **本番トラフィックでのみ顕在化するパフォーマンス劣化**
- **長時間稼働で初めて現れるメモリリーク**
- AI生成コードの量が増えるほど、リスクの総量も増加

> **発想の転換**: 安全の判断基準を「人間の目」から「データとシステム」に移す。壊れたときに**速く検知し速く戻せる**仕組みを作ることで、「なんとなく怖い」を**数値に基づく判断**に置き換える。

## リリース安全基盤の現在地（4層構造）

### 01 Progressive Delivery: Feature Flagで段階ロールアウト

| 項目 | 内容 |
|---|---|
| ツール | DevCycle（Feature Flagサービス） |
| 導入時期 | AI活用本格化以前から導入済み |
| ロールアウトステップ | 自社テナントのみ → 一部顧客 → 全体 |
| 判断基準 | 一定期間、関連する不具合が0件であること |

**直近の実績**: コア機能（音声解析周り）の改修で自社テナントのみに先行リリース → エラー検知 → 修正後に改めてロールアウト → 安全にリリース完了。AI生成コードではなかったが、**既存の運用をAI生成コードにもそのまま適用できる確信**を得た。現在、AI生成コードへのFeature Flag組み込みを検討中。

### 02 Observability: アラートの信頼性確保と異常検知

- **Sentryのアラートノイズ撲滅**から着手
  - ClaudeのSlack連携でノイズアラート（対応不要だが繰り返し発火するもの）をコツコツ除去
  - 地味だが安全基盤の**土台**となる作業
- **新たな検知の追加**: SentryのTrace機能を使ったN+1クエリの検知など
- AI駆動開発固有ではなく、プロダクト全体の検知能力の底上げだが、AI安全基盤としても効く

### 03 Error Triage Automation: 検知から修正PRまでの自動化

検知から修正提案までを一気通貫で回すワークフローを構築中:

1. **GitHub Actions**で定期的にSentryを監視
2. 新規エラー検知 → **Issue自動作成**
3. **Claude API**でエラーの原因調査と修正方針の起案
4. **改善PRの生成**まで自動化
5. 最終的に**人間がPRをレビューして判断**

> まだ検証段階だが、パイプラインが回り始めれば**エラー検知から修正PR作成までの時間を大幅に短縮**できる見込み。

### 04 SLO / Error Budget: データ駆動のリリース判断

- 現在の判断は「不具合0件かどうか」のオール・オア・ナッシング → リリース速度と安定性のバランスを柔軟にコントロールできない
- **SLO（サービスレベル目標）とエラーバジェット**の導入を目指す
  - 例: 可用性99.9%を設定 → 月あたり約43分のダウンタイムが許容される（= エラーバジェット）
  - 余白が十分 → リリース速度を上げる / 減ってきた → 安定化に集中
- **リリース判断を感覚ではなく数字で行う**仕組みをAI駆動開発の文脈で設計していく段階

## これで何が変わるのか

| 前回の課題 | リリース安全基盤による解決 |
|---|---|
| **巨大PR問題** | Feature Flagによる段階ロールアウト＋高速ロールバックで、PRが大きくなっても致命傷にならない。PRの粒度だけに安全を依存しない構造を構築 |
| **レビューが追いつかない問題** | レビューの焦点が「コードの一行一行」→「段階ロールアウトの各ステップの判断」に移行。人間の介入ポイントが減り、AIへの委譲範囲が広がる |
| **「出して大丈夫か？」の不安** | 品質ゲート通過 ＋ 段階ロールアウトで実際の挙動確認 → Go判断がしやすくなる。SLO/エラーバジェットで数字による裏付けも可能に |

## その先に見える世界

リリース安全基盤が整った先の更なる自動化ビジョン:

1. **Feature Flag操作のAI委譲**: AIがメトリクスに基づいて段階ロールアウトの各ステップを自律的に判断し、異常検知時は自動ロールバック。まずは定型的な判断から段階的に自動化し、不確実性の高い判断は人間が行う線引き。
2. **AIが生ログに瞬時にアクセスできる環境の整備**: Sentry・CloudWatch・Grafanaなど複数の監視基盤のログをAIがリアルタイム確認 → sandbox環境でのリリース前検証でも本番環境でのリリース後監視でも、AIが自律的にフィードバックサイクルを回せる。

> この2つが揃うと、**AIが自分でコードを書き、自分で検証し、自分でリリースし、問題があれば自分で直す**というループが成立。人間に残るのは**最終承認**と**ループ自体の設計・改善**。

## おわりに

シリーズ全体の流れ:
- **Phase 1→2**: 何をAIに任せるかを整理
- **Phase 2→3**: どうすれば安全に任せ切れるかを設計
- **Phase 3→4**: 壊れたときどう直すかに向き合う

各Phaseで手放してきた前提:
- 「全部自分で書く」
- 「自分がやった方が早い」
- 「壊れないようにする」
- そして今 → **「人間がコードを見なければ安全は担保できない」という前提を手放そうとしている**

> Feature Flagやアラート整備は、AI時代に突然必要になったものではなく、**ずっと大事だったものがAI時代にさらに重要になっただけ**。まっとうなエンジニアリングプラクティスこそが最大の武器。

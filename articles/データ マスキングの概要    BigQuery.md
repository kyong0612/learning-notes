## 概要

BigQueryは列レベルでのデータマスキングをサポートしている。データマスキングを使用すると、ユーザーグループごとに列データを選択的に難読化しながら、列へのアクセス自体は許可できる。この機能は**列レベルのアクセス制御**の上に構築されており、完全アクセス権からマスクされたアクセス権、アクセス権なしまで、きめ細かいデータアクセスの範囲を構成できる。

## 主要なトピック

### データマスキングの利点

- **属性ベースのアクセス制御**: ポリシータグに基づくコンテキストに応じたデータアクセスを提供
- **大規模なポリシー適用**: データポリシーを作成しポリシータグと関連付け、任意の数の列に適用可能
- **既存クエリの変更不要**: 列レベルのアクセス制御とは異なり、ユーザーのロールに基づいて既存のクエリが自動的にマスキングを適用
- **データ共有の効率化**: 機密列をマスクすることで、より大きなグループでテーブルを共有可能

### データマスキングのワークフロー

データをマスクする方法は2つある：

1. **列にデータポリシーを直接設定する**（プレビュー）
   - データポリシーを作成 → 列にデータポリシーを割り当て
   - ポリシータグや追加の分類を作成せずにマスキングルールをデータにマッピング可能

2. **ポリシータグを使用してデータをマスクする**
   - 分類と1つ以上のポリシータグを設定
   - ポリシータグのデータポリシーを構成（マスキングルール + プリンシパルのマッピング）
   - ポリシータグをBigQueryテーブルの列に割り当て
   - マスクされたデータにアクセスするユーザーに「BigQueryのマスクされた読み取りロール」を割り当て

> **ベストプラクティス**: データポリシーレベルでマスクされた読み取りロールを割り当てること。プロジェクトレベル以上で割り当てると過剰な権限が付与される可能性がある。

### データのマスキングルール

マスキングはクエリに関連する他のすべてのオペレーションよりも優先される。使用可能なルールは以下の通り：

| ルール | 対象データ型 | 動作 |
|--------|-------------|------|
| **カスタムマスキングルーティン** | `STRUCT`を除くすべて | UDFを列に適用。出力は定義された関数による |
| **年月日マスク** | `DATE`, `DATETIME`, `TIMESTAMP` | 年に切り捨て（例: `2030-07-17` → `2030-01-01`） |
| **デフォルトのマスキング値** | すべて | データ型に基づくデフォルト値を返す（`STRING`→`""`, `INTEGER`→`0`など） |
| **メールマスク** | `STRING` | ユーザー名を`XXXXX`に置換（例: `abc123@gmail.com` → `XXXXX@gmail.com`） |
| **先頭の4文字** | `STRING` | 先頭4文字を残し残りを`XXXXX`に置換 |
| **末尾の4文字** | `STRING` | 末尾4文字を残し残りを`XXXXX`に置換 |
| **ハッシュ（SHA-256）** | `STRING`, `BYTES` | SHA-256ハッシュ値を返す。JOINでの使用に適する |
| **null化** | すべて | `NULL`を返す。値とデータ型の両方を非表示 |

> **注意**: メールマスク・先頭/末尾の4文字ルールでは、値が4文字以下または無効な形式の場合、SHA-256ハッシュ値が返される。

#### データのマスキングルールの優先順位（階層）

ポリシータグには最大9つのデータポリシーを構成可能。ユーザーが複数グループに所属する場合、以下の優先順位（上が低優先、下が高優先）で適用ルールが決定される：

1. null化（最低優先）
2. デフォルトのマスキング値
3. 年月日マスク
4. 先頭の4文字
5. 末尾の4文字
6. メールマスク
7. ハッシュ（SHA-256）
8. カスタムマスキングルーティン（最高優先）

### ロールと権限

| 目的 | 必要なロール |
|------|-------------|
| 分類・ポリシータグの管理 | Data Catalog ポリシータグ管理者（`datacatalog.categoryAdmin`） |
| データポリシーの作成・管理 | BigQuery Data Policy 管理者 / BigQuery 管理者 / BigQuery データオーナー |
| ポリシータグの列への適用 | `datacatalog.taxonomies.get` + `bigquery.tables.setCategory` |
| マスクされたデータのクエリ | BigQueryのマスクされた読み取り（`bigquerydatapolicy.maskedReader`） |

#### マスクされた読み取りロールときめ細かい読み取りロールの相互作用

| ロール | 結果 |
|--------|------|
| ロールなし | アクセス拒否 |
| マスクされた読み取りロールのみ | マスクされた列データを表示 |
| きめ細かい読み取りロールのみ | マスクされていない元の列データを表示 |
| 両方のロール | ポリシータグ階層内の付与位置により決定 |

### ポリシータグ階層での承認の継承

- ロールは列に関連付けられたポリシータグから評価され、**昇順**でチェックされる
- サービスは最初に出現した割り当て済みのロールを使用する
- **重要**: ポリシータグ階層の下位でマスクされた読み取りロールが付与されている場合、上位できめ細かい読み取りロールがあっても、マスクされたデータが返される

### 費用に関する考慮事項

- null化またはデフォルトのマスキング値ルールを使用してマスクされた列をクエリした場合、その列はスキャンされないため、処理バイト数が減少し課金も減る

| 機能 | 互換性 | 備考 |

| BigLakeテーブル | ✅ 互換 | |
| BigQuery Storage Read API | ✅ 互換 | |
| BigQuery BI Engine | ✅ 互換 | マスキング有効クエリはBI Engineで高速化されない |
| BigQuery Omni | ✅ 互換 | |

| テーブル名の変更 | ✅ 互換 | |
| タイムトラベル | ✅ 互換 | 現在のスキーマのポリシータグが適用 |

| コピージョブ | ❌ 非互換 | 全列への完全アクセス権が必要 |
| スナップショット | ❌ 非互換 | 全列への完全アクセス権が必要 |
| ワイルドカードテーブルクエリ | ❌ 非互換 | 全列への完全アクセス権が必要 |

## 出力でのデータマスキングの例

以下は同一テーブル（Accounts: SSN, 優先度, ライフタイムバリュー, 作成日, メール）に対する各グループのクエリ結果：

| グループ | SSN | 優先度 | ライフタイムバリュー | メール |
|----------|-----|--------|----------------------|--------|
| data-users（マスクされた読み取り） | NULL | "" | 0 | NULL |
| accounting（SSNにきめ細かい読み取り） | 元の値 | "" | 0 | NULL |
| sales-exec（Confidentialにきめ細かい読み取り） | NULL | 元の値 | 元の値 | NULL |
| fin-dev（Financialにマスクされた読み取り） | NULL | "" | SHA-256ハッシュ値 | NULL |
| その他 | アクセス拒否 | — | — | — |

## 制限事項・注意点

- **同一プロジェクト要件**: データポリシー、ポリシータグ、ルーティンはすべて同じプロジェクト内に存在する必要がある
- **データポリシー上限**: ポリシータグあたり最大9個（うち1つは列レベルのアクセス制御用に予約）
- **ポリシータグ階層深さ**: ルートから最下位まで最大5レベル
- **組織要件**: ポリシータグの分類を含むプロジェクトは組織に属する必要がある
- **アクセス制御の自動適用**: データポリシーを持つ分類にはアクセス制御が自動適用され、オフにするにはすべてのデータポリシーの削除が必要
- **マテリアライズドビュー**: 既存のマテリアライズドビューがある場合、関連ベーステーブルの繰り返しレコードマスキングクエリは失敗する
- **パーティション分割テーブル**: パーティション分割列またはクラスタ化列に対するデータマスキングクエリは非サポート
- **レガシーSQL非サポート**: GoogleSQLのみ対応
- **カスタムマスキングルーティン**: 削除しても使用するデータポリシーは残存し、空のマスキングルールが残る（7日後に自動消去される場合がある）
- **エディション制限**: 特定のBigQueryエディションでは使用不可

## 概要

カウシェ（toC向けECプラットフォーム）では、PRの83%が人間のコードレビューを経ずにAIレビュー（Claude Code Action）のみで自動マージされている。本記事では、レビュールールを毎晩自動改善する仕組みを構築することで、AIレビューの精度をauto-mergeが許容できるレベルまで引き上げた取り組みが解説されている。

## 主要なトピック

### 自動マージが成り立つ前提条件

83%の自動マージはAIレビューの精度だけでなく、以下の前提の上に成り立っている。

#### コード品質の担保

- **E2Eテスト**: バックエンドではgRPCエンドポイントごとに正常系・異常系をステータスコード別に検証。テストと実装は原則セット
- **Design Doc**: 機能開発前にL1（メモレベル）〜L4（アーキテクチャレベル）の粒度別テンプレートで設計方針を明文化。AIの実装方針もこれに基づく
- **開発方針の明文化**: コーディング規約・テストポリシー・API設計ガイドラインをLiving Docとして維持。レイヤー構造（handler/usecase/domain）、命名規約、エラーハンドリング方針をAIレビューが準拠性チェック

#### 速度と品質を並列で回すカルチャー

- 「できるだけ早く本番に反映させてユーザーに価値を届ける」思想
- 即座に対応が不要な改善点のためにマージを止めない。改善は別PRで追いかける
- AIレビューのSUGGESTION（改善提案）が放置されないよう仕組み化：未対応の指摘は次回PRレビュー時に再指摘、同カテゴリの繰り返し指摘は過去PRを引用して重点チェック

### AIレビューの構成

#### 3人の専門家ペルソナによる並列レビュー

AIレビューは単一のAIではなく、3つの専門家ペルソナに並列でレビューを依頼し、結果を統合する構造。

| ペルソナ | レビュー観点 |
|----------|------------|
| **シニアGoエンジニア** | コード品質、規約準拠、テストカバレッジ |
| **シニアアーキテクト** | 設計の整合性、セキュリティ（認証バイパス・情報漏洩・インジェクション等）、ドメイン知識との照合 |
| **GCP専門家** | DBクエリの効率、インフラへの影響、非同期処理の冪等性 |

#### 各専門家が参照するコンテキスト

- **レビュー設定ファイル**: 危険度マップ、ドメイン知識
- **コーディングガイドライン**: サービスごとのアーキテクチャ説明、ディレクトリ構造、命名規約、テスト方針
- **サービス横断パターン集**: DBトランザクションの制約、API規約（変更内容に応じて該当セクションのみ選択読み込み）
- **検出ルール**: 自動改善ループが過去の見逃しから自動生成したルール
- **開発者プロファイル**: 過去実績に基づきSUGGESTION量を調整（Approve/Reject判定基準は変えない）
- **PR本文・コメント・実装計画**: 作成者の補足コメント、過去のAIレビューコメント、段階的実装の中間状態の許容

#### 指摘のトリアージ

各専門家の指摘はBUGとSUGGESTIONに分類後、以下のフィルタリングを実施：
- diffの変更に根拠がない指摘の除外
- 前回レビューと矛盾する指摘の検出
- 曖昧な指摘の除外

### auto-mergeを実現するためにやったこと

#### 判定原則とAI/人間の役割分担

**判定原則**: 「却下すべき理由がないならApprove」

- **マージをブロックするもの（BUG）**: ランタイムエラー、データ不整合、セキュリティ脆弱性、トランザクション境界の誤り等
- **ブロックしないもの（SUGGESTION）**: 改善提案としてコメントに残すが、マージは止めない

**人間レビュー必須の領域**（AIの判定にかかわらずエスカレーション）：
- DBスキーマ変更、Proto定義の変更（ロールバック困難、クライアント互換性に影響）
- Terraform変更（plan結果を人間が確認してからapply）
- 認証・決済フロー（auto-mergeせず人間が最終判断）
- 実装計画、Design Doc（方向性を決めるもの）
- 大規模なアーキテクチャ変更、複数サービスにまたがる変更

#### レビュールールの自動改善ループ（毎晩実行）

ルールの鮮度を保つため、5つのエージェント（Claude Code Action）が毎晩ループで改善を実行：

| エージェント | 役割 |
|------------|------|
| **Measure** | 直近のPR統計を集計（承認率、誤判定数、ルールごとの発火率、開発者ごとの実績） |
| **Explore** | リポジトリやインフラを巡回し、数字だけでは見えないボトルネックや未知のパターンを探索 |
| **Improve** | MeasureとExploreの出力を読み、検出ルールやパターン集を実際に編集 |
| **Reflect** | 3つのエージェントの出力を横断的に読み、翌日の優先順位を決定 |
| **Audit** | 全エージェントの変更を検証し、改善が逆効果になっていないかチェック |

**連続性の確保**: 各エージェントは毎回新しいセッションで動くため、**思考ジャーナル**（前回の判断記録）を書かせて次回実行時に読み込ませる。映画『メメント』のように前回の自分が残したメモを頼りに判断する構造。

**ルールの自動生成と整理**:
- 外部レビューツールがAIの見逃しを検出 → 翌日Measureが記録 → Improveが類似ケースを横断分析して検出ルールを自動生成（数十件が生成済み、増加中）
- Measureがルールごとの発火数と効果数を追跡し、効果スコア（効果数÷発火数）が低いルールはImproveがアーカイブ
- ルール数に上限を設定し、超過時は効果スコア順に整理

**安全対策**:
- エージェントごとに書き込み可能なファイルを制限（権限の分離）
- ループ最後にAuditが全変更を検証

### 再現のためのステップ

1. **Step 1: 前提を整える** — E2Eテスト・コーディング規約・CIパイプラインが先に必要。最低限コーディング規約と危険度マップを用意
2. **Step 2: レビューコメントから始める** — いきなりauto-mergeではなく、数週間はAIコメント投稿のみで精度を確認。判定原則は最初から「却下すべき理由がないならApprove」に設定
3. **Step 3: 計測と改善の循環** — 承認率・誤判定数・見逃し数を定期集計し、指標が安定したらauto-merge切り替え

## 重要な事実・データ

- **83%**: 人間レビューなしでAIレビューのみで自動マージされるPRの割合
- **17%**: 人間がレビューする割合（DBスキーマ・設計判断・アーキテクチャ変更など）
- **4ヶ月**: 運用期間
- **3ペルソナ**: AIレビューの並列構成（シニアGoエンジニア、シニアアーキテクト、GCP専門家）
- **5エージェント**: 毎晩の自動改善ループ（Measure, Explore, Improve, Reflect, Audit）
- **数十件**: 自動生成された検出ルール数（増加中）
- **Claude Code Action**: AIレビューの基盤技術

## 結論・示唆

### 著者の結論

- AIレビューの質は「何を見つけるか」ではなく「**何を見ないか**」で決まる。指摘を絞ったら本当に対応すべきものだけが残った
- 「何を見ないか」の判断を毎晩自動で更新し続けるのがこの仕組みの核心
- 定型的なPRレビューから解放された分、設計方針の共有やアーキテクチャ判断に時間を使えるようになった

### 実践的な示唆

- AIレビューの導入はコード品質の基盤（テスト・規約・CI）が整っていることが大前提
- 判定原則は「却下すべき理由がないならApprove」が推奨。主観的なRejectが頻発すると開発者がAI指摘をノイズとして無視するリスクがある
- ルールの手動メンテは持続不可能。自動改善ループの構築が重要
- AI/人間の役割分担を明確にし、不可逆な変更は人間が握ることで安全性を担保

## 制限事項・注意点

- AIが検出できるのは**ルール化できるパターンに限定**される。設計のトレードオフやビジネス上の妥当性の判断は人間にしかできない
- カウシェの環境（モノレポ、迅速なロールバック可能、バグ時に即座に修正デプロイできる体制）が前提であり、すべてのチームにそのまま適用できるわけではない
- 決済・認証など不可逆な領域は引き続き人間レビューが必須
- AI/人間の境界をどこに置くかは現在も試行錯誤中

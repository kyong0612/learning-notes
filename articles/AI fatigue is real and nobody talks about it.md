## 要約

AIエージェントインフラのコアメンテナー（[OpenFGA](https://openfga.dev)、[agentic-authz](https://github.com/Siddhant-K-code/agentic-authz)、[Distill](https://distill.siddhantkhare.com)等）である著者が、AI活用による生産性向上の裏で深刻化する「AI疲労」の実態を率直に語り、持続可能なAI活用のための具体的な戦略を提示する記事。

---

### 誰も警告しなかったパラドックス

- AIは個々のタスクを確実に速くする（3時間→45分）。しかし、**日々の仕事はより楽になるのではなく、より困難になった**
- タスクが速く終わる→タスク数が増える→上司も自分も期待値を引き上げる→ベースラインが上昇
- 以前は1日1つの設計問題に集中できた。今は1日に6つの問題に触れる。**AIは疲れないが、人間はコンテキストスイッチで疲弊する**
- **核心的パラドックス: AIは「生産コスト」を下げるが、「調整・レビュー・意思決定のコスト」を増大させる。そしてそのコストはすべて人間が負担する**

### レビュアーへの変質

![AIがコンベアベルトに人間がレビューできるより速くコードを投下する](https://siddhantkhare.com/blog/ai-fatigue-is-real/3.png)

- 以前の仕事: 問題を考える→コードを書く→テスト→出荷（**創造者**）
- 現在の仕事: プロンプト→出力を読む→評価→正確性・安全性・アーキテクチャ整合性を判断→修正→再プロンプト（**品質検査官**）
- **生成的タスク（creating）はフロー状態を生む。評価的タスク（reviewing）は意思決定疲労を生む**
- AI生成コードは同僚のコードより慎重なレビューが必要。コードは自信に満ちて見え、コンパイルも通り、テストも通る可能性があるが、**本番環境で初めて表面化する微妙なバグを含む可能性がある**
- この問題への対策として、エージェントのセキュリティと認可（最小権限、スコープ付きトークン、監査証跡）が重要になる

### 非決定性の問題

![同じプロンプト、同じAI、異なる結果](https://siddhantkhare.com/blog/ai-fatigue-is-real/4.png)

- エンジニアは決定論（同じ入力→同じ出力）の上に訓練されている。**AIはその契約を破った**
- 月曜日に完璧に動いたプロンプトが、火曜日には構造的に異なる出力を返す。スタックトレースもログもない
- プロンプトのバージョン管理やテンプレート化で軽減は可能だが、**確率的システムと協働しているという根本的な不一致は解消できない**
- 著者はこの苛立ちから[Distill](https://distill.siddhantkhare.com)を構築した（LLM呼び出しなし、エンベディングなし、純粋なアルゴリズムによる〜12msのコンテキスト重複排除）
- **うまく対処している人は、AI出力を「賢いが信頼性の低いインターンの最初の草稿」として扱い、30%は書き直すことを前提にしている**

### FOMOトレッドミル

- 直近数ヶ月だけでも: Claude Codeのサブエージェント/スキル/Agent SDK、GPT-5.3-Codex、Gemini CLI、GitHub MCP Registry、A2Aプロトコル、Kimi K2.5のスウォームアーキテクチャなど、追いきれない量のリリース
- 著者は毎週末新ツールを評価→翌週には「もっと良い」ツールが登場→移行の繰り返し。**各移行で週末を消費し、測定不能な5%の改善しか得られない**
- **知識の減衰**: 2025年初頭に2週間かけて構築したプロンプトエンジニアリングワークフローが、3ヶ月後のモデル更新で陳腐化
- **採用した戦略**: ツールを追いかけるのではなく、その下のインフラ層（コンテキスト効率、エージェント認可、監査証跡、ランタイムセキュリティ）に深く取り組む。**ツールは去来する。解決すべき問題は残る**

### 「もう1回だけプロンプト」の罠

- 最初の出力が70%→プロンプト改善→75%だが以前の正しい部分が壊れる→80%だが構造が変化→**45分経過、ゼロから書けば20分で済んだ**
- これを「**プロンプトスパイラル**」と呼ぶ。AIのヤクシェービング。コードのデバッグではなくプロンプトのデバッグをしている
- 生産的に感じるため特に危険（反復し、少しずつ良くなっているように見える）
- **著者のルール: 3回試行してダメなら自分で書く。例外なし。** このルールが最も時間を節約した

### 完璧主義と確率的出力の衝突

- エンジニアは完璧主義的傾向がある。AIの出力は常に「まあまあ良い」（70-80%）
- **「ほぼ正しい」は「完全に間違っている」より厄介**。完全に間違いなら捨てて最初からやり直せる。ほぼ正しいと1時間微調整に費やす
- **対策**: AI出力を「ドラフト」とメンタルラベル付けする。この認識の変化だけでフラストレーションが半減
- **AIに最も苦戦するのは、しばしば最も優秀なエンジニア**（最も高い基準を持つ人）。AIは「不完全な出力から素早く価値を抽出する能力」を報いる

### 思考力の萎縮

![AIを見ながらソファに座る脳、思考筋にクモの巣](https://siddhantkhare.com/blog/ai-fatigue-is-real/5.png)

- 著者がホワイトボードで並行性問題を推論するよう求められた時、**概念は知っていたが、その筋肉を何ヶ月も使っていなかった**ため苦戦した
- GPS依存でナビゲーション能力が萎縮するのと同じ現象。**AIに最初の思考を外注し続けると、ゼロから考える能力が劣化する**
- **対策: 毎日最初の1時間はAIなしで過ごす**。紙で考え、手でアーキテクチャを描き、遅い方法で問題を推論する。非効率だが思考力を維持し、AI出力の評価能力も向上する

### 比較の罠

- SNSは「AIを完璧に使いこなしている人」のハイライトリールで溢れている
- 「AIでプロンプトがうまくいかず3時間無駄にした」「AI生成コードが本番障害を引き起こした」「疲れている」とは誰も投稿しない
- **対策**: デモだけしている人ではなく、実際にビルドしてシップしている人に注目する。**フィードが「情報を得た」ではなく「遅れている」と感じさせるなら、それは役に立っていない**

### 実際に効果があったこと

| 戦略 | 内容 |
|---|---|
| **AIセッションのタイムボックス** | 30分のタイマーを設定。時間切れたら手元のものを出荷するか、自分で書く |
| **AI時間と思考時間の分離** | 午前は思考、午後はAI支援の実行 |
| **AIから70%を受け入れる** | 完璧な出力を求めない。残りは自分で直す |
| **ハイプサイクルへの戦略的対応** | 1つのメインツールを深く使う。新ツールは数ヶ月の実績後に評価 |
| **AIの効果をログに記録** | 2週間のログ（タスク、AI使用有無、所要時間、満足度）で、AIが有効な場面と逆効果な場面を特定 |
| **すべてをレビューしない** | セキュリティ境界、データ処理、エラーパスに集中し、残りは自動テストと静的解析に委ねる |

### 持続可能性の問題

- テック業界のバーンアウト問題はAI以前から存在する。**AIはそれを悪化させている**。AIが悪いのではなく、以前は人間を守っていた自然な速度制限をAIが取り除いたため
- 以前はタイピング速度、思考速度、調査時間が天井だった。**AIがその制御装置を取り除き、唯一の限界が認知的持久力になった**
- 著者は2025年後半にバーンアウト。派手な崩壊ではなく、ただ**気にしなくなった**（コードレビューがスタンプ作業に、設計判断が「AIの提案通り」に）
- 皮肉にも、バーンアウト期に最良の仕事が生まれた。すべてのAIツールを追うのをやめ、実際に壊れているものを考えたとき、問題が初めてクリアに見えた→Distill、agentic-authz、AgentTraceの着想

### 本当のスキル

> AI時代の本当のスキルはプロンプトエンジニアリングでもモデル選択でもワークフローでもない。**「いつ止めるか」を知ることだ。**

- AI出力が十分かを知ること。自分で書くべき時を知ること。ラップトップを閉じる時を知ること
- **私たちはシステムに持続可能性のための設計（サーキットブレーカー、バックプレッシャー、グレースフルデグラデーション）を行う。自分自身にも同じことをすべきだ**
- この時代に成功するのは、AIを最も多く使う人ではなく、**最も賢く使う人**
- 「疲れているのは、やり方が間違っているからではない。これが本当に大変だからだ」

## 概要

広木大地氏（株式会社レクター代表取締役、日本CTO協会理事）による Dev Summit プレゼンテーション。新刊『AIエージェント 人類と協働する機械』の内容に基づき、**AIで生産性が上がったと錯覚してしまう3つの構造的理由**を解説する。

---

## 期待と現実のギャップ

### 期待
- コーディング時間の大幅削減
- 複雑なタスクも自動化
- AIエージェントで10倍以上の生産性

### 現実
- 「AI疲れ」という新たな問題
- 場合によっては遅くなることすら
- **2〜3倍程度で頭打ち**

> **体感の速さは、そのまま価値の増加を意味しない。** 価値は最終成果（売上・利益・顧客価値・プロダクト）でしか確定しない。チャットで指示してその結果を眺めているだけでは生産性は上がらない。

この体感と現実の不一致には**3つの構造的な理由**がある。

---

## 理由① 本質的複雑性が残る

> 一部が10倍早くなっても全体は10倍にならない

### 本質的複雑性 vs 偶有的複雑性（Fred Brooks, 1986「銀の弾丸などない」）

| 本質的複雑性 | 偶有的複雑性 |
|---|---|
| AIでも高速化困難 | AIが得意・大幅に高速化 |
| 技術では解決できない | 技術で削減可能 |
| 人間の要求の曖昧さ | バグやパフォーマンス問題 |
| 問題領域そのものの複雑さ | 実装に伴う技術的複雑さ |

### アムダールの法則 — 並列化できない部分が全体を支配する

**並列化できる（AI向き）：** 環境構築、ドキュメント作成、単体テスト生成、実装・コーディング

**並列化できない（人間）：** レビュー・承認、要件定義、設計判断、顧客の課題の特定

- 本質的30%の場合 → 偶有的を10倍にしても全体は**約2.7倍**
- 本質的50%の場合 → 偶有的を10倍にしても全体は**約1.8倍**

> **多くの企業が2-3倍で止まる理由は、本質的複雑性が30-50%を占めているから。**

### AI疲れ（AI Fatigue）

意思決定はHPではなく**MP（精神力）を削る**。人間のMPはそうそう変わらない。

HBR（2026.2）Ranganathan & Ye, UC Berkeley Haas の調査「AI Doesn't Reduce Work—It Intensifies It」によると、AIは作業を減らさず**3つの次元で仕事を強化する**：

1. **タスクの拡張** — PMがコードを書き、リサーチャーがエンジニアリングを担当。AIが「認知的ブースト」となり従来は他者に委ねていた領域に踏み込む
2. **境界の消失** — 仕事が「ambient（常在的）」になり、休息が回復として機能しなくなる
3. **マルチタスクの増加** — 手動コーディングとAI生成を並行、複数エージェントを同時稼働し、注意の切り替えが絶え間なく続く

### AIの自律性と認知負荷

- **低い自律性**：AIが5分動く → 確認・指示 → AIが5分動く… → 人間の作業密度が異常に高まる
- **高い自律性**：指示10分 → AIが1-2時間稼働 → 確認5分 → 認知負荷の大幅な軽減

**コンテキストエンジニアリング**でAIの自律性を高める：
1. ルール化・明文化 — 繰り返すフィードバックをドキュメント化
2. 手順の明確化 — 判断基準や参照先を明示的に提示
3. フィードバックループ — 戻れる仕組み（Issue、コメント）を整備

> 明文化が進むほど、AIは長く走り、人間は高次の判断に集中できる。

### 本質的複雑性への2つのアプローチ

1. **比率を下げる**（アーキテクチャと設計の力）— 認知負荷の削減、テンプレート化、関心事の分離、DDD
2. **発生源に向き合う**（問題解決プロセス自体のAI化）— 価値観の構造化、コンテキストの提供、暗黙知の形式知化

## 理由② 消える生産性（Vanishing Productivity）

> 速くなったのに、どこにも届かない

**定義：** 効率化によって生まれた時間や余力が、組織の慣性に吸収され、最終的な価値創造につながらない現象

### 組織の慣性に吸収される

空いた時間は**ブルシット・ジョブ**に吸収される。または「ランチを長く、同僚と会話する時間が増えた」（ゆとりのある労働）に消える。

### Faros AI（2025）10,000名以上の開発者テレメトリ分析

| チームレベル（改善あり） | 組織レベル（改善なし） |
|---|---|
| 1日あたりのPR処理数 +47% | 品質KPI：改善なし |
| マージされたPR数 +98% | DORA指標：改善なし |
| タスク完了数 +21% | 全社スループット：有意な相関なし |

> PRレビュー時間 +91%、バグ発生率 +9% — **下流のボトルネックが価値を吸収する**

### AI生産性の「神話」と現実（California Management Review 2025.10, Gruda & Aeon）

| 神話 | 現実 |
|---|---|
| AIを導入すれば誰でも生産性が上がる | 37研究のレビューで、コード品質の低下と再作業が効果を相殺。恩恵は初心者に偏る |
| 人間+AIのチームが最強の組み合わせ | Nature Human Behaviourの106実験メタ分析で、人間+AIは単独の優秀な方より悪い成果を出す場合がある |
| AIは人間より創造的 | 28実験（8,214名）で、AIは新規性を高めるがアイデアの多様性は激減。集団の創造性は低下 |
| ミクロの効率化がマクロ経済に波及する | 371推定値のメタ分析で、AI採用と労働市場成果の間に頑健な関係は見出されていない |
| 自動化すれば認知負荷が下がる | 74研究のレビューで、高精度だが不完全な自動化により誤判断が12%増加 |
| ルーティンの自動化でストレスが減る | ルーティンが消えると残るのは判断と例外処理だけ。高密度な意思決定がバーンアウトを誘発 |

### マクロ経済への影響

GDPレベルでも**2035年に+1.5%、2055年に+3%程度**と微小（ペンシルベニア大学ウォートンスクール推計）。

### 消える生産性への解決策 — 価値を軸に仕事を再設計する

1. **やめる仕事を決める** — 価値の低い仕事を積極的に削減
2. **成果指標を変える** — 忙しさではなく価値で測る
3. **学習へ再投資する** — 効率化で生まれた時間を能力開発に

> 生産性向上は、単体では成果にならない。価値創造を同時に設計する。

## 理由③ 競争は相対優位で決まる

> 周りも同じ速度で上がっている

### 絶対速度 vs 相対速度

- **絶対速度**：「自分がどれだけ速くなったか」→ 体感の指標
- **相対速度**：「周りより相対的に有利か」→ 競争の指標

> 速度向上は**競争の参加資格（Table Stakes）**でしかない。本質的な競争はAIとではなく、人と人、企業と企業で起きている。

### AIは均質化の力である

- 優位性の源泉は「残余の異質性」— AIが届かない人間の創造性と実行力に移行（MIT Sloan Management Review 2025）
- AIの利用が普遍化すると、それ自体は持続的な差別化にならない

### 価値を出せている企業はごくわずか（BCG 2024, 59カ国・1,000名CxO調査）

- 74%の企業がPoC止まりでスケールに苦戦
- 全社規模で価値を創出しているのは**わずか4%**
- AIから成果を出せているのは26%
- 成功企業はリソースの**70%を人とプロセスに投資**している

### 赤の女王の競争

> 「同じ場所にとどまるためには、全力で走り続けなければならない」— 鏡の国のアリス

- 落ち着いてから参入では遅すぎる
- 相対的なアービトラージがあるときに付加価値の効率が上がる
- AIは民主化され世界中に配られている

### 相対優位への解決策 — 学習速度が相対位置を決める

- **学習速度が遅い組織**：改善が一回の施策で終わり、相対的に後退
- **学習速度が速い組織**：改善が常態化し、同じAIでも相対優位を積み上げる

> 「生産性が上がったか」は「相対的に前進したか」で検証すべき。

## ボトルネックのシフト：実装から意思決定へ

### プログラミング vs エンジニアリング

| プログラミング | エンジニアリング |
|---|---|
| 既知の解法を正確に実装する | 未知の問題を試行錯誤で探索する |
| 速さと正確さが価値 | 仮説→実験→学習のサイクル |
| 正解がある、入出力が明確 | 正解がない中で判断する |
| → **AIが最も得意な領域** | → **人間の仕事として残る** |

### 実装中心 → 意思決定中心へ

- **これまで**：作業量が成果に見えやすい、How（どう作るか）に時間が溶ける
- **これから**：長期判断・創造性を含む、「遅い・迷う・揉める」領域、What / Why に寄る

> 暗黙知を取り出し、AIが扱える形に変換することが新しいエネルギー源になる。

## AI時代の「新しい正しさ」— アンラーニング12項目

### できる化 → 自働化 → 自創化

1. **できる化（頼む）**：人がやっていた仕事をAI活用で「自分でできるようにする」
2. **自働化（任せる）**：ワークフローやAgenticな仕組みで仕事をAIに任せる
3. **自創化（指し示す）**：仮説検証・創造フェーズも自動化し、人間は方向性の承認のみ

### アンラーニング12項目

| # | 旧常識 | 新しい正しさ |
|---|---|---|
| ❶ | タスクは1個ずつ順番にやる | 100件あるなら100件同時に仕掛ける |
| ❷ | 自分はToDoをこなす側 | 人間はToDoを生み出す側 |
| ❸ | 作業を速くすれば全体が良くなる | 全体のアウトプット構造を変える |
| ❹ | 5人で半年のプロジェクト粒度 | タスクの粒度を10〜20倍に引き上げる |
| ❺ | 議論してから作業する | 作業が先、議論は後 |
| ❻ | フロント/バックの役割分担 | 実装は全員がやり、得意な人はイネーブルメントに |
| ❼ | 5万行のコードは大きい | 5万行は書き捨てサイズ、10〜100万行が1人の範囲 |
| ❽ | レビューは人間 | AIレビュー→人間は方針判断に集中 |
| ❾ | 1リポジトリで1チーム | 目的の単位でリポを切る、一人1リポでもいい |
| ❿ | 1チームは2Pizza | 2-3人でも十分なアウトプットが出せる |
| ⓫ | PR粒度は小さいほうがいい | AIレビュー前提なら大きくてもいい |
| ⓬ | エンジニアじゃない人はソフトウェアをつくれない | 非エンジニアもエージェントでプロダクトに直接触れる |

## 知の探索へのシフト

### 暗黙知と「問い」

暗黙知とは「言葉にされていない判断基準」や「経験則の集積」。これらから知識を生み出す知識創造のシステムが次の価値の源泉になる。

1. **問いを立てる** — 何を解きたいのか言葉にする → 暗黙知の入口
2. **指示を出す** — 判断基準を明文化する → 形式知への変換
3. **結果を検証する** — 期待とのズレから学ぶ → 知識の精緻化

### SECI × AIエージェント

| フェーズ | 内容 |
|---|---|
| 共同化（暗黙知→暗黙知） | 会議・メール・顧客対話から暗黙知を自動収集 |
| 表出化（暗黙知→形式知） | 切片化→グルーピング→インサイト生成の3ステップ |
| 連結化（形式知→形式知） | 既存知識との差分で改善提案を構造化 |
| 内面化（形式知→暗黙知） | リアルタイム実践支援、個別最適化された学習パス |

### オーケストレーターとしての人間

- **人間**：優先順位を調整、提案を承認・却下、全体の方向性を決める
- **AI**：実行結果を報告、選択肢を提示、継続的に改善を回す

> "生産性"は、手を動かす量ではなく**「知が増える速度」**へ置き換わる。

## 実践事例：AIエージェントで書籍を20倍の生産性で執筆

### 前作との比較

| | 前作『エンジニアリング組織論への招待』 | 本書『AIエージェント 人類と協働する機械』 |
|---|---|---|
| 生産性 | 通常 | 20倍以上 |
| 執筆時間 | 1日に数時間 | 隙間時間のみ |
| 執筆期間 | 約1年 | 約1ヶ月強 |

### 執筆データ
- 総文字数：**35万文字**
- 構成：4部16章60節
- 期間：初稿1週間+洗練3週間
- GitHub Issue：**774件**
- **筆者の直接執筆：ゼロ文字**（すべてAIとの対話で生成）

### 20倍を実現した設計

1. **偶有的複雑性の削減** — 一晩で15万文字を生成、60節を同時並列で執筆
2. **本質的複雑性への対処** — 暗黙知をコンテキストとして提供、アウトライン設計で事前解決

### 自律性を実現する3つの設計

- **アウトライン設計**：各節の役割と論理的流れ、「何を書くべきか」を事前解決
- **文体指示書**：AI臭さの脱臭、用語統一、トーン＆マナーの明文化
- **コンテキスト設計**：経験と価値観を構造化、文献調査、1on1議事録から知見抽出

### ガードレールとプリンシプル

- **ガードレール**（最低基準）：ファクトチェック、文章ルール、他エージェントからの指摘反映
- **プリンシプル**（価値観と判断基準）：何が重要な価値観なのか明文化し、自律的な修正を機能させる

### 「積み上げ型」から「削り出し型」へ

- **従来（積み上げ型）**：部分から全体へ、レンガを一個ずつ積み上げる
- **AI時代（削り出し型）**：全体像から詳細へ解像度を上げる、木から彫刻を削り出すように

## 明日からできること

1. **本質的複雑性を特定する** — 「AIでは速くならない部分」はどこか、チームで洗い出す
2. **余力の行き先を決める** — AIで生まれた時間を何に使うか、事前に決めておく
3. **学習ループを1つ作る** — 相対速度を上げるための改善サイクルを、小さく1つ始めてみる

## まとめ

| 構造的理由 | 要約 | 解決策 |
|---|---|---|
| ❶ 本質的複雑性が残る | 一部が10倍早くなっても全体は10倍にならない | 比率を下げる / 発生源に向き合う |
| ❷ 生産性が消える | 次の一手が設計されていない | 価値を軸に仕事を再設計する |

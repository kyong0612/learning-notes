## 概要

Anthropic の Claude Code チームエンジニア **Thariq Shihipar** による、Claude Code Skills の実践的ガイド。Anthropic 内部では数百の Skills が活発に使用されており、その経験から得られた「どんな種類の Skills を作るべきか」「どう書くか」「どうチームに広めるか」の知見を体系的にまとめている。

> Skills は Claude Code で最も広く使われている拡張ポイント（extension points）の一つ。柔軟で作りやすく、配布も簡単。

---

## Skills とは何か

よくある誤解として「Skills は単なる Markdown ファイル」と思われがちだが、実際には **フォルダ** であり、以下を含むことができる：

- スクリプト
- アセットファイル
- データ
- 設定ファイル（config.json など）

エージェントはこれらを発見・探索・利用できる。さらに Claude Code では[豊富な設定オプション](https://code.claude.com/docs/en/skills#frontmatter-reference)があり、動的フック（hooks）の登録も可能。

**最も面白い Skills は、設定オプションとフォルダ構造を創造的に活用しているもの。**

## Skills の9つのカテゴリ

Anthropic 内部の全 Skills を整理した結果、以下の反復的なカテゴリに分類できることが判明。**優れた Skills は1つのカテゴリにきれいに収まり、混乱を招く Skills は複数カテゴリにまたがる。**

### 1. ライブラリ & API リファレンス

ライブラリ、CLI、SDK の正しい使い方を教える Skills。内部ライブラリや、Claude が間違えやすい外部ライブラリ向け。

**例：**
- `frontend-design` — デザインシステムの理解を向上
- `internal-platform-cli` — 内部 CLI の各サブコマンドと使用例
- `billing-lib` — 内部課金ライブラリのエッジケースと落とし穴

### 2. プロダクト検証

コードが正しく動作するかテスト・検証する方法を記述する Skills。Playwright、tmux などの外部ツールと組み合わせる。

> **検証 Skills の精度向上に、エンジニア1人が1週間集中して取り組む価値がある。**

テクニック例：
- Claude に出力プロセスの動画を記録させる
- 各ステップでプログラム的な状態アサーションを強制する

**例：**
- `tmux-cli-driver` — TTY が必要なインタラクティブ CLI テスト
- `checkout-verifier` — Stripe テストカードでチェックアウト UI を駆動し、請求書の最終状態を検証
- `signup-flow-driver` — ヘッドレスブラウザで登録→メール認証→オンボーディングフローを実行

### 3. データ取得 & 分析

データ・モニタリングスタックに接続する Skills。認証情報付きのデータ取得ライブラリ、ダッシュボード ID、ワークフロー説明を含む。

**例：**
- `grafana` — データソース UID、クラスタ名、問題→ダッシュボード対応表
- `cohort-compare` — 2つのユーザー群のリテンション/コンバージョンを比較し、統計的有意差をマーク
- `funnel-query` — 登録→アクティベーション→課金のファネルに必要なイベント結合情報

### 4. ビジネスプロセス & チーム自動化

反復的なワークフローを1コマンドに自動化する Skills。指示はシンプルだが、他の Skills や MCP に依存することがある。

> **過去の実行結果をログファイルに保存すると、モデルの一貫性維持と振り返りに有効。**

**例：**
- `weekly-recap` — マージ済み PR + クローズ済みチケット + デプロイ記録 → 週報
- `create-ticket` — スキーマ強制（有効な enum 値、必須フィールド）+ 作成後ワークフロー
- `standup-post` — タスクトラッカー + GitHub アクティビティ + Slack → 差分のみの日報

### 5. コードスキャフォールディング & テンプレート

特定機能のボイラープレートコードを生成する Skills。スクリプトと組み合わせて使用。**自然言語の要件があり、純粋なコードだけではカバーできないスキャフォールディングに最適。**

**例：**
- `create-app` — 認証・ログ・デプロイ設定済みの新規内部アプリ
- `new-migration` — DB マイグレーションテンプレート + よくある落とし穴
- `new-workflow` — アノテーション付きのサービス/ワークフロー/ハンドラの雛形

### 6. コード品質 & レビュー

チーム内のコード品質基準の執行とコードレビュー支援。決定論的スクリプトやツールを含めて信頼性を最大化。**フックの一部として自動実行するか、GitHub Action で実行することを推奨。**

**例：**
- `testing-practices` — テストの書き方と対象に関するガイダンス
- `code-style` — Claude がデフォルトで苦手なコードスタイルの強制
- `adversarial-review` — 新しい視点のサブエージェントを生成してコードを批評させ、修正を繰り返す

### 7. CI/CD & デプロイ

コードのプル・プッシュ・デプロイを行う Skills。他の Skills を参照してデータを収集する場合がある。

**例：**
- `cherry-pick-prod` — 隔離 worktree → cherry-pick → コンフリクト解決 → テンプレート PR 作成
- `deploy-service` — ビルド → スモークテスト → 段階的トラフィック切り替え＋エラー率比較 → 悪化時自動ロールバック
- `babysit-pr` — PR 監視 → flaky CI リトライ → マージコンフリクト解決 → 自動マージ有効化

### 8. ランブック（Runbooks）

現象（Slack メッセージ、アラート、エラーシグネチャなど）を受け取り、マルチツール調査フローを案内し、構造化レポートを生成する Skills。

**例：**
- `log-correlator` — リクエスト ID から関連するすべてのシステムのログを取得
- `oncall-runner` — アラート取得 → 共通の原因チェック → 調査結果をフォーマット
- `service-debugging` — 現象をツール→クエリパターンにマッピング

### 9. インフラ運用

日常的なメンテナンスと運用タスクを実行する Skills。**破壊的操作にはセーフティガードレールが必要。**

**例：**
- `cost-investigation` — ストレージ/帯域コスト急増の原因調査（具体的なバケットとクエリパターン付き）
- `dependency-management` — 組織の依存関係承認ワークフロー
- `orphan-cleanup` — 孤立した Pod/Volume を検出 → Slack 通知 → 待機 → ユーザー確認 → カスケード削除

## Skills 作成のベストプラクティス

### 当たり前のことを書かない

Claude Code はコードベースをよく理解しており、プログラミングにも詳しい。**知識提供が主目的の Skill なら、Claude の通常の思考パターンを破る情報に焦点を当てる。**

> [frontend-design Skill](https://github.com/anthropics/skills/blob/main/skills/frontend-design/SKILL.md) が好例。Anthropic のエンジニアがユーザーとの反復を通じて、Inter フォントや紫のグラデーションなどの「典型的な AI デザイン」を避けるように構築した。

### 踩坑点（Gotchas）セクションを作る

> **どんな Skill でも最も情報量が多いのは Gotchas セクション。** Claude がその Skill を使う際に遭遇する共通の失敗パターンを蓄積していく。理想的には、継続的に Skill を更新してこれらを記録する。

### ファイルシステムと渐進的披露（Progressive Disclosure）を活用する

Skill はフォルダであり、**ファイルシステム全体をコンテキストエンジニアリングと渐進的披露のツールとして活用すべき。**

- 詳細な関数シグネチャや使用例を `references/api.md` に分離
- 最終出力がMarkdownファイルの場合、`assets/` にテンプレートを配置
- `references/`、`scripts/`、`examples/` フォルダを活用

Claude に Skill 内のファイル一覧を伝えておけば、必要な時に自動的に読み込む。

### Claude を縛りすぎない

Skills は再利用性が高いため、**指示を具体的に書きすぎない。** 必要な情報は与えるが、状況に応じた柔軟性を残す。

### 初期セットアップを考慮する

ユーザーコンテキストが必要な Skills は、初回実行時に情報を収集する仕組みを入れる。

- 設定情報を Skill ディレクトリの `config.json` に保存
- 構造化された選択肢を提示する場合は `AskUserQuestion` ツールを使用

### description フィールドはモデル向け

> **description はサマリーではなく、「いつこの Skill をトリガーすべきか」を記述する。** Claude Code はセッション開始時に全 Skills の description を一覧としてスキャンし、リクエストに対応する Skill があるか判断する。

### 記憶とデータストレージ

Skills はデータを内部保存することで記憶を実現できる：

- シンプル：追記型テキストログや JSON ファイル
- 高度：SQLite データベース

> Skill ディレクトリのデータはアップグレード時に削除される可能性があるため、`${CLAUDE_PLUGIN_DATA}` を安定したストレージディレクトリとして使用する。

### スクリプトと生成コードの保存

Claude に最も強力なツールの一つは **コード** そのもの。データ取得用の関数ライブラリなど、ヘルパー関数を提供することで、Claude は組み合わせと意思決定に集中できる。

### オンデマンドフック

Skill が呼び出された時だけアクティブになり、セッション中は有効であり続けるフック。主観的で常時実行は不要だが、特定状況で有用なケース向け。

**例：**
- `/freeze` — 特定ディレクトリ外への Edit/Write を阻止（デバッグ時に「関係ないコードを勝手に修正する」のを防止）
- `/careful` — `PreToolUse` で `rm -rf`、`DROP TABLE`、`force-push`、`kubectl delete` をインターセプト（本番環境操作時のみ有効化）

## Skills の配布

### 2つの共有方法

| 方法 | 適用場面 |
|------|----------|
| **プラグインマーケットプレイス** | 規模が大きい組織向け。ユーザーがアップロード・インストール可能 |
| **リポジトリにコミット** （`./.claude/skills`） | 少数のリポジトリで協業する小規模チーム向け |

リポジトリにコミットされた各 Skill はモデルのコンテキストに負荷を加えるため、**規模が拡大したら内部プラグインマーケットプレイスが推奨。**

### マーケットプレイスの運営

Anthropic では専任の中央チームはなく、**最も有用な Skills が自然に浮上するアプローチ** を採用：

1. 試してほしい Skill を GitHub のサンドボックスフォルダにアップロード
2. Slack などで推薦
3. 十分な支持を得たら PR でマーケットプレイスに移動

> **品質が低い・重複した Skills は簡単に作れてしまうため、公式リリース前の審査メカニズムが重要。**

### Skills の組み合わせ

Skills 間の依存関係（例：ファイルアップロード Skill を CSV 生成 Skill が呼び出す）は現時点では正式にサポートされていないが、**名前で他の Skills を参照すれば、インストール済みであればモデルが呼び出す。**

### Skills の効果測定

`PreToolUse` フックを使って社内で Skill の使用状況をログ記録。人気の Skills や、トリガー頻度が低い Skills を発見できる。

- [計測用サンプルコード](https://gist.github.com/ThariqS/24defad423d701746e23dc19aace4de5)

## 結論

- Skills は AI エージェントにとって非常に強力で柔軟なツールだが、**まだ早期段階** であり、最適な活用方法は模索中
- この記事は権威的ガイドではなく、**実践で検証された実用テクニック集** として捉えるべき
- **最良の学習方法は自分で作り始めること** — 多くの Skills は数行のテキストと1つの Gotcha から始まり、エッジケースの蓄積で徐々に改善された
- Skills の理解を深める最良の方法は、実際に手を動かして実験すること

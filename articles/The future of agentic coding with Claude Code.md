## タイムスタンプ付き要約

### [00:00] イントロダクション

- Boris Chernyが自己紹介。Anthropicの技術スタッフメンバーであり、Claude Codeの創設者
- すでにClaude CodeやAIコーディングツールに詳しい聴衆に向けて、まず俯瞰的な視点から話し、その後具体的なTipsに掘り下げる構成
- **TLDR**: モデルは指数関数的に進化しコーディング能力が急速に向上しているが、プロダクトがそれに追いつくのに苦労している。Claude Codeは最小限のプロダクトを構築し、プロダクトの最適な形について意図的にオピニオンを持たない方針

### [01:56] プログラミングの進化史 — 言語の抽象化

- **1930〜40年代**: スイッチボードによる物理的なプログラミング。ソフトウェアという概念は存在しなかった
- **1950年代**: パンチカードの登場。Boris の祖父はソ連初期のプログラマーの一人で、1970年代まで大量のパンチカードを自宅に持ち帰っていた
- **1950年代後半**: 高水準言語の出現。アセンブリ → COBOL → 型付き言語 → C++
- **1990年代初頭**: 新しい言語ファミリーの爆発的増加（Haskell系、JavaScript、Java、C系の進化、Python）
- **現代**: 言語の抽象化が収束。TypeScript ≒ Rust ≒ Swift ≒ Go — どの言語も似たような感覚で書ける時代に

### [03:23] プログラミングUXの進化

- **パンチカード時代**: IBM 029（当時の「MacBook」）でパンチカードにプログラミング
- **最初のテキストエディタ「Ed」**: Ken Thompsonが発明。50年経った今もUnixシステムに搭載されている。カーソルもスクロールバックもない超シンプルな設計で、テレタイプマシン向けに構築された
- **1980年 Smalltalk-80**: プログラミング用の最初のGUI。**1980年にライブリロードを実現**していた（ReactJSで今でも苦労する機能）
- **1991年 Visual Basic**: GUIベースのコードエディタを主流に
- **Eclipse**: 静的解析による型先行入力を主流に。最初の大規模サードパーティエコシステム
- **Copilot**: 単一行→複数行の型先行入力
- **Devin**: 「自然言語がコードになる」という概念を初めて主流化した製品

### [07:13] 検証の進化とClaude Codeのアプローチ

- 検証は手動デバッグ → ファジング、脆弱性テスト、Netflixのカオステストなどの確率的検証へ進化
- **Claude Codeの設計思想**:
  - ターミナルからスタートし、モデルへの**低レベルアクセス**を提供
  - 意図的にシンプルで汎用的。派手なUIやスキャフォールディングを避ける
  - 理由1: Anthropicはモデル企業であり、モデルそのものを体験してほしい
  - 理由2: **最適なUXがまだわからない**。だからシンプルに始める
- 引用: "The more general model always wins"（Bitter Lesson）— モデルの周辺も含め、より汎用的なものが常に勝つ

### [09:02] Claude Codeの製品ラインナップ

#### ターミナル製品
- あらゆるターミナルで動作（iTerm 2、WSL、SSH、tmux、VS Code、Cursor）
- 完全にオピニオンを持たない設計

#### IDE統合
- IDEで実行すると差分表示が大きく美しくなり、診断情報も取り込む
- CursorやWindsurfほど洗練されていないのは意図的 — モデルを低レベルで体験させることが目的

#### GitHub統合
- `/install GitHub app`コマンドでセットアップ完了
- ユーザーのコンピュート上で実行され、データもユーザー側に留まる
- 既存のスタックを変更する必要なし

#### SDK（最も極端な形態）
- `claude -p` でプログラマティックに使用可能
- **Unixユーティリティとしてのモデル**: GCPログをパイプして`jq`で処理するなど
- 引用: "モデルをUnixユーティリティとして使う方法はまだ10%しか探索されていない"

### [11:55] Claude Code活用Tips

#### Tip 1: コードベースQ&A
- 新しいユーザーへの最もシンプルな導入方法
- Anthropicでは初日から全エンジニアにClaude Codeを教え、**オンボーディング期間を2〜3週間から約2日に短縮**
- 日常使い: 「今週何をシップした？」とgitコミットを確認させる

#### Tip 2: Claudeにツールの使い方を教える
- 従来のIDEはプラグイン構築が必要だったが、Claude Codeは既存のツールをそのまま使える
- 手順: CLIツールの`--help`を読ませ → 学んだことをCLAUDE.mdに書かせる → 完了
- ブリッジもエクステンションも不要
- MCP（Model Context Protocol）でグループツールや高度な機能（ストリーミングなど）にも対応

#### Tip 3: 探索→計画→承認→実行のワークフロー
- Claudeにコードを書く前にまず探索させ、計画を立てさせ、承認を得てから実行
- **拡張思考（Extended Thinking）**: コンテキストが十分にある状態で使うと効果的。最初から思考させるとトークンの無駄
- **Plan Mode**: Shift+Tabで切り替え。Claudeは読み取り専用で計画を作成し、承認を待つ

#### Tip 4: テスト駆動開発（TDD）
- TDDは実践では難しいが、コーディングツールと組み合わせると非常にうまくいく（モデルが実行するから）
- ワークフロー: テストを先に書かせる → コミット → コードを書かせる → コミット
- **一般原則**: Claudeに反復対象（ターゲット）を与えると結果の品質が大幅に向上
  - ユニットテスト、統合テスト、iOSシミュレータのスクリーンショット、Puppeteerなど
  - 3Dプリンタ＋カメラでClaude Codeを使った例も
  - 引用: "1回目はそこそこだが、2〜3回目の反復でかなり良くなる"

#### Tip 5: コンテキストの追加
- **CLAUDE.md**: リポジトリのルートに配置。サブフォルダやホームフォルダにも配置可能
- **スラッシュコマンド**: `commands`フォルダ内のMarkdownファイルがスラッシュメニューから利用可能。再利用可能なワークフローに有用
- **記憶化**: `#`を入力してClaudeに記憶させる。どのメモリに追加するか選択可能
- メモリUXはまだ初期段階だが、機能する最初のバージョンであり、フィードバックを求めている

### [17:01] Q&A: 並列ワークロードの管理

- パワーユーザーは**マルチClaude**を活用 — 複数のターミナルタブで異なるチェックアウトやワークツリーを使い、Claudeを並列実行
- GitHub Actionsでも複数のアクションを生成して並列処理可能
- Claude間の調整が必要な場合は**Markdownファイルへの書き込み**で対応

## キーメッセージ

1. **モデルは指数関数的に進化し、プロダクトが追いつくのに苦労している**: AI開発ツールの最適なUXはまだ未知であり、Claude Codeは意図的にシンプルで汎用的な設計を採用
2. **Bitter Lesson（苦い教訓）の適用**: より汎用的なモデル・ツールが常に勝つ。モデルの周辺も含め、汎用性を重視する設計
3. **プログラミングUXは指数関数的に進化中**: パンチカード→Ed→Smalltalk→Visual Basic→Eclipse→Copilot→Devin→Claude Codeへの進化。次の飛躍はさらに速くなる
4. **「オピニオンを持たない」設計哲学**: 最適なプロダクト形態がわからないため、最小限のプロダクトで学習し反復する。ターミナル、IDE、GitHub、SDKと多様な利用形態を提供
5. **反復対象を与えることが品質向上の鍵**: TDD、スクリーンショット、テストなどClaudeが出力を検証できる仕組みを用意すると、2〜3回の反復で大幅に品質が向上する

## 結論・示唆

### 話者の結論

- プログラミング言語の抽象化は収束したが、プログラミングの**UXは指数関数的に進化**し続けている
- Claude Codeは意図的に最小限のプロダクトとして設計されている。理由は、モデルの進化が速すぎて最適なプロダクト形態を特定できないため
- 「モデルをUnixユーティリティとして使う」という新しいパラダイムはまだ10%しか探索されておらず、巨大な可能性が残っている

### 実践的な示唆

- [ ] Claude Codeを**コードベースQ&A**に活用し、オンボーディングを加速させる
- [ ] **CLAUDE.md**に積極的に投資し、プロジェクト固有の知識をClaudeに教える
- [ ] 複雑なタスクは**Plan Mode**（Shift+Tab）で計画を先に作り、承認してから実行する
- [ ] **TDDワークフロー**を採用し、Claudeに反復対象を与えて品質を向上させる
- [ ] **並列ワークロード**を試す — git worktreeやGitHub Actionsで複数のClaudeを同時に稼働させる
- [ ] MCP統合で既存のツールチェーンをClaudeに接続する

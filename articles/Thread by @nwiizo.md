## 背景知識

### クリステンセンのジョブ理論 (Jobs-to-be-Done)

ハーバード・ビジネス・スクール教授クレイトン・クリステンセンが提唱した理論。顧客は製品そのものではなく、「特定の状況で成し遂げたい進歩（ジョブ）」を解決するために製品を「雇用」するという考え方。ジョブには機能的側面だけでなく、感情的・社会的側面が含まれる。

### Fred Brooks「銀の弾などない」における複雑性の二分化

Fred Brooksが1986年の論文で提唱した概念:

- **本質的複雑性 (Essential Complexity)**: ソフトウェアが解決しようとする問題そのものに固有の困難さ。要件定義、設計、テストなど。技術革新では根本的に解消できない。
- **偶有的複雑性 (Accidental Complexity)**: 実装プロセス（言語、ツール、手法）に起因する困難さ。高水準言語やフレームワークの進化により段階的に軽減されてきた。

### スレッドの論理構造

```
本質的複雑性 = 顧客の真のジョブの発見・定義（人間の洞察力が必須）
偶有的複雑性 = 実装・コーディング・ツール周りの作業（AIが高速化可能）

→ AIは偶有的複雑性をゼロに近づけられても、
  本質的複雑性（全体の価値創出の上限）には対処できない
→ したがって、「何を作るか」を見極める能力の重要性はAI時代にむしろ増大する
```

## 重要な示唆

1. **AIの限界**: AIが得意とするのは偶有的複雑性（コード実装、作業効率化）の領域であり、本質的複雑性（正しい問題の発見・定義）は依然として人間の洞察に依存する
2. **価値の源泉の移動**: AI時代において、実装スキルよりも「顧客の真のジョブを見極める力」がエンジニアの差別化要因となる
3. **現場の実態**: 正しいジョブ定義ができる人材は全体の5%程度しかおらず、大多数は表面的な要件をそのまま実装して失敗している
4. **構造的不変性**: 技術がどれだけ進歩しても、本質的複雑性が全体の生産性の上限を決める構造は変わらない（Brooksの主張と一致）

## 概要

「LLMは空間的な推論が苦手」「複雑な構成図はぐちゃぐちゃになるはず」──著者が長年持っていたこの前提を、Claude Code（Opus 4.6モデル）でAWSアーキテクチャの構成図をdraw.io XML形式で段階的に複雑化しながら生成することで検証した記事。結論として、27コンポーネント・48本以上の接続線を持つ複雑な構成図でも、階層構造・対称性・グルーピングがかなりの精度で再現され、「LLMでは複雑な図は作れない」という前提はもはや成り立たなくなっていた。

## 主要なトピック

### 「ぐちゃぐちゃな図」の定義

構成図が「読みにくい」と感じる具体的な特徴を5点に整理：

- **論理的なグルーピングの欠如**: VPC・サブネット・AZの包含関係が図の配置から読み取れない
- **空間の無駄遣いと密集の共存**: 一部がスカスカで一部が異常に密集
- **ラベルの重なり**: コンポーネント名やコネクタのラベルが他オブジェクトと重なる
- **配置の一貫性欠如**: 同一レイヤーのコンポーネントが上下左右バラバラ
- **矢印の交差過多**: コネクタ同士が何度も交差し接続先が追いにくい

### グラフ理論から見た「複雑な図は無理」という仮説

#### 平面グラフの辺数上限

- 単純連結平面グラフでは **m ≤ 3n - 6** が成り立つ（n：頂点数、m：辺数）
- コンポーネント10個なら辺24本を超えると交差は不可避

#### 交差補題（Crossing Lemma）

- 辺密度が高くなると交差数は **辺数の3乗に比例** して急増
- コンポーネント追加の影響は線形ではなく急激に悪化する

#### LLMにとっての困難さ

- グラフの交差数最小化問題は **NP困難**
- 人間はヒューリスティクス（重要度による優先、グルーピング、対称性、階層構造）で整理するが、LLMはテキスト（XML）として座標値を生成するため空間推論は苦手なはず──というのが著者の仮説だった

### 実践: 3段階の検証

#### Step 1: シンプルな構成（ALB + EC2×2 + RDS）

| 指標 | 値 |
|------|-----|
| コンポーネント数 (n) | 5 |
| 接続線数 (m) | 5 |
| 辺密度 (m/n) | 1.0 |
| 平面上限 (3n-6) | 9 |

- トラフィックの流れ（上→下）が直感的に読み取れる
- AZ-aとAZ-bの左右分割も適切
- この程度なら問題なしと予想通り

#### Step 2: 中程度の構成（+ WAF, CloudFront, Route53, ElastiCache, SQS, Lambda, CloudWatch 等）

| 指標 | 値 |
|------|-----|
| コンポーネント数 (n) | ~19 |
| 接続線数 (m) | 29 |
| 辺密度 (m/n) | 1.5 |
| 平面上限 (3n-6) | 51 |

- **予想が外れ始めた段階**
- マルチAZの左右対称配置がきれいに分かれている
- Internet → CloudFront → ALB → EC2 → RDS の階層構造が自然
- VPC/AZ/サブネットのグルーピングが適切
- CloudWatchへの監視線の交差やSQS/Lambdaの配置に若干の課題はあるが「叩き台としてそのまま使える」レベル

#### Step 3: 複雑な構成（+ Cognito, API Gateway, ECS Fargate, DynamoDB, SNS, EFS, S3 Logs）

| 指標 | 値 |
|------|-----|
| コンポーネント数 (n) | ~27 |
| 接続線数 (m) | 48+ |
| 辺密度 (m/n) | 1.8 |
| 平面上限 (3n-6) | 75 |

- **予想はさらに覆された**
- 監視系の破線が実線と視覚的に区別されている（破線＋グレー）
- ALBからEC2×2、ECS Fargate×2への4方向接続がAZ別にきれいに分岐
- VPC外サービスが図の上部に2段でまとまっている
- Cross-AZ接続や並走する接続線など一部複雑さはあるが、「かなり良い」レベル

### 崩れた3つの前提

1. **「LLMは空間的な推論が苦手」→ Opus 4.6では当てはまらない**: XMLの座標値を「数字として」適切に配置でき、各コンポーネントの空間的な位置関係を理解して生成している
2. **「複雑さが増せばヒューリスティクスは再現できない」→ パターン学習が想像以上に効いている**: トレーニングデータに含まれるAWS構成図のパターンにより、階層構造・対称性・グルーピングをかなりの精度で再現
3. **「一発で複雑な図は作れない」→ 段階的な構築で精度を維持できる**: Step 1→2→3と段階的にプロンプトを出すことで、前のステップの構造を踏まえた高品質な図を生成

### 遭遇した落とし穴：アイコンの描画問題

- AIが生成しがちな `shape=mxgraph.aws4.ec2_instance` + `fillColor=#ED7100` の組み合わせでは、fillColorがアイコン全体を塗りつぶし、ただのオレンジ色の矩形になる
- 正しいパターンは `shape=mxgraph.aws4.resourceIcon;resIcon=mxgraph.aws4.ec2` 形式
- **生成後にアイコンの表示確認が必要**

### 品質向上のためのテクニック

1. **図を分割する**: 関心事ごと（リクエストフロー/非同期処理/管理系）やレイヤーごとに分割
2. **プロンプトやSkillsでルールを整備する**: 配置ルール（座標揃え、トラフィック方向、グリッドサイズ等）を明示
3. **最小限のXMLスケルトンを先に渡す**: VPC/サブネットの外枠をあらかじめ用意してその中にコンポーネントを配置させる
4. **draw.ioの自動レイアウト機能と組み合わせる**: 特にSugiyama法に基づく階層的レイアウトはネットワーク構成図と相性が良い

## 重要な事実・データ

- **平面グラフの辺数上限**: m ≤ 3n - 6（オイラーの公式に基づく）
- **交差補題**: 辺密度が高いグラフでは交差数が e³/n² に比例する下界を持つ
- **交差数最小化問題**: NP困難（Garey & Johnson, 1983）
- **GenAI-DrawIO-Creator研究**: draw.io図の自動生成で初回セマンティック精度94%を達成するも、コンポーネント数増加でレイアウト品質が低下する傾向あり（Yu et al., 2026）
- **Step 3の実績**: 27コンポーネント・48+接続線の図でも構造的な整理が保たれた

## 結論・示唆

### 著者の結論

- 「LLMには複雑な図は作れない」という前提はOpus 4.6で**完全に崩れた**
- 当初は「なぜぐちゃぐちゃになるのか」を解説する記事にするつもりだったが、前提そのものが覆された
- LLMの進化は「ここまでは無理だろう」と思っている境界線を静かに超えてきている

### 実践的な示唆

- 複雑なAWS構成図もClaude Code（Opus 4.6）で「そのまま使えるレベル」に近い品質で生成可能
- 段階的なプロンプト設計（シンプル→複雑へ）で精度を維持できる
- アイコンのスタイル定義（`resourceIcon`パターン）には注意が必要
- 図の分割、ルール整備、XMLスケルトン、自動レイアウト機能の併用でさらに品質向上が可能

## 制限事項・注意点

- アイコンが正しく描画されない問題（`fillColor`によるベタ塗り）が発生する可能性あり
- 段階的プロンプトによる品質向上の可能性があるため、一発で複雑な図を生成した場合の品質は未検証
- レイアウトの品質評価は著者の主観に基づいている
- 検証はAWSアーキテクチャの構成図に限定されており、他の種類の図への汎用性は未確認

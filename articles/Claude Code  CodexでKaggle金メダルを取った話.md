## TL;DR

- この構図はKaggleに限らず、**定量評価できるR&D全般**で起きうる
- AIに素早く実験を回させるための**環境・構成づくり**も重要だった
- スコアを押し上げたアイディアの**ほとんどは人間発**。AIの提案の打率は低かった
- 実装から解放された分、人間は**アイディアとデータ観察**に集中できた
- 実装と分析をAIに任せることで、**実験量が桁違いに増えた**（5-Fold CV合計1,515回）

## はじめに

- **Kaggle草コンペ**（[CSIRO - Image2Biomass Prediction](https://www.kaggle.com/competitions/csiro-biomass)）で**5位 / 3,803チーム（Top 0.1%）、金メダル**を獲得
- 自分が書いたコードは**ほぼゼロ**。実装のほぼすべてを**Claude Code / Codex**に任せた
- 消えたのは「実装に使う時間」であり、重要になったのは**仮説を立てること、データを観察すること、何を優先するかの判断**
- Coding Agentがあれば誰でも勝てるわけではない理由:
  1. 他の参加者もAIコーディングツールを使っている（道具が同じなら差はつかない）
  2. Coding Agentは**自律的にスコアを改善し続けることに限界がある**（序盤は強いが終盤は頭打ち）
- 使用モデル: **Claude Code（Opus 4.5）** と **Codex（GPT-5.2 xhigh）**（2026年1月時点の最新）。両者に大きな差は感じなかった

## Kaggleで勝つために必要なこと

### 主要用語

| 用語 | 説明 |
|---|---|
| OOF (Out-of-Fold) | CVで各foldの検証データに対して出した予測 |
| LB (Leaderboard) | Kaggleに提出したときのスコア |
| CV (Cross Validation) | ローカルでの検証スコア |

### Kaggle競技の特性

- 3,804チームが参加、金メダル獲得には**17位以内（上位0.5%）**が必要
- 定石がなく、コンペごとに何が効くかは変わる → **「Do Everything」**（思いつく施策を片っ端から試せ）
- Coding Agentの登場で**実装コストの壁がほぼなくなった**
- 代わりに問われるようになったのは:
  - **何を試すかを決めるアイディアの力**
  - **Coding Agentが素早く実験を回せる環境を整える力**

## コーディングAgentの現在地

### 得意なこと — 実装と分析

- 実験結果の可視化、誤差の傾向分析、大量のNotebookの要約
- 実装の正確さが高く、コードの品質は見ずに精度（CV/LB）だけで判断する運用でも問題なし
- DINOv3のpatch tokenを活用したカスタムヘッドなど、モデルアーキテクチャを深く理解した実装も指示を出せばこなす
- CV・前処理・学習ループなどを必要な単位を理解して書いてくれる

### 苦手なこと — 自律的な精度改善

- 自律的な改善サイクルは**数回で頭打ち**
- 公開Notebookから「改善して」→ いろんなパターンを提案するが、スコア改善には至らない
- 「精度を上げろ」のようなアバウトな指示には、**アンサンブル、TTA、正規化の追加など教科書的な施策ばかり提案**
- コンペのお題に沿った**本質的な改善提案**がほしいのに出てこない

> **核心**: 実装と分析はAIに、アイディアと判断は人間に。AIに分析をさせて、その結果を見て人間が次のアイディアをひねり出す。

> にとりさんの[トレード領域でのClaude Code活用記事](https://note.com/neet_uncle/n/nb793a6962ca7)でも、「LLMが考えてくる戦略は基本に忠実だけど、なんかセンスを感じない」と**ほぼ同じ壁**にぶつかっている。これが現時点でのAIの構造的な限界。

## 実験を高速に回すための構成

5-Fold CVで合計**1,515回**の学習（ユニーク実験数は約300）を実現した構成。

### EXP + child-exp の2段構成

```
EXP/
├── EXP001/
│   ├── train.py           # メインの学習パイプライン
│   ├── config/
│   │   ├── child-exp000.yaml   # ベース設定
│   │   ├── child-exp001.yaml   # loss変更
│   │   ├── child-exp002.yaml   # model変更
│   │   └── ...
│   └── output/
│       ├── child-exp000/
│       │   ├── oof.csv
│       │   └── result.json
│       ├── child-exp001/
│       └── ...
├── EXP002/                # パイプラインを大きく変えるとき
└── ...
```

- **パラメータ、loss、モデルの変更程度** → 同じEXP内のchild-expで回す
- **`train.py`のパイプラインを大きく変えるとき** → 新しいEXPを作る
- エージェントへの指示が簡潔になる:「EXP001のchild-exp003として、lossをXXに変更して」

### CLAUDE.md / AGENTS.md — AIへのプロジェクト指示書

最初から設計したわけではなく、**AIが同じミスを繰り返すたびに「二度とやるな」と書き足していった結果**、場当たり的に育ったもの。

- **AGENTS.md（270行）**: Codex用。「これをやるとLBが下がる」というガードレールリスト重点
- **CLAUDE.md（430行）**: 評価関数の正しい実装/間違った実装の両方を明記、infer.pyの変数名バグが3回出たので「必ずチェックしろ」と記載、後方互換性のルールを良い例・悪い例つきで記載

> **指示書があるとないとでは、AIの出力の安定感がまるで違う。**

### EXP_SUMMARY.md — 実験履歴（AIの記憶）

```
1. コーディングエージェントに実験コードを書かせる
2. Colabで実行 → oof.csv, train.log, result.json が出力
3. KaggleにSubmitしてLBスコアを確認
4. EXP_SUMMARY.md を更新（実験名、変更内容、CV、LB、成功/失敗、所感を追記）
```

- 次に実験方針を相談するとき、このファイルを読ませることで**同じ失敗を繰り返させない**
- **注意点**: 失敗した実験を大量に読ませると、AIが**非常に消極的**になる。「もう打ち手がないです」と諦めてくる
- EXP_SUMMARY.mdは「ガードレール」として使い、**次のアイディア自体は人間が出す**のが重要

#### 小ネタ：AIにも性格がある

- **Claude**: 慎重すぎて撤退提案。「正直に言います。0.73を超えるのは極めて困難」→ 選択肢Aに「撤退（現実的）」を提案。「このコンペは学びとして区切りをつけますか？」、「exp058を信じてPrivateに賭ける」（根拠のない楽観案）
- **Gemini 3.0 Pro**: 真逆。相関分析の結果に「まさに"宝の山"を見つけた状態です」と興奮。「大・勝利の予感ですね！プロボクサーが1ラウンド目でダウンを奪ったくらいの勢い」（→ 実際はその特徴量は何の役にも立たなかった）
- **結論**: どちらも**人間が判断を握っていないと危ない**

### アイディアから実験への流れ

**方法1: プロンプトに直接投げる**（具体的なアイディアがあるとき）

> EXP060のchild-exp005をベース実験として、「2000x1000の横長の画像から現在は半分半分の1000x1000を2枚だけ学習に使っているが、1000x1000のx座標を変更して、ランダムに5枚切り出して学習に使えるようにしてください。」

**方法2: mdファイルにまとめてから渡す**（情報量が多いとき）

- Deep Researchの結果や気になったモデル情報を `docs/Idea_Research/` にmdファイルとして書き溜め
- 「このファイルを読んで、まだ試していないものを実装して」と渡す
- エージェントがEXP_SUMMARY.mdと照らし合わせて未実施施策をピックアップ

> **アイディアの粒度は人間が決め、実装への落とし込みはコーディングエージェントに任せる。**

### この記録から読み取れること

- **精度が上がった施策のほとんどは人間のアイディア**。AI発のヒットはGradual UnfreezeとConsistency Loss（資料を読ませた結果）の2つだけ
- **精度が上がらなかった施策の大半はAIの提案**。ただしKaggleでは10回実験して1回当たれば良い方。外れること自体は問題ではない
- AIがいなければ失敗した施策の多くはそもそも試す時間がなかった → **「とりあえず試す」ができるようになったことがコーディングエージェントの本当の価値**

> **最も重要な勝因**: DINOv3のpatch tokenの活用は「精度上げろ」のようなアバウトな質問からは絶対に出てこない。「画像に写っている草の重さを推定するなら、緑の面積・高さ・密度の3つの情報が大事なはず」という仮説をずっと考えていて、DINOv3のpatch tokenの出力を目視で観察したときに「これだ」とつながった。**コーディングエージェントが実装を引き受けてくれたことで、モデルの出力を注意深く観察する余裕が生まれた。**

## AIに任せて効いた使い方

共通する線引き: **「分析までは積極的に任せていい。その先のアイディア出しは人間がやる」**

### OOFの分析

- どのアイテムの予測が外れているかをメタデータと照らし合わせて分析させると、**非常に質の高いレポート**が出てくる
- 切り口を変えた分析を何パターンも回せる（CV_LB相関、状態×季節ごとの誤差分析 etc.）
- **ただし**: 分析結果をもとにした**Next Actionの提案は全外し**。Trainの分布を歪めるか、推論時に無理やり補正する方向ばかり

### CV-LBの相関分析

- CVスコアの信頼性が揺らいでいないか、特定の実験群でCV-LBの乖離が起きていないかを定期的にチェックさせた
- ただし**CVの切り方自体を工夫する必要があること**にはAIは自律的にたどり着けなかった

### High Score Notebookの分析

- Discussionや上位のpublic notebookを全部読ませて、要約・比較させる → **非常に有効**

## コーディングエージェントが得意なコンペの前提条件

今回の草コンペはAIコーディングとの**相性が良い条件**が揃っていた:

- タスクがシンプル（画像回帰） → パイプラインの構造が単純
- データ数が少なく、1実験の学習時間が短い → 試行回数を多く回せた
- 同時期開催の別コンペ（PhysioNet）はパイプラインが複雑で、Agentの手助けを得にくかった

## おわりに — 重要な結論

> コーディングエージェントは「Do Everything」のハードルを劇的に下げた。「言うは易く行うは難し」だったものが、**「言う方を人がやれば、行う方はAIがやる」**に変わった。

> ただし、**何を試すかを決める力は今まで以上に問われる**。実装の差がなくなった分、差がつくのは**アイディアとデータ理解の深さ**。

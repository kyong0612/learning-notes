## 概要

BigQuery データセット レプリケーションを使用すると、**2つの異なるリージョンまたはマルチリージョン間**でデータセットの自動レプリケーションを設定できる。

### デフォルトの冗長性

- BigQuery はデータセットのロケーション内の **2つの異なるゾーン**にデータのコピーを常に保存（同期二重書き込み）
- マルチリージョン ロケーションを選択しても、**複数リージョン間のレプリケーションやリージョン冗長性は提供されない**（データは地理的位置内の単一リージョンに保存）

### クロスリージョン レプリケーション

任意のデータセットを複製すると、指定された別のリージョンに**セカンダリ レプリカ**が作成される。

- **プライマリ レプリカ**: 書き込み可能（初回作成時のリージョン）
- **セカンダリ レプリカ**: 読み取り専用（追加されたレプリカ）
- プライマリへの書き込みはセカンダリに**非同期で複製**
- 各リージョン内で2ゾーン冗長 → 合計 **4つのゾーンコピー**
- ネットワーク トラフィックは Google Cloud ネットワーク外部に出ない

## 料金

| 項目 | 詳細 |
|------|------|
| データ レプリケーション | [データ レプリケーションの料金](https://cloud.google.com/bigquery/pricing?hl=ja#data_replication)に従い別途課金 |
| ストレージ | セカンダリ リージョンのストレージは個別コピーとして課金。長期保存テーブル/パーティションはセカンダリでアクティブ ストレージにリセットされない |

- データ レプリケーションは BigQuery が管理し、**スロットリソースは使用されない**

### セカンダリ リージョンのコンピューティング容量

- セカンダリでジョブ/クエリを実行するには、セカンダリ リージョン内の**スロット購入**または**オンデマンド クエリ**が必要
- プライマリと同数のスロットでなくても可（ただしスロット数削減はパフォーマンスに影響）

## ロケーションに関する留意事項

- レプリカのロケーションは初期データセットのロケーションと**別にする**必要がある
- マルチリージョンの地域内では複製不可
- ビュー、マテリアライズド ビュー、BigLake以外の外部テーブルなど**コロケーション要件**があるリソースは、ソースと異なる/互換性のないロケーションへのレプリカ追加でジョブエラーが発生する可能性あり

### Cloud Storage のコロケーション

Cloud Storage 上のデータをクエリするには、Cloud Storage バケットとレプリカを**同じ場所**に配置する必要がある。

## 制限事項

### 障害復旧で構成不可のリージョンペア

| プライマリ | セカンダリ |
|-----------|-----------|
| `eu-west4` | `eu` マルチリージョン |
| `eu-west1` | `eu` マルチリージョン |
| `us-west1` | `us` マルチリージョン |
| `us-central1` | `us` マルチリージョン |

### 検索インデックスの制約

- 8時間以内に切り替えると、インデックス生成が**8時間遅延**
- レプリカ切り替え時、インデックスは以前のプライマリから**削除**され、昇格リージョンで**再生成**
- メタデータのみがセカンダリに複製（インデックス データ自体は複製されない）

### リソースの動作

セカンダリ レプリカでは以下の操作が**サポートされていない**:
- テーブル スナップショットの作成
- テーブルのクローンの作成

#### プライマリ vs セカンダリ リソース比較

| リソース | セカンダリでの動作 |
|---------|------------------|
| **マネージド テーブル** | 違いなし |
| **行アクセス ポリシー** | 違いなし |
| **モデル** | マネージド テーブルとして保存 |
| **BigLake テーブル** | サポートされていない |
| **外部テーブル** | 定義のみ複製。バケットが同ロケーションにない場合クエリ失敗 |
| **論理ビュー** | 同ロケーションにないリソース参照時はクエリ失敗 |
| **マテリアライズド ビュー** | 参照テーブルが同リージョンにない場合クエリ失敗。最大ステイルネスを超える可能性あり |
| **リモート関数** | 接続はリージョン単位。異なるロケーションの接続参照時は実行失敗 |
| **ルーティン / UDF / ストアド プロシージャ / TVF** | 異なるロケーションのリソース参照時は実行失敗 |
| **検索インデックス** | メタデータのみ複製 |
| **テーブル クローン** | セカンダリでディープコピーとして課金 |
| **テーブル スナップショット** | セカンダリでディープコピーとして課金 |
| **列に対するデータポリシー** | 異なるロケーションの UDF を参照するカスタムポリシーはクエリ失敗 |

## 停止のシナリオ

> **重要**: クロスリージョン レプリケーションは、**リージョン全体が停止した場合の障害復旧計画として使用するものではない**。プライマリ リージョンの全体停止時にセカンダリをプロモートすることはできない。

| プライマリ | セカンダリ | 停止リージョン | 影響 |
|-----------|-----------|--------------|------|
| リージョン1 | リージョン2 | リージョン2 | セカンダリの読み取り専用ジョブが失敗 |
| リージョン1 | リージョン2 | リージョン1 | リージョン1のジョブ全失敗。リージョン2で読み取り専用ジョブは継続可能（ただし同期完了までデータは古くなる） |

障害復旧の準備については[マネージド障害復旧](https://docs.cloud.google.com/bigquery/docs/managed-disaster-recovery?hl=ja)を参照。

## データセット レプリケーションの操作

### 必要な権限

`bigquery.datasets.update` 権限が必要。

### データセットを複製する

[ALTER SCHEMA ADD REPLICA DDL ステートメント](https://docs.cloud.google.com/bigquery/docs/reference/standard-sql/data-definition-language?hl=ja#alter_schema_add_replica_statement)を使用する。

```sql
-- プライマリ レプリカを us-central1 に作成
CREATE SCHEMA my_dataset OPTIONS(location='us-central1');

-- セカンダリ リージョンにレプリカを追加
ALTER SCHEMA my_dataset
ADD REPLICA `my_replica`
OPTIONS(location='us-east4');
```

- レプリケーション中もプライマリへのクエリ実行可能（処理能力は低下しない）
- 作成確認: [INFORMATION_SCHEMA.SCHEMATA_REPLICAS](https://docs.cloud.google.com/bigquery/docs/information-schema-schemata-replicas?hl=ja) の `creation_complete` 列
- セカンダリでクエリするにはロケーションを**明示的に設定**する必要あり

### セカンダリ レプリカを昇格させる

プライマリ リージョンがオンラインの場合のみ昇格可能。セカンダリが追いつくと**数秒以内**に完了。

```sql
ALTER SCHEMA my_dataset SET OPTIONS(primary_replica = 'us-east4')
```

**注意点**:
- 昇格開始時に完全に複製されていないテーブルは**ステイル読み取り**を返す
- 昇格進行中はテーブルへの**すべての書き込みでエラー**が返される
- ジョブのロケーションをセカンダリ リージョンに明示的に設定する必要あり
- 昇格確認: `INFORMATION_SCHEMA.SCHEMATA_REPLICAS` の `replica_primary_assignment_complete` 列

### データセットのレプリカを削除する

```sql
ALTER SCHEMA my_dataset
DROP REPLICA IF EXISTS `us`;
```

- データセット全体の削除前に、**すべてのセカンダリ レプリカを先に削除**する必要がある

### データセットのレプリカを一覧表示する

[INFORMATION_SCHEMA.SCHEMATA_REPLICAS](https://docs.cloud.google.com/bigquery/docs/information-schema-schemata-replicas?hl=ja) ビューにクエリする。

## データセットのリージョン間移行

クロスリージョン レプリケーションを使用したリージョン間移行の手順（例: `US` → `EU`）:

### 1. レプリカ追加

```sql
ALTER SCHEMA my_migration
ADD REPLICA `eu`
OPTIONS(location='eu');
```

### 2. セカンダリ昇格

```sql
ALTER SCHEMA my_migration SET OPTIONS(primary_replica = 'eu')
```

### 3. 旧レプリカ削除（任意）

```sql
ALTER SCHEMA my_migration
DROP REPLICA IF EXISTS us;
```

## 顧客管理の暗号鍵（CMEK）

セカンダリ レプリカ作成時に CMEK は**自動的に複製されない**。レプリカのロケーションに `replica_kms_key` を設定する必要がある。

### CMEK の動作ルール

| 条件 | 動作 |
|------|------|
| ソースに `default_kms_key` あり | レプリカに `replica_kms_key` を指定**必須** |
| ソースに `default_kms_key` なし | `replica_kms_key` は設定**不可** |
| ソースに `default_kms_key` なし + 個別テーブルに CMEK あり | レプリカでテーブルのクエリ**不可**（`replica_kms_key` 設定か `default_kms_key` 追加で対応） |

### 鍵のローテーション

- ローテーション後も複製データセットはクエリ可能
- ローテーション前の鍵バージョンが削除されると、該当テーブルのクエリ不可（鍵バージョン更新が必要）
- セカンダリでのローテーションは全テーブルの鍵バージョンを更新
- プライマリでのローテーションはローテーション後に作成されたテーブルのみ更新

### CMEK を使用したレプリカ作成例

```sql
ALTER SCHEMA my_dataset
ADD REPLICA `us-west1`
OPTIONS(location='us-west1',
  replica_kms_key='my_us_west1_kms_key_name');
```

### CMEK の制限事項

- レプリカ作成後に複製された Cloud KMS 鍵の**更新不可**
- レプリカ作成後にソースの `default_kms_key` 値の**更新不可**
- 指定した `replica_kms_key` が宛先リージョンで無効な場合、データセットは複製されない

## 列に割り当てられたデータポリシー

- データポリシーはリージョン リソース（テーブルと同一リージョンに存在する必要あり）
- セカンダリ レプリカ作成時に**自動的に複製**される
- セカンダリでは**読み取り専用**（更新不可）
- プライマリで全テーブル列からポリシー適用解除すると、セカンダリでポリシーが孤立 → **手動クリーンアップ必要**
- レプリカ削除時もデータポリシーは**自動削除されない** → 手動削除が必要
- 名前の競合: セカンダリに同一IDのデータポリシーが既存の場合、レプリケーション失敗

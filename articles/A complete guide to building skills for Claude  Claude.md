## 概要

スキルとは、特定のタスクやワークフローの処理方法をClaudeに教えるための**フォルダとして パッケージ化された一連の指示**である。毎回の会話で好みやプロセス、専門知識を説明し直す代わりに、一度教えて継続的に活用できる。

**このガイドで学べること：**
- スキル構造の技術要件とベストプラクティス
- スタンドアロンスキルとMCP連携ワークフローのパターン
- テスト、反復、配布の方法

**対象者：**
- 一貫したワークフローをClaudeに実行させたい開発者
- MCP連携にワークフロー知識を追加したいコネクタビルダー
- 繰り返しタスクを自動化したいパワーユーザー
- 組織全体でClaudeの動作を標準化したいチーム

**所要時間：** skill-creatorを使用して最初の動作するスキルを構築・テストするのに約15-30分

## Chapter 1: 基本原則（Fundamentals）

### スキルの構成要素

```
your-skill-name/
├── SKILL.md          # 必須 - メインスキルファイル
├── scripts/          # オプション - 実行可能コード（Python, Bashなど）
├── references/       # オプション - 必要に応じてロードされるドキュメント
└── assets/           # オプション - テンプレート、フォントなど
```

### 3つの設計原則

#### 1. Progressive Disclosure（段階的開示）

| レベル | 内容 | ロードタイミング |
|--------|------|------------------|
| 第1レベル | YAMLフロントマター | 常にシステムプロンプトにロード |
| 第2レベル | SKILL.md本文 | スキルが関連すると判断された時 |
| 第3レベル | リンクされたファイル | 必要に応じて探索 |

#### 2. Composability（構成可能性）
複数のスキルを同時にロード可能。他のスキルと協調動作するよう設計する。

#### 3. Portability（移植性）
Claude.ai、Claude Code、APIで同一動作。一度作成すれば全プラットフォームで動作。

### MCP + Skills の関係

| MCP（接続性） | Skills（知識） |
|--------------|---------------|
| サービスへの接続を提供 | サービスの効果的な使用方法を教える |
| リアルタイムデータアクセスとツール呼び出し | ワークフローとベストプラクティスを捕捉 |
| **Claudeが何ができるか** | **Claudeがどうすべきか** |

**キッチンのアナロジー：**
- MCP = プロのキッチン（ツール、食材、機器へのアクセス）
- Skills = レシピ（価値あるものを作るためのステップバイステップ指示）

## Chapter 2: 計画とデザイン（Planning and Design）

### ユースケースから始める

コードを書く前に、スキルが実現すべき2-3の具体的なユースケースを特定する。

**良いユースケース定義の例：**
```
Use Case: プロジェクトスプリント計画
Trigger: ユーザーが「スプリントの計画を手伝って」と言う
Steps:
1. Linear（MCP経由）から現在のプロジェクト状況を取得
2. チームのベロシティと容量を分析
3. タスクの優先順位付けを提案
4. 適切なラベルと見積もりでLinearにタスクを作成
Result: タスクが作成された完全に計画されたスプリント
```

### 3つのユースケースカテゴリ

#### カテゴリ1: ドキュメント＆アセット作成
一貫した高品質の出力（ドキュメント、プレゼン、アプリ、デザイン、コード）を作成。
- 埋め込みスタイルガイドとブランド標準
- 一貫した出力のためのテンプレート構造
- 外部ツール不要

#### カテゴリ2: ワークフロー自動化
一貫した方法論の恩恵を受けるマルチステッププロセス。
- バリデーションゲート付きステップバイステップワークフロー
- 反復的な改善ループ

#### カテゴリ3: MCP強化
MCPサーバーが提供するツールアクセスを強化するワークフローガイダンス。
- 複数のMCP呼び出しを順番に調整
- ドメイン専門知識を埋め込む

### 技術要件

#### 重要なルール

| 項目 | 要件 |
|------|------|
| SKILL.md命名 | 正確に`SKILL.md`（大文字小文字区別） |
| フォルダ命名 | kebab-case使用（`notion-project-setup` ✅） |
| README.md | スキルフォルダ内に含めない |

#### YAMLフロントマター

**最小必須形式：**
```yaml
---
name: your-skill-name
description: 何をするか。ユーザーが[特定のフレーズ]を要求したときに使用。
---
```

**フィールド要件：**
- `name`（必須）: kebab-caseのみ、スペースや大文字なし
- `description`（必須）: 何をするか + いつ使うか（1024文字以下）
- `license`（オプション）: MIT, Apache-2.0など
- `compatibility`（オプション）: 環境要件
- `metadata`（オプション）: author, version, mcp-serverなど

**セキュリティ制限：**
- XMLタグ（`<` `>`）禁止
- "claude"や"anthropic"を含む名前は予約済み

### 効果的なdescriptionの書き方

**構造：** `[何をするか] + [いつ使うか] + [主な機能]`

**良い例：**
```yaml
description: FigmaデザインファイルをAnalysisし、開発者ハンドオフドキュメントを生成。
ユーザーが.figファイルをアップロード、「デザインスペック」「コンポーネントドキュメント」
「デザインからコードへのハンドオフ」を要求したときに使用。
```

**悪い例：**
```yaml
description: プロジェクトを支援する。  # 曖昧すぎる
description: 洗練されたマルチページドキュメントシステムを作成。  # トリガー欠如
```

## Chapter 3: テストと反復（Testing and Iteration）

### テストレベル

| レベル | 用途 |
|--------|------|
| 手動テスト（Claude.ai） | クエリを直接実行して動作観察。高速反復。 |
| スクリプトテスト（Claude Code） | 変更に対する再現可能な検証の自動化 |
| プログラムテスト（Skills API） | 定義されたテストセットに対する体系的評価 |

### 推奨テストアプローチ

#### 1. トリガーテスト
```
トリガーすべき:
- "新しいProjectHubワークスペースをセットアップして"
- "ProjectHubにプロジェクトを作成する必要がある"

トリガーすべきでない:
- "サンフランシスコの天気は？"
- "Pythonコードを書いて"
```

#### 2. 機能テスト
- 有効な出力が生成される
- API呼び出しが成功
- エラー処理が機能
- エッジケースがカバー

#### 3. パフォーマンス比較

| 指標 | スキルなし | スキルあり |
|------|-----------|-----------|
| やり取り回数 | 15メッセージ | 2つの確認質問のみ |
| 失敗API呼び出し | 3回（リトライ必要） | 0回 |
| 消費トークン | 12,000 | 6,000 |

### skill-creatorスキルの活用

skill-creatorはスキルの構築と反復を支援：
- 自然言語記述からスキルを生成
- 適切にフォーマットされたSKILL.mdを生成
- 問題のフラグ付けとテストケースの提案
- 反復的な改善

## Chapter 4: 配布と共有（Distribution and Sharing）

### 現在の配布モデル（2026年1月）

**個人ユーザーの場合：**
1. スキルフォルダをダウンロード
2. フォルダをzip化
3. Claude.aiでSettings > Capabilities > Skillsからアップロード
4. またはClaude Codeのスキルディレクトリに配置

**組織レベル：**
- 管理者がワークスペース全体にスキルを展開可能（2025年12月18日出荷）
- 自動更新と集中管理

### オープンスタンダード

Agent SkillsはMCPと同様にオープンスタンダードとして公開。同じスキルがClaude以外のAIプラットフォームでも動作すべき。

### API経由でのスキル使用

- `/v1/skills`エンドポイントでスキル管理
- Messages APIリクエストに`container.skills`パラメータでスキル追加
- Claude Agent SDKとの連携

## Chapter 5: パターンとトラブルシューティング

### 5つの主要パターン

#### パターン1: シーケンシャルワークフロー調整
特定の順序でマルチステッププロセスが必要な場合に使用。
- 明示的なステップ順序
- ステップ間の依存関係
- 各段階でのバリデーション

#### パターン2: マルチMCP調整
ワークフローが複数のサービスにまたがる場合に使用。
- 明確なフェーズ分離
- MCP間のデータ受け渡し
- 次フェーズ前のバリデーション

#### パターン3: 反復的改善
出力品質が反復で向上する場合に使用。
- 明示的な品質基準
- 反復的な改善
- バリデーションスクリプト

#### パターン4: コンテキスト認識ツール選択
同じ結果でもコンテキストによって異なるツールを使用。
- 明確な決定基準
- フォールバックオプション
- 選択の透明性

#### パターン5: ドメイン固有インテリジェンス
スキルがツールアクセス以上の専門知識を追加する場合に使用。
- ロジックに埋め込まれたドメイン専門知識
- アクション前のコンプライアンス
- 包括的なドキュメント

### トラブルシューティング

| 問題 | 原因 | 解決策 |
|------|------|--------|
| スキルがアップロードできない | ファイル名が正確に`SKILL.md`でない | 大文字小文字を確認して修正 |
| スキルがトリガーしない | description が曖昧すぎる | トリガーフレーズを追加 |
| スキルが頻繁にトリガーしすぎる | 範囲が広すぎる | 否定トリガーを追加、より具体的に |
| MCP接続問題 | 認証/接続の問題 | MCP単独でテスト、認証確認 |
| 指示に従わない | 指示が冗長/曖昧 | 簡潔に、重要な指示を上部に |

## 概要

カミナシのエンジニアShimmyが、0→1の新規プロダクト開発で実践した設計原則とガードレールについて解説した記事。AIとの協働で力を発揮するアーキテクチャを追求した結果、たどり着いた設計原則は**AI以前から良いとされてきたもの**であった。記事は「関心の分離」「価値の高いテスト」「依存方向の決定」の3つの設計原則と、それらを静的解析ツールとGit hooksで決定論的に守らせる仕組みを具体的なコード例とともに紹介している。

技術スタック: TanStack Start（フルスタックReactフレームワーク）を使用し、フロントエンドとバックエンドが同一コードベース。

## 主要なトピック

### 1. AIの力を引き出す設計の3つの条件

著者が掲げる設計原則は以下の3つ：

| # | 原則 | AI協働でのメリット |
|---|------|-------------------|
| 1 | **関心の分離** | AIのコンテキストに載せやすく、並列開発でもコンフリクトしにくい |
| 2 | **価値の高いテスト** | モック不要でリファクタリングに強い出力値ベーステスト |
| 3 | **依存方向の決定** | AIは迷わず、静的解析で機械的に強制できる |

> これらはAIのために特別に取り入れた考えではない。今まで良い設計とされていたものが、結果的にAIとの協働でさらに力を発揮するようになった。

### 2. 関心の分離 — Feature-First構成

#### AIとの相性

- 関連ファイルが1ディレクトリに集約 → 「このディレクトリを読んで修正して」で完結
- ファイルが散らばると、AIがコンテキストウィンドウを無駄に消費する
- git worktreeで複数AIエージェントを並列稼働させても、関心事が分かれていればコンフリクトしにくい

#### Feature-Firstのディレクトリ構成

```
features/
├── 関心事A/
│   ├── domain/           # Functional Core: 純粋関数のみ
│   ├── infrastructure/   # Imperative Shell: DBアクセスなどI/O
│   ├── server/           # API層: domainとinfrastructureを組み立てるエンドポイント
│   ├── components/       # 複数ページで再利用するUI
│   ├── hooks/            # カスタムフック
│   └── index.ts          # Public API
├── 関心事B/
│   ├── domain/
│   ├── infrastructure/
│   ├── server/
│   └── index.ts
└── ...
```

#### 結合と距離のバランス

- **Feature内部 → 高凝集**: 同じ関心事のコードが1ディレクトリにまとまる。結合強度は高いが距離が短いので問題ない
- **Feature間 → 疎結合**: 各featureは`index.ts`を通じてPublic APIのみ公開。内部構造は隠蔽

> 参考: 『ソフトウェア設計の結合バランス』(Vlad Khononov)

#### Public API境界

各featureの`index.ts`は外部に公開するものだけをexport：

```typescript
// features/xxx/index.ts
export { calculateSomething, type Quantity } from "./domain/calculations";
export { useSaveRecord } from "./hooks/use-save-record";
export { recordsQueryOptions } from "./queries";
// domain/の内部ヘルパー、infrastructure/のDB操作詳細 などは非公開
```

#### featureを横断するケース

| パターン | 用途 | 構造 |
|----------|------|------|
| **shared/** | 共通ビジネスロジック | 純粋関数を`shared/lib/`に配置。依存方向は常に `features/` → `shared/` |
| **routes/** | 1ページで複数feature利用 | 各featureのPublic APIを組み合わせてページを構築 |

routes層のページコンポーネントが各featureからデータを取得し組み合わせてUIを描画する：

```typescript
// routes/xxx/$id/index.tsx
const [recordsA, recordsB, recordsC] = await Promise.all([
  fetchFeatureA(id, date),
  fetchFeatureB(id, date),
  fetchFeatureC(id),
]);
const rows = buildRowData(recordsA, recordsB, recordsC);
return <DataGrid rows={rows} ... />;
```

### 3. 価値の高いテスト — 出力値ベーステスト + FCIS

#### 価値の高い単体テストの4つの柱

『単体テストの考え方/使い方』(Vladimir Khorikov) より：

1. 退行（リグレッション）に対する保護
2. **リファクタリングへの耐性**（二値的 → まず確保すべき）
3. 迅速なフィードバック
4. 保守しやすさ

リファクタリング耐性がないテスト（偽陽性が多い）はテストへの信頼を損ない、やがて無視されるようになる。

#### 出力値ベーステスト

関数の内部実装ではなく**振る舞い（Input→Output）**を検証：

```typescript
// domain層の純粋関数
const progressRate = (actualTotal: number, targetQuantity: number): ProgressRate | null => {
  if (targetQuantity === 0) return parseProgressRate(0);
  const ratio = actualTotal / targetQuantity;
  const percentage = roundToOneDecimal(ratio * 100);
  return parseProgressRate(percentage);
};
```

```typescript
describe("progressRate", () => {
  it("実績と目標から進捗率を返す", () => {
    expect(progressRate(75, 100)).toBe(75.0);
  });
  it("100%を超える場合も正しく計算する", () => {
    expect(progressRate(120, 100)).toBe(120.0);
  });
  it("目標が0のとき0を返す", () => {
    expect(progressRate(50, 0)).toBe(0);
  });
});
```

DB初期化・クリーンアップ不要、モック設定・DI構築も不要。

#### Functional Core, Imperative Shell（FCIS）

出力値ベーステストを自然に書くには、テスト対象が純粋関数である必要がある。そこで FCIS パターンを採用：

| 層 | 役割 | 依存ルール |
|----|------|-----------|
| **domain/** | Functional Core。純粋関数のみ | 外部I/Oに依存しない |
| **infrastructure/** | Imperative Shell。DBアクセスなどI/O | domainに依存してよい |
| **server/** | API層。エンドポイント提供 | domainとinfrastructureの両方に依存してよい |

> 参考: 「Boundaries」(Gary Bernhardt)、『Domain Modeling Made Functional』(Scott Wlaschin)

### 4. 設計を決定論的に守らせるガードレール

#### なぜ静的解析か

- CLAUDE.md/AGENT.mdのルール → AIはある程度従うが**確率的**（100%守るとは限らない）
- レビューで指摘 → 気づいた時点で**書き直しコスト**が発生
- 静的解析 → **決定的**。ルール違反は必ずエラー。git commit時に弾けばAIが即座に修正・再試行できる（**Shift-Left Testing**の考え方）

#### dependency-cruiserの4つのルール

| # | ルール名 | 内容 |
|---|---------|------|
| 1 | **純粋なdomain層** | `domain/`から`infrastructure/`や`server/`へのimport禁止。I/O系ライブラリ（drizzle-orm等）への依存も禁止 |
| 2 | **Feature間の疎結合** | あるfeatureから別featureへの直接import禁止（`shared/`経由で共有） |
| 3 | **Public API境界** | featureの外部から`index.ts`を通さず内部ディレクトリを直接参照することを禁止 |
| 4 | **shared/の独立性** | `shared/`から`features/`へのimport禁止。依存は常に一方向 |

これら4ルールに加え、infrastructure層の制約、循環依存防止、server層のRLSバイパス防止など**全13個のルール**を定義。

#### 補完ツール

| ツール | 役割 |
|--------|------|
| **knip** | 未使用コード検出。AIのリファクタリングで残りがちな未使用exportやファイルを検出 |
| **Biome** | Linter & Formatter。AIが生成しがちな`any`型、非nullアサーション、awaitされていないPromiseを検出 |

#### lefthookによるGit hooks統合

```yaml
# lefthook.yml
pre-commit:
  parallel: true
  commands:
    biome:
      run: pnpm check
      stage_fixed: true
    depcruise:
      root: "apps/web/"
      run: pnpm --filter web depcruise

pre-push:
  parallel: true
  commands:
    types:
      run: pnpm check-types
    knip:
      run: pnpm knip
    test:
      run: pnpm test
```

- **pre-commit**: Biome lint/format + dependency-cruiser（高速なツール）
- **pre-push**: TypeScript型チェック + knip + テスト実行（時間がかかるツール）

> アウトプットをレビューで検査するのではなく、**プロセスそのものを信頼できるものにする**。

## 結論・示唆

### 著者の結論

- AIの力を引き出すアーキテクチャは、**AI以前から大事だと言われていた設計原則**そのものであった
- CLAUDE.mdで複雑にルールを作り込むより、**責務が明確でSimpleなツール**をGit hooksで統合するほうが保守性が高い
- フィードバックを開発者の手元に近いところに寄せることで、設計は**決定論的**に守られる
- **良い設計をちゃんと守ること**、それがAIを最大限活用することにつながる

### 実践的な示唆

- Feature-First構成で関心事を1ディレクトリに集約し、AIのコンテキスト効率を上げる
- domain層を純粋関数で構成し、出力値ベーステストを書きやすくする（FCIS パターン）
- 設計ルールをドキュメントではなく**静的解析のルール**として表現する
- dependency-cruiser, knip, Biomeなどを**lefthookでGit hooksに統合**し、コミット/プッシュ時に自動検証する
- AIへの指示（CLAUDE.md等）は確率的なので、**決定的な仕組み（静的解析）で補完**する

## 制限事項・注意点

- 記事はTanStack Startを使ったフルスタックReactプロダクトの事例であり、他のフレームワークではディレクトリ構成の調整が必要（ただしNext.jsのapp/ディレクトリでも同様の構成が取れると言及あり）
- コード例は抽象/単純化されたもの
- 単体テストに焦点を絞った議論であり、E2Eテストやインテグレーションテストには触れていない
- dependency-cruiserの13ルール中、記事で詳細解説されているのは4ルールのみ

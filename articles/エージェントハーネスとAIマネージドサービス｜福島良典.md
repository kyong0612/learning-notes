## 概要

LayerX CEO 福島良典氏による記事。2026年のAI活用の主題は「AIエージェントを組織で機能させ、アウトカムに変える」ことであり、その本質的価値はAIエージェント単体ではなく、エージェントを包み込むインフラ＝**エージェントハーネス**にあると主張する。ハーネス＋AIエージェント＋決定論的コードをセットにした「業務の完成品」を**AIマネージドサービス**という造語で定義し、今後のAI企業の主戦場を描いている。

## 主要なトピック

### エージェントハーネスとは何か

- **定義**: 長時間実行されるタスクを管理するためにAIモデルを包み込むインフラストラクチャ。エージェントそのものではなく、エージェントの動作を制御し、信頼性・効率性・制御を確保するソフトウェアシステム
- エージェントフレームワーク（ツール構成やエージェントループの実装）よりも**高いレベル**で動作する
- 具体的な機能: プロンプトのプリセット、ツール呼び出しの独自処理、ライフサイクルフック、計画立案、ファイルシステムへのアクセス、サブエージェントの管理
- **比喩**: AIエージェント＝シェフ、ハーネス＝調理場。シェフの腕前がいくら高くても、調理場の設計が悪ければ良い料理は作れない
- 能力の上限を上げるのはモデルプロバイダー（Anthropic, OpenAI等）の仕事、ドメインに合ったハーネスをチューニングするのがApplication企業の仕事

参考: [philschmid.de - Agent Harness 2026](https://www.philschmid.de/agent-harness-2026)

### AIエージェントの仕事の長期化とベンチマーク問題

- かつてのAI活用は短いタスク実行が中心だったが、現在は**ツール・ナレッジ・メモリ・ワークフロー定義**等を渡して長時間自律実行させるパラダイムに移行
- AIベンチマークは飽和してきているが、**ロングランタスクを測るベンチマークが不十分**
- AIが99%の精度を持っていても、1%の誤りが後続ステップの精度を無意味にする
- 本当に必要なのは、エージェントが長時間自律実行し続けられるかを測る指標と、そのための環境整備のエンジニアリング

### 「ドリフト」問題とハーネスによる対処

長時間稼働時にAIエージェントが起こす典型的問題:

| 問題 | 具体例 |
|------|--------|
| **終了条件の誤認** | 「もう十分」と途中で完了宣言してしまう |
| **品質の自己過信** | 「コードは完璧です」と評価するがデザインは崩れている |
| **累積的なズレ** | 少しずつズレた仕事が積み重なる |

ハーネス設計による対処法:

| 対処法 | 説明 |
|--------|------|
| **終了条件の外部化** | チェックボックス形式で進行管理を外部に持たせる |
| **視覚的品質チェック** | Playwrightなどでエージェントに「目」を与える |
| **Hook（決定論的な実行）** | Tool実行後に必ずリンターを実行するなど、逸脱してはいけない通過点を設ける |

### コーディングハーネスからリアルタスクハーネスへ

- コーディングでうまくいった手法がリアルタスクに波及する（Claude Code → Claude Coworkの流れ）
- 今後「営業ハーネス」「経理ハーネス」「労務ハーネス」等のドメイン特化ハーネスが生まれる
- ハーネスの本質は**self-evolving system（自己改善システム）**の実現
- AIエージェントはハーネス内で「計画 → 実行 → 評価 → 反芻 → 再計画」の自己改善ループを回す
- ハーネスはモデルの進化に合わせて継続的に開発・運用し続ける必要がある

> ※「エージェントハーネス」は [Compound AI System](https://bair.berkeley.edu/blog/2024/02/18/compound-ai-systems/) や [コンテキストエンジニアリング](https://x.com/karpathy/status/1937902205765607626) など類似概念との重なりがあるが、いずれも「AIモデル単体ではなくシステム全体の設計が性能を決める」という共通の問題意識から生まれている。

## AIマネージドサービスという考え方

### 新しい言葉が必要な理由

- AI企業が提供しているのは「AIエージェント」単体ではなく、**AIエージェント（シェフ）＋ ハーネス（調理場）＋ 決定論的なコード（レシピ）** というセット
- すべてをAIに推論させる必要はない。ハーネスの中で生成された決定論的コードをサービスの一部として活用する形が有用
  - 例: **Generative UI**（AIがUIコードを生成しユーザーが触る）、AIが自動更新する正規表現によるパターンマッチ
- 顧客が欲しいのは「AIエージェント」ではなく**AIマネージドサービス（業務の完成品）**

### 「マネージドサービス」の系譜

- クラウドサービスの本質は**「進化を買う」**こと（AWSは単なるサーバーではなくインフラの進化そのもの）
- 進化には機能だけでなく、セキュリティ・法追従も含まれる → マネージドサービスの方が内製よりコスパが良い
- AIマネージドサービスはこの延長線上にあり、進化の担い手にAIも加わる
- **「自己進化を続ける業務の完成品」= AIマネージドサービス**

### 具体例: 給与計算でのハーネス設計

- ナイーブな設計: 給与計算の全コンテキストをAIに渡して推論 → **ナンセンス**（計算式はプログラムで実行すべき）
- 正しい設計: **AIは「推論がアップサイドで失敗のダウンサイドが少ない部分」に使う**
  - 典型例: **異常検知**（例: 住所変更申請がないのに通勤経路が変わっている → 通勤手当への影響を示唆）
  - 推論が間違っていたら人がリジェクトすればよい（ダウンサイドが少ない）
  - 確定した情報からの給与計算は決定論的コードで実行

### AIによる業務提供の3つの型

| 型 | 例 | 対象 | 特徴 |
|----|-----|------|------|
| **① AIマネージドサービス型** | バクラク、Harvey、Ramp | バックオフィス等の共通領域・非競争領域 | ハーネス＋AIエージェント＋決定論コードがドメイン特化にパッケージング。各社の業務に合わせてパーソナライズされる |
| **② AIプラットフォーム/SDK＋FDE型** | Ai Workforce、Sierra、Palantir | 内製エンジニア不足の企業の競争優位ワークフロー | AIプラットフォーム＋FDE（Forward Deployed Engineer）のセット |
| **③ 内製型** | 三井物産デジタル・アセットマネジメント | ②と同じだが内製エンジニアが豊富な場合 | Claude Code/Cowork等を活用した自社構築 |

- ①②型の提供ベンダーは、**内製よりコスパが高いことを自信を持って主張できなければならない**

### 具体例: ヘルプデスクエージェント＋ハーネス

LayerXの[ヘルプデスクエージェント](https://bakuraku.jp/helpdesk-agent/)を例にした解説:

- 基本機能（ナレッジからコンテキストを引いてAIに回答させる）はClaude Codeで誰でも作れる
- **本質的価値はナレッジの鮮度と正しさを常に改善し続けるハーネス**にある

**ナレッジ自動改善ループの仕組み:**

1. 従業員の質問（例:「ホテルは何万円まで取っていいですか？」）に対しナレッジに該当情報がなければ経理担当者にルーティング（ヒューマンインザループ）
2. 経理担当の回答（「3万円までOK」）をハーネスが「ナレッジに追加しませんか」と提案
3. 管理者が承認すれば次回から自動回答可能に
4. 社内規定変更時（3万→3万5000円）に旧ルールが残っていると矛盾が発生 → ハーネスがナレッジを巡回し矛盾を検知、「どちらを採用しますか？」と確認

> **ナレッジの鮮度は時間とともに構造的に劣化する。この鮮度劣化を継続的に解消するハーネスこそが価値。**

### 「内製できるのでは？」への回答

- 答え: **「作れる」** — 十分にClaude Codeを使いこなすエンジニアがいれば作れる
- しかし本当の問いは「作れるか否か」ではなく**経済合理性**

考慮すべき3つの論点:

| 論点 | 内容 |
|------|------|
| **① エンジニアの人件費** | 長期運用可能なAIエージェント＋ハーネスを設計・構築・運用できるエンジニアは高度人材。Claude Codeは月200ドルでもエンジニアの人件費は桁違い |
| **② 機会コスト** | 希少なエンジニアをコモディティ的な機能の構築・運用に使う意味はあるか |
| **③ 継続的な改善コスト** | モデルの進化でハーネスの在り方自体が変わる。微調整だけでなく抜本的修正も必要。作り切って終わりではない |

- AWSを自前で構築しないのと同じ構造が業務エージェントにも起きる
- **Anthropic自身も全てを内製化せず、AIマネージドサービスを導入している**のが証左
- ただし、ベンダーはポジショントークに陥らず、内製よりコスパが高いことを常に証明し続ける必要がある

## 結論・示唆

### 著者の結論

- 今後の主戦場は**ハーネス＋エージェントをドメインと顧客業務にマッチさせること**
- 世の中は「AIエージェント」を買っているように見えて、実はハーネスと一緒に買っている
- 顧客がほしいのは**「業務の完成品」**であり、それが内製よりコスパが高いことをベンダーは証明し続けなければならない
- LayerXは「バクラク」（①型）と「Ai Workforce」（②型）の2つの型で業務の完成品を提供していく

### 実践的な示唆

- AIエージェントの導入を検討する際は、エージェント単体の性能ではなく**ハーネス（実行環境・運用基盤）の設計**に注目すべき
- 内製 vs マネージドサービスの判断は、構築コストだけでなく**エンジニア人件費・機会コスト・継続改善コスト**の総合で考える
- AIに任せる部分は「推論がアップサイド・失敗のダウンサイドが少ない領域」に限定し、確定的処理は決定論的コードで実行する設計が合理的
- ナレッジベースの鮮度劣化は構造的問題であり、**自動改善ループの仕組み**を組み込むことが重要

## 制限事項・注意点

- 「エージェントハーネス」という用語は2026年4月時点でまだ定義が安定しておらず、Compound AI Systemやコンテキストエンジニアリングなど類似概念との重なりがある
- 著者はLayerX CEOであり、AIマネージドサービス提供側の立場からの論考である点に留意が必要
- 「AIマネージドサービス」は著者の造語であり、業界標準の用語ではない

## 概要

本記事は、Claude Codeを中心としたAIコーディング運用を実務レベルで仕組み化する方法論を提示する。著者は松尾研究所所属の中川氏で、小規模開発体制において「開発速度と品質を両立」させるための5つの実践的パターンを紹介している。

### 背景と問題意識

AIコーディングツールの使い方には「速く書く」と「プロセス固定化で品質・再現性を上げる」の2つの方向性がある。著者は後者の実務効果が高いと主張し、特に小規模体制での品質維持とナレッジ共有に焦点を当てている。

## 5つの実践パターン

### 型1「並列開発」：git worktreeによるコンテキスト分離

**目的**：モジュール単位でAIとの会話コンテキストを分離し、競合を最小化する

**実装方法**：
- `git worktree`で複数ウィンドウを常設
- 1モジュール = 1 Claude Code セッション
- 例：FastAPI バックエンド用、React フロントエンド用で別々のウィンドウとworktreeを使用

**効果**：
- 異なるモジュールの変更が混在しない
- AIが前のタスクの文脈を引きずらない
- 並行作業が可能になる

### 型2「プロンプト管理」：Markdownファイルでの指示一元化

**目的**：タスク管理ツール不要で、プロンプトの再利用性を確保

**実装方法**：
- プロンプトをMarkdownファイルとして管理
- Claude Codeに投げる指示を一元化
- バージョン管理下に置いて履歴を追跡

**効果**：
- 同じ指示を何度も書く必要がない
- プロンプトの改善履歴を管理できる
- チーム内でのプロンプト共有が容易

### 型3「実装↔レビュー自動ループ」：Subagent + Custom Slash Command

**目的**：実装と品質チェックを強制的に交互実行し、品質を担保

**実装方法**：

**モジュール別Subagentの定義**：
- FastAPI専用subagent（Python Expert）
- React専用subagent（TypeScript Expert）
- 各subagentに実装用とレビュー用の詳細な定義を記述

**カスタムスラッシュコマンド**：
- 実装とレビューを自動的に交互実行
- レビュー結果に基づいた修正を自動化

**具体例（記事から）**：
- FastAPI Python Expert：型安全性、エラーハンドリング、テスト駆動開発の強制
- React TypeScript Expert：コンポーネント設計、状態管理、パフォーマンス最適化

**効果**：
- 人手によるレビュー待ちを削減
- 一定の品質基準を自動的に維持
- レビュー観点の標準化

### 型4「ナレッジ一元化」：module/README.md + CLAUDE.md

**目的**：仕様・実装詳細・テスト・ドキュメントを固定化し、機能追加時に同期更新

**実装方法**：
- 各モジュールの`README.md`に仕様と実装詳細を記述
- `CLAUDE.md`でClaude Code向けの指示を定義
- 機能追加時に自動的に両方を更新

**効果**：
- ドキュメントと実装の乖離を防ぐ
- 新規メンバーのオンボーディングが容易
- AIが最新の仕様を参照できる

### 型5「Skillsへの指示退避」：長文インストラクション埋没の回避

**目的**：CLAUDE.mdに書いても従わない指示をSkillsで定義し、確実に実行させる

**実装方法**：
- 後方互換コードの排除などの厳格なルールをSkillsとして定義
- 長文のインストラクションをSkillsに退避
- 必要に応じてSkillsを呼び出す

**具体例**：
- 後方互換コードの強制削除
- 特定のコーディング規約の徹底
- セキュリティチェックの自動化

**効果**：
- 重要な指示が埋もれない
- 指示の実行率が向上
- コンテキスト制限の回避

## 技術スタック

記事で言及されている技術環境：
- **バックエンド**：FastAPI（Python）
- **フロントエンド**：React（TypeScript）
- **開発ツール**：Claude Code、git worktree
- **品質保証**：Subagent、Custom Slash Command

## 結論

著者は、AIコーディングツールの価値は「速く書く」ことよりも「プロセス固定化で品質・再現性を上げる」ことにあると指摘している。特に小規模開発体制では、これらの5つの型を組み合わせることで、開発速度を維持しながら品質を担保できるとしている。

重要な示唆として、AIツールは単なる生産性向上ツールではなく、開発プロセス自体を再設計する機会を提供するものであると結論付けている。
